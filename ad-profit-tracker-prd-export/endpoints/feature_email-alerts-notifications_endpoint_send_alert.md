# Feature Email Alerts Notifications Endpoint Send Alert

POST /api/email/send-alert

## Description

REQUEST: POST { userId: string (required — from verified token or x-internal-secret), type: string (required — one of the 9 notification type enum values), title: string (required, max 100 chars), body: string (required, max 500 chars), networkId?: string|null, severity: 'info'|'success'|'warning'|'error' (required), metadata?: map }. Dual-auth: Bearer token (user-initiated) or x-internal-secret header (server-side Cloud Function triggers).
PROCESS:
1. Check auth: if Authorization header present, verify Firebase ID token via admin.auth().verifyIdToken(token) and derive uid. If x-internal-secret header present, validate against INTERNAL_SYNC_SECRET env var and use userId from request body. Return 401 if neither is valid.
2. Validate all required fields: type must be one of the 9 allowed notification type values, severity one of 4 values, title max 100 chars, body max 500 chars. Return 400 if invalid.
3. Read users Firestore document for uid via Admin SDK. Extract users.notificationPreferences[type].emailEnabled. If emailEnabled === false, return 200 with { sent: false, reason: 'disabled_by_user' } without sending.
4. Extract delivery email from users.preferences.alertDeliveryEmail. If not set, fall back to Firebase Auth user record email via admin.auth().getUser(uid). If no email found, return 400 with 'No delivery email configured'.
5. Check rate limit: max 10 alert emails/hour per user — return 429 with Retry-After header if exceeded. Use a rate limit key of '{uid}_email_alert_{type}'.
6. Use a transactional email provider SDK (e.g., Nodemailer with SMTP, Resend, SendGrid, Postmark) configured via environment variables (EMAIL_PROVIDER_API_KEY, EMAIL_FROM_ADDRESS, EMAIL_PROVIDER). Render an HTML email template for the given type: subject line 'Ad Profit Tracker Alert — [title]', body with severity badge color (red=error, amber=warning, green=success, blue=info), notification body text, networkId context if provided, metadata key-value pairs, and a footer with 'Manage email preferences' link pointing to /settings.
7. Send the email. On provider failure, write audit log with status='failure' and return 502.
8. Write audit log entry to auditLogs Firestore collection via Admin SDK: action='email_sent', resourceType='notification', resourceId=type, metadata={ emailType: type, recipientEmail: deliveryEmail, subject, networkId, severity, triggeredBy: 'user'|'system' }, status='success'.
9. Return delivery confirmation.
SUCCESS (200): { sent: true, recipientEmail: string, type: string, severity: string }
ERRORS:
- 400: invalid type/severity, missing required fields, no delivery email configured
- 401: neither valid Firebase ID token nor valid internal secret
- 404: users document not found
- 429: rate limit exceeded (10/hour per user per type)
- 502: email provider delivery failure (retryable)
- 500: Firestore read failure
SECURITY: Dual-auth pattern. Internal calls use x-internal-secret. User calls use Firebase ID token. userId always from verified token for user calls. deliveryEmail always from Firestore/Firebase Auth — never from request body. Rate limited to prevent email flooding.
