# Rate Limit Endpoint Status

GET /api/rate-limits/status

## Description

REQUEST: GET with optional query param: networkId? (one of 'exoclick'|'rollerads'|'zeydoo'|'propush' — omit for all networks). Bearer token in Authorization header.
PROCESS:
1. Verify Firebase ID token via admin.auth().verifyIdToken(token) — return 401 if invalid. uid from verified token only.
2. Validate networkId if provided — return 400 if not one of the 4 supported networks.
3. For each target network (or all 4), read the current Bottleneck limiter state from the in-memory limiter registry (lib/rate-limit/limiterRegistry.ts) keyed by userId+networkId. Extract: running (current concurrent jobs), queued (jobs waiting), reservoir (remaining tokens in current window), nextReservoirRefresh (when tokens replenish).
4. Also read per-endpoint user-facing rate limit state from the in-memory sliding window counters (lib/rate-limit/userLimiter.ts) for the authenticated uid: remaining requests for /api/sync/manual (10/hour), /api/stats/backfill (3/hour), /api/scheduled/retry-failed (5/hour per network), /api/reconciliation/run (10/hour), /api/errors/circuit-breaker/reset (10/hour).
5. Read circuitBreakerOpen and retryCount from networkConfigs Firestore collection for each network (getDocs, filter userId==uid) to enrich the response with circuit breaker context.
6. Compute isThrottled per network: true if reservoir==0 or circuitBreakerOpen==true.
7. Return consolidated rate limit state per network plus per-endpoint user quotas.
SUCCESS (200): { networks: [{ networkId: string, isThrottled: boolean, reservoir: number, reservoirMax: number, nextReservoirRefreshAt: string|null, running: number, queued: number, circuitBreakerOpen: boolean, retryCount: number }], userQuotas: [{ endpoint: string, limit: number, windowSeconds: number, remaining: number, resetAt: string }], polledAt: string }
ERRORS:
- 400: invalid networkId
- 401: unauthenticated
- 500: Firestore read failure or limiter registry unavailable
SECURITY: userId from verified token only. Read-only endpoint. No writes. Limiter registry is server-side only — never exposes raw API keys or internal secrets.
