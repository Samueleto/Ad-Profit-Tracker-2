# Feature Historical Data Storage Endpoint Backfill

POST /api/stats/backfill

## Description

REQUEST: POST { networkId?: string (optional — omit for all active networks), dateFrom: string (YYYY-MM-DD, required), dateTo: string (YYYY-MM-DD, required) }.
PROCESS:
1. Verify Firebase ID token via admin.auth().verifyIdToken(token) — return 401 immediately if invalid. uid from verified token only.
2. Validate networkId if provided: must be one of ['exoclick','rollerads','zeydoo','propush'] — return 400 if not.
3. Validate dateFrom and dateTo: required, valid YYYY-MM-DD, dateFrom <= dateTo, max 90-day range — return 400 if invalid.
4. Check rate limit: max 3 backfill requests/hour per user — return 429 with Retry-After header if exceeded. Backfills are expensive operations.
5. Determine target networks: if networkId provided, use [networkId]; otherwise query networkConfigs Firestore collection for all documents where userId==uid AND isActive==true.
6. For each target network, verify an API key exists in apiKeys collection where userId==uid AND networkId==network — skip networks without keys and add to skipped array.
7. For each network with a valid key, call the shared server-side helper lib/sync/syncNetworkForUser(uid, networkId, dateFrom, dateTo) which handles decryption, API calls, exponential backoff, and batch upserts into adStats. This reuses the exact same helper as POST /api/sync/manual — no new data ingestion logic.
8. Use Promise.allSettled() to fan out across all target networks in parallel — do not let one failure block others.
9. After each network completes, update networkConfigs document: lastSyncedAt=serverTimestamp(), lastSyncStatus='success'|'failed', lastSyncError=null|errorMessage.
10. Write one audit log entry per network directly to auditLogs Firestore collection via Admin SDK: action='manual_sync_triggered', resourceType='sync', resourceId=networkId, metadata={ scheduledSync: false, latencyMs, rowsFetched, dateFrom, dateTo, triggeredBy: 'backfill' }, status='success'|'failure'.
11. Collect results and return summary.
SUCCESS (200): { triggered: string[], skipped: string[], failed: string[], results: [{ networkId: string, status: 'success'|'failed'|'skipped', rowsFetched: number|null, latencyMs: number|null, errorMessage: string|null }], dateFrom: string, dateTo: string, triggeredAt: string }
ERRORS:
- 400: invalid networkId, invalid date range, range exceeds 90 days
- 401: invalid/expired Firebase ID token
- 429: rate limit exceeded (3/hour per user — backfills are expensive)
- 500: Firestore read/write failure or unhandled exception
SECURITY: API keys decrypted server-side only via shared helper, never returned or logged. userId always from verified token. All Firestore writes scoped to authenticated uid. Rate limited more aggressively than manual sync (3/hour vs 10/hour) because backfills cover multi-day ranges and generate significantly more Firestore writes.
