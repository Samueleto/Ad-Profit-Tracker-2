The audit logging feature handles sensitive user activity data, so let's make sure every layer is properly secured.

**Token verification on every request:** All three endpoints — GET /api/audit/logs, GET /api/audit/logs/export, and DELETE /api/audit/logs/clear — must verify the Firebase ID token via admin.auth().verifyIdToken(token) before doing anything else. The userId used in every Firestore query must come from the verified token's uid field, never from a query param or request body. If a client tries to pass a userId in the request, ignore it entirely.

**Firestore security rules:** The auditLogs collection rules should enforce that users can only read and delete their own documents (where userId == request.auth.uid). The createdAt and userId fields must be immutable — once written, they cannot be changed. Create operations should only be allowed when the userId in the document matches the authenticated user. These rules act as a second layer of defense behind the API routes.

**Metadata sanitization:** The POST /api/audit/log endpoint (called internally by other endpoints) sanitizes the metadata object server-side before writing to Firestore. Make sure any field whose key contains 'key', 'token', 'secret', or 'password' is stripped out. This prevents accidental credential logging. Add a utility function for this sanitization that's easy to test in isolation.

**Rate limiting:** Enforce these rate limits at the API route level:
- GET /api/audit/logs: 60 requests/minute per user
- GET /api/audit/logs/export: 5 requests/hour per user (exports are expensive)
- DELETE /api/audit/logs/clear: 3 requests/day per user (destructive, prevent accidental repeated purges)
- POST /api/audit/log: 60 requests/minute per user (prevents log flooding)

Use the verified uid as the rate limit key, not the IP address (users behind NAT could share IPs). Return 429 with a Retry-After header when limits are exceeded.

**IP and User-Agent capture:** For POST /api/audit/log, the ipAddress must be extracted server-side from x-forwarded-for or req.socket.remoteAddress — never from client-supplied headers or body. The userAgent comes from the User-Agent request header. These are stored for security audit purposes.

**Export hard cap:** GET /api/audit/logs/export must enforce a hard limit of 10,000 documents regardless of what filters are applied. This prevents runaway Firestore reads from a single export request.

**Clear logging:** DELETE /api/audit/logs/clear must write a final audit entry (directly via Admin SDK, not via POST /api/audit/log to avoid recursion) recording that logs were cleared, including the deletedCount and scope, before returning success.
