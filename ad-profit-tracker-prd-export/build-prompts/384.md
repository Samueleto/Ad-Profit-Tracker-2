The SettingsPage deals with sensitive operations, so errors need to be handled clearly and helpfully without leaking any technical details to the user.

Here's how each failure scenario should behave:

**Initial status load failure** (GET /api/keys/status fails): Don't show a blank page or broken cards. Show an error state with a message like 'Unable to load connection status' and a 'Retry' button. The individual network cards should still render — just show their status as unknown rather than crashing.

**Save failures** (POST /api/keys/save):
- 400 (invalid input): Show an inline error below the input field — something like 'Invalid API key format'. This shouldn't need a toast since it's a field-level issue.
- 401 (session expired): Show a toast saying 'Your session has expired. Please sign in again.' and redirect to login.
- 429 (rate limited): Show a toast saying 'Too many attempts. Please wait a moment and try again.'
- 500 (server error): Show a toast saying 'Something went wrong saving your key. Please try again.' Keep the form open so the user can retry without re-entering the key.
- Network failure (no response): Show a toast saying 'Connection error. Check your internet and try again.'

**Disconnect failures** (DELETE /api/keys/delete):
- 404 (key not found): Treat this as a success — the key is already gone, so update the UI to 'Not connected' anyway.
- 401: Same session-expired flow as above.
- 500 or network failure: Show a toast saying 'Unable to disconnect. Please try again.' and revert any optimistic UI update.

**General principle**: Never show raw error messages, HTTP status codes, or stack traces to the user. Map every error to a plain-English message. For unexpected errors, 'Something went wrong' is fine — just make sure there's always a clear next action (retry, sign in again, etc.).

On the server side, make sure the API routes log errors with enough detail to debug (including the networkId and a timestamp) but never log the raw API key or the encrypted key.
