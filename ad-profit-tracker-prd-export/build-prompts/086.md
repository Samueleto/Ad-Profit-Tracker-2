We need Firestore configured so the adStats collection queries run fast and the data is locked down properly.

**Composite indexes** — The six ExoClick endpoints run several specific queries against the `adStats` collection. Add these composite indexes to `firestore.indexes.json`:

- `userId ASC + networkId ASC + date ASC` — used by `GET /api/networks/exoclick/stats` to fetch a date range for one user and network
- `userId ASC + networkId ASC + date DESC` — used by `GET /api/networks/exoclick/stats/latest` to find the most recent date
- `userId ASC + networkId ASC + date ASC + country ASC` — used by `GET /api/networks/exoclick/stats/by-country` and `GET /api/networks/exoclick/raw-response` which filter on country

Also add a single-field index on `networkId` to support the scheduled-sync endpoint's cross-user collection-group query.

**Security rules** — Update `firestore.rules` to cover the `adStats` collection:
- Any authenticated user can create a document, but only if the `userId` field in the document matches their own Firebase Auth UID (`request.auth.uid`)
- A user can read their own documents only (`resource.data.userId == request.auth.uid`)
- A user can update their own documents, but `createdAt` and `userId` fields must remain unchanged (immutable)
- Deletes should be soft-deletes only — don't allow hard deletes from the client. You can enforce this by simply not granting delete permission in the rules (the server-side Admin SDK bypasses rules anyway)

Note: the sync endpoints write to `adStats` using the Firebase Admin SDK, which bypasses security rules entirely — so the rules here are purely for any direct client-side reads. The API endpoints are the authoritative access path.
