We're building a Next.js + Firebase app. For this step, I need you to make sure the preferences endpoints and Firestore access are properly secured end-to-end.

For both GET /api/settings/preferences and PATCH /api/settings/preferences, the very first thing each handler does must be to extract the Bearer token from the Authorization header and verify it using Firebase Admin SDK's `admin.auth().verifyIdToken(token)`. If the token is missing, malformed, or expired, return 401 immediately — don't proceed with any Firestore access. The UID used for all Firestore operations must always come from the verified token result, never from the request body or query parameters.

For the PATCH endpoint specifically, enforce a strict whitelist of allowed fields. Only these paths should ever be written to Firestore: `preferences.timezone`, `preferences.currency`, `preferences.defaultDateRange`, `preferences.notifications.dailySummaryEmail`, `preferences.notifications.weeklyReportEmail`, and `preferences.updatedAt`. Build the Firestore update object by explicitly mapping only these fields from the request body — never spread the request body directly into the update object. This prevents a malicious client from writing arbitrary fields to the users document.

Validate every field value before writing:
- timezone: must be a valid IANA timezone string. Use `Intl.supportedValuesOf('timeZone')` to check against the runtime's supported list.
- currency: must be one of a hardcoded whitelist of ISO 4217 codes (USD, EUR, GBP, JPY, CAD, AUD, CHF, CNY, INR, MXN — expand as needed).
- defaultDateRange: must be exactly one of `last_7_days`, `last_14_days`, `last_30_days`, `this_month`.
- notification fields: must be strictly boolean (typeof value === 'boolean').
Return 400 with field-specific error messages for any invalid value.

For Firestore security rules on the users collection, make sure the rules enforce that a user can only read and write their own document (where `request.auth.uid == resource.data.uid` or equivalent). The preferences fields should be writable but `createdAt` must be immutable after creation.

Add rate limiting to the PATCH endpoint — no more than 20 preference updates per user per minute is reasonable. Use a simple in-memory counter or a Firebase-compatible rate limiting approach.

On the client side, make sure the Firebase ID token is always fetched fresh via `getIdToken()` before each API call — never store it in localStorage or a cookie manually.
