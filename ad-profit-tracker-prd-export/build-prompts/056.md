We need two Firebase SDKs wired up correctly — the client-side `firebase/auth` for the frontend (attaching Bearer tokens to API calls) and `firebase-admin` on the server side (verifying those tokens in the three API routes).

For the client side: make sure `firebase/auth` is available and that there's a utility function — something like `getAuthToken()` — that calls `getIdToken()` on the current Firebase user and returns the token string. This will be used by the polling hooks to attach an `Authorization: Bearer <token>` header to every call to `/api/sync/live-state`, `/api/sync/activity-feed`, and `/api/sync/progress`. Also expose a way to force-refresh the token via `getIdToken(true)` — this is needed when the SyncStatusPanel gets a 401 back, so it can try refreshing before giving up and redirecting.

For the server side: set up a Firebase Admin SDK initialization module at `lib/firebaseAdmin.ts`. It should initialize the admin app once (guard against re-initialization in Next.js hot reload) using a service account — either from a JSON file or from environment variables (`FIREBASE_PROJECT_ID`, `FIREBASE_CLIENT_EMAIL`, `FIREBASE_PRIVATE_KEY`). Export an `adminAuth` instance and an `adminDb` (Firestore) instance. All three API routes — `GET /api/sync/live-state`, `GET /api/sync/activity-feed`, and `PATCH /api/sync/progress` — will import from this module to call `admin.auth().verifyIdToken(token)` and to run Firestore queries.

For the `PATCH /api/sync/progress` endpoint specifically, also make sure the `INTERNAL_SYNC_SECRET` environment variable is documented in `.env.example` — this endpoint accepts either a Firebase ID token or this secret header (`x-internal-secret`) for calls coming from scheduled sync helpers.

Make sure all sensitive values (`FIREBASE_PRIVATE_KEY`, `INTERNAL_SYNC_SECRET`) are in `.env.local` and never committed. Add them to `.env.example` with placeholder values.
