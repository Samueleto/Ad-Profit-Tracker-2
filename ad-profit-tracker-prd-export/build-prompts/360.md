# Step 360 of 437: Lock down report builder routes and API calls

================================================================================

**Category:** Security & Authentication
**Flow:** feature_custom-report-builder

================================================================================

The report builder touches several sensitive endpoints and stores user data. Let's make sure everything is properly secured on the client side (the server-side security is handled in the API endpoints themselves, but the frontend needs to hold up its end).

**Route protection**

Confirm that `app/(app)/reports/page.tsx` sits inside the `(app)` route group. The existing `middleware.ts` should already enforce Firebase Auth for this group — verify it does. If a user somehow lands on /reports without being authenticated, they should be redirected to / (the sign-in page), not shown a broken page.

**Token attachment**

Every API call from the report builder must include the Firebase ID token as a Bearer token. Use the shared `getAuthHeader()` utility built in the state management step — make sure it's used consistently in `useReportPreview`, `useRunReport`, and `useSavedReports`. Never hardcode tokens or read them from localStorage directly.

**Token refresh on 401**

If any API call returns a 401, the client should attempt a token refresh by calling `firebase.auth().currentUser?.getIdToken(true)` (the `true` forces a refresh). Retry the original request once with the new token. If the retry also returns 401, show a 'Session expired' toast and redirect the user to /. Do this in a shared response handler so you don't duplicate the logic in every hook.

**Client-side date range enforcement**

Before any API call is made (preview, run, export), check that the date range doesn't exceed 90 days. If it does, block the call and show an inline warning. This is a UX guard — the server also enforces this, but we don't want to waste a round trip.

**Config validation before API calls**

Before calling `POST /api/reports/save` or `POST /api/reports/run`, validate the config client-side:
- metrics must be non-empty
- networks must be non-empty
- groupBy must be one of the three valid values
- dataQuality must be one of the three valid values
- If dateRangePreset is 'custom', both dateFrom and dateTo must be set

If any check fails, show an inline error near the relevant control rather than letting the API return a 400.

**Saved report name sanitization**

When the user types a report name in the Save Report inline field, trim whitespace and enforce the 1-100 character limit client-side before submitting. Show a character counter and disable the submit button if the name is empty or over 100 characters.

**No sensitive data in URLs**

The deep-link feature uses `?reportId=<id>` in the URL. Make sure the report config itself (metrics, networks, date ranges) is never put in the URL — only the report ID. The actual config is always fetched from the server using the authenticated user's session.
