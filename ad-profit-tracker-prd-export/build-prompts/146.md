We need TypeScript types and constants that all the email endpoints and the `EmailAlertPreferencesSection` component will share. Centralizing these prevents the 9 notification type strings from being scattered as magic strings across 7 different files.

Create `src/lib/email/types.ts` with the following:

**Notification types enum/constant** — Export a `NOTIFICATION_TYPES` array (or const enum) with all 9 values: `sync_failure`, `sync_success`, `export_complete`, `export_failure`, `schedule_delivered`, `schedule_failed`, `reconciliation_anomaly`, `circuit_breaker_opened`, `rate_limit_exceeded`. Also export a `NotificationType` TypeScript type derived from this.

**Severity type** — Export `Severity = 'info' | 'success' | 'warning' | 'error'`.

**Notification metadata** — Export a `NotificationTypeConfig` interface with: `type: NotificationType`, `label: string` (human-readable name), `description: string` (short description of when it fires), `severity: Severity`, `isCritical: boolean`. Then export a `NOTIFICATION_TYPE_CONFIGS` map that defines all 9 types with their labels, descriptions, severities, and whether they're critical. Critical types are `sync_failure`, `reconciliation_anomaly`, and `circuit_breaker_opened`. This config drives both the UI toggle rows in `EmailAlertPreferencesSection` and the default `emailEnabled` values in the API.

**Preferences types** — Export `NotificationPreference` with `type`, `enabled`, `emailEnabled`, `isDefault`. Export `NotificationPreferencesResponse` matching the shape returned by `GET /api/notifications/preferences`: `{ preferences: NotificationPreference[], alertDeliveryEmail: string | null, lastTestEmailSentAt: string | null }`. Export `PatchPreferencesRequest` matching the extended `PATCH /api/notifications/preferences` body.

**Audit log types** — Export `EmailAuditLog` matching the shape returned by `GET /api/email/log`: `{ id, emailType, recipientEmail, subject, status, sentAt, triggeredBy }`.

**Summary types** — Export `SummaryType = 'daily' | 'weekly' | 'monthly'` and `SendSummaryRequest` matching the `POST /api/email/send-summary` body.

These types should be imported by all 7 endpoints and the `EmailAlertPreferencesSection` component — no more inline type definitions scattered around.
