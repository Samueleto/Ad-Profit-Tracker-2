We're building the Help & Documentation Center on Next.js + Firebase. For this step, make sure the help pages and their API calls are properly secured end to end.

**Route-level protection**

Both HelpCenterPage (/help) and HelpArticlePage (/help/[articleId]) are inside the (app) route group. Verify that middleware.ts enforces Firebase Auth on these routes — the matcher pattern must cover /help and /help/[articleId]. If the middleware uses a pattern like `/app/(.*)`, make sure /help is either inside that group or explicitly added to the matcher.

**Bearer token on every API call**

Every API call made from these pages must include the Firebase ID token as a Bearer token in the Authorization header. Use `getIdToken()` from firebase/auth to get the current token before each request. Don't cache the token in module scope — always call getIdToken() so you get a fresh token (Firebase handles the refresh internally). The calls that need this are:
- GET /api/help/articles (HelpCenterPage, initial load and category filter)
- GET /api/help/search (HelpCenterPage, search)
- GET /api/rbac/my-permissions (HelpCenterPage, role-specific guides)
- GET /api/help/articles/:articleId (HelpArticlePage)
- PATCH /api/help/articles/:articleId/view (HelpArticlePage, fire-and-forget)
- POST /api/help/feedback (HelpArticlePage)
- GET /api/help/articles?category=...&limit=4 (HelpArticlePage, related articles)

**Token refresh on 401**

If any API call returns 401, attempt a forced token refresh using `getIdToken(true)` (the `true` forces a server-side refresh). Retry the original request once with the new token. If the retry also returns 401, the session is genuinely expired — redirect to / and show a 'Session expired' toast. Don't retry more than once — that would loop.

**Permission-gated articles**

The server already enforces permission checks on GET /api/help/articles/:articleId (returns 403 if the user lacks requiredPermission or requiredRole). On the client side, don't try to pre-filter articles by permission — let the server be the source of truth. When HelpArticlePage receives a 403, show the 'Access Denied — this article requires elevated permissions' UI with a link back to /help. Don't expose the article title or any content in the 403 state.

**Draft article protection**

The server returns 404 for unpublished (isPublished=false) articles. On the client, treat a 404 the same way regardless of whether it's a genuinely missing article or a draft — show 'Article not found' with no indication that a draft exists. This prevents content leakage.

**Search result safety**

GET /api/help/search already filters out permission-gated articles server-side. On the client, don't add any client-side filtering on top of this — trust the server's response. Just render what comes back.

**No sensitive data in URLs**

The article ID in /help/[articleId] is fine to expose in the URL. But make sure the search query (q param) isn't being logged anywhere client-side in a way that could leak sensitive search terms. If the app has any analytics or logging, make sure search queries from the help center are either excluded or anonymized.
