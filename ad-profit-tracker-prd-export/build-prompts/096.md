This feature reads from two existing Firestore collections and writes to one. We need to make sure the right indexes and security rules are in place.

**Collections used:**

1. `networkConfigs` — Already exists in the project. The rate limit status endpoint reads from this collection, filtering by `userId == uid` to get `circuitBreakerOpen` and `retryCount` per network. Make sure there's a composite index on `(userId ASC, networkId ASC)` so this query runs efficiently.

2. `auditLogs` — This is where rate limit violations and reset actions get logged. The violations endpoint queries this with `userId == uid AND resourceType == 'rate_limit' AND action == 'rate_limit_exceeded'`, ordered by `createdAt DESC`, with optional filtering on `resourceId` (networkId). You'll need a composite index on `(userId ASC, resourceType ASC, action ASC, createdAt DESC)` for this query to work. Also add an index on `(userId ASC, resourceType ASC, createdAt DESC)` for the base query without action filtering.

For the reset endpoint's audit log writes, the document shape should be: `{ userId, action: 'network_config_updated', resourceType: 'rate_limit', resourceId: networkId, metadata: { scope, resetBy, previousReservoir }, status: 'success', createdAt: Firestore.FieldValue.serverTimestamp() }`.

For violation log writes (written elsewhere in the app when limits are hit), the document shape should include: `{ userId, action: 'rate_limit_exceeded', resourceType: 'rate_limit', resourceId: networkId, metadata: { endpoint, limitType, limitValue, windowSeconds, retryAfterSeconds }, createdAt }`.

**Firestore security rules:** The `auditLogs` collection should only be readable and writable by the server (Firebase Admin SDK). Client-side access should be blocked entirely — all reads go through the API endpoints which use the Admin SDK. Add a rule that denies all client reads and writes to `auditLogs`.

Deploy these indexes via `firestore.indexes.json` so they're version-controlled.
