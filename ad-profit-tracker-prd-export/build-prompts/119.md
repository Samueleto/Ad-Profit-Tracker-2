We need TypeScript types and validation utilities for the preferences data before we build the API endpoints or UI.

First, define the TypeScript types. Create a `UserPreferences` interface that matches the nested `preferences` object in the `users` collection. It should have: `timezone` (string), `currency` (string), `defaultDateRange` (a union type of the four allowed enum values: `'last_7_days' | 'last_14_days' | 'last_30_days' | 'this_month'`), `notifications` (an object with `dailySummaryEmail: boolean` and `weeklyReportEmail: boolean`), and `updatedAt` (a Firestore `Timestamp` or `null`). Also export a `DEFAULT_PREFERENCES` constant with the system defaults: timezone `'UTC'`, currency `'USD'`, defaultDateRange `'last_7_days'`, both notification flags `false`, updatedAt `null`.

Second, create validation helpers that the PATCH endpoint will use:
- A timezone validator that checks whether a given string is a valid IANA timezone. Use `Intl.supportedValuesOf('timeZone')` if available in the runtime, or maintain a curated list. Return a boolean.
- A currency validator that checks against a list of valid ISO 4217 codes. Include at minimum: USD, EUR, GBP, JPY, CAD, AUD, CHF, CNY, INR, MXN, BRL, SGD, HKD, NOK, SEK, DKK, NZD, ZAR, KRW, AED. Return a boolean.
- A date range validator that checks the value is one of the four allowed enum strings.

Third, create a helper function `buildPreferencesUpdateObject(patch: Partial<UserPreferences>)` that takes a partial preferences update and returns a Firestore-safe dot-notation update object (e.g., `{ 'preferences.timezone': 'America/New_York', 'preferences.updatedAt': serverTimestamp() }`). This keeps the PATCH endpoint clean and ensures we never accidentally overwrite unrelated fields.

Put all of this in a shared location (like `src/lib/preferences.ts` or `src/types/preferences.ts`) so both the API routes and the frontend can import from it.
