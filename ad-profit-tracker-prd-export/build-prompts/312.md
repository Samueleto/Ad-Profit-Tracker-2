We're building an API rate limiting and throttling system. For this step, I need you to connect the rate limit hooks into the parts of the app that actually make outbound API calls — so that throttling state is visible and respected throughout the app, not just on a dedicated rate limits page.

First, identify where the app currently calls the four rate-limited user endpoints: `/api/sync/manual`, `/api/stats/backfill`, `/api/scheduled/retry-failed`, `/api/reconciliation/run`, and `/api/errors/circuit-breaker/reset`. In each of those places, use `useRateLimitStatus` to check the `userQuotas` array before allowing the user to trigger the action. If the `remaining` count for that endpoint is 0, disable the trigger button and show a tooltip or inline message like "Quota reached — resets at [resetAt time]". Don't block silently — always tell the user why the button is disabled.

For the four ad networks (exoclick, rollerads, zeydoo, propush), wherever the app shows network-specific actions or status, use `useRateLimitStatus` with the relevant `networkId` to check `isThrottled`. If `isThrottled` is true (either because `reservoir == 0` or `circuitBreakerOpen == true`), show a visible throttled indicator next to that network — a badge, icon, or status chip is fine. Include the `nextReservoirRefreshAt` time if available so users know when things will recover.

Wire up the `useRateLimitReset` hook to any existing admin or settings UI where operators can manually reset a network. The reset form should let the user pick a `networkId` and a `scope` (`'outbound'`, `'user-quota'`, or `'both'`). After a successful reset, the status should refresh automatically (this should already work from the state management step).

If the app has a notifications or toast system, hook into it from `useRateLimitReset` — show a success toast on reset and an error toast on failure.

Finally, make sure the `useRateLimitConfig` hook is called somewhere early in the app (a layout or provider component is fine) so the config data is warm in memory before users navigate to any rate-limit-related UI. This avoids a loading flash when they first open those views.
