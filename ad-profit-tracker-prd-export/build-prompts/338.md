We're building an API rate limiting and throttling system. For this step, I need you to review and harden the security posture of all four rate limit endpoints.

For `GET /api/rate-limits/status`: Make sure the Firebase ID token is verified via `admin.auth().verifyIdToken()` before doing anything else. The `uid` used to look up limiter state and Firestore `networkConfigs` documents must come exclusively from the verified token — never from a query parameter or request body. The response must only include data for the authenticated user's limiters. Double-check that the Bottleneck limiter registry lookup is keyed by `userId+networkId` and that there's no way to read another user's limiter state by guessing a key.

For `POST /api/rate-limits/reset`: This endpoint uses a dual-auth pattern — either a Firebase Bearer token or an `x-internal-secret` header matching the `INTERNAL_SYNC_SECRET` environment variable. Make sure the logic is: check Bearer token first, then fall back to internal secret, and return 401 if neither is valid. Never accept both simultaneously as a way to escalate privileges. User-authenticated resets must be rate-limited at 10/hour (enforce this with the same sliding window counter used elsewhere). Internal-secret calls must not be rate-limited. Audit log writes must only happen for user-authenticated calls — internal calls skip the audit log to avoid flooding. Critically, this endpoint must never reset circuit breaker state — that belongs to `POST /api/errors/circuit-breaker/reset` only. Add a guard that explicitly ignores any attempt to reset circuit breaker state here.

For `GET /api/rate-limits/config`: Auth is required even though the config contains no secrets — this prevents unauthenticated enumeration of your rate limit configuration. Verify the token, but no user-scoped data filtering is needed since config is global.

For `GET /api/rate-limits/violations`: The Firestore query for `auditLogs` must always include `userId==uid` as a filter — never return violations from other users. The `endpoint` filter is applied in-memory after the Firestore query, which is fine, but make sure the in-memory filter can't be bypassed by a crafted `endpoint` value (just do a strict string equality check).

Across all four endpoints: make sure the `INTERNAL_SYNC_SECRET` is never logged, never included in any response body, and loaded only from environment variables. Add a startup check that throws if `INTERNAL_SYNC_SECRET` is missing or empty in production.
