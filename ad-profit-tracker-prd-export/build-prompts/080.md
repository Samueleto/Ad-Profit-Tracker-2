We need to set up Firestore with the right security rules and make sure the `users` collection is ready before we start writing to it.

**Firestore Security Rules:**
Create or update your `firestore.rules` file with rules for the `users` collection:
- **Create**: Only allowed if the requesting user is authenticated (`request.auth != null`) and the document ID matches their Firebase Auth UID (`request.auth.uid == userId`). This prevents one user from creating a document on behalf of another.
- **Read**: Only the document owner can read their own document (`request.auth.uid == resource.data.uid`).
- **Update**: Only the document owner can update their own document. Additionally, enforce that `createdAt` and `uid` fields are immutable — if either field is present in the incoming data, it must match the existing value.
- **Delete**: Disallow direct client-side deletes. Deletion goes through the `DELETE /api/auth/delete-account` endpoint using the Admin SDK, which bypasses security rules. Set the rule to `allow delete: if false`.

Note: Our API routes use the Firebase Admin SDK, which bypasses Firestore security rules entirely. The rules above protect against direct client-side Firestore access.

**Firestore Indexes:**
The `users` collection doesn't need composite indexes for now — we're only ever fetching a single document by UID. No `firestore.indexes.json` changes needed at this stage.

**Firebase Console Setup:**
Make sure the following are enabled in your Firebase project:
- Firestore Database is created (in production or test mode — switch to the rules above once created)
- Firebase Authentication is enabled with two providers turned on: **Email/Password** and **Google**

Deploy the security rules using `firebase deploy --only firestore:rules`.
