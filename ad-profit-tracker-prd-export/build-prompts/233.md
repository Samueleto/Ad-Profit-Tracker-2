We're building an ad network sync dashboard. For this step, I need you to build the error monitoring and retry management UI. There are no standalone page routes for this feature — everything renders as panels and sections that integrate into the existing dashboard.

Here's what to build:

**Circuit Breaker Status Panel** — This is the main health overview widget. It calls `GET /api/errors/circuit-breaker/status` on mount and shows a summary row at the top: how many networks are healthy vs. degraded (from the `summary.networksHealthy` and `summary.networksDegraded` fields). Below that, render one status card per network (exoclick, rollerads, zeydoo, propush). Each card shows: the network name, a green/red indicator for whether the circuit is open or closed, the current `retryCount`, and — if the circuit is open — when it opened (`openedAt`) and when it will auto-reset (`autoResetAt`). If `isAutoResetPending` is true, show a subtle badge saying 'Auto-reset pending'. Each open circuit card should have a 'Reset Circuit' button that calls `POST /api/errors/circuit-breaker/reset` with the network ID. After a successful reset, refresh the panel data. If the reset returns 429, show a message like 'Too many resets — try again later.' If it returns 400 with 'Circuit breaker is not open', show that message inline.

**Error Log Panel** — A paginated table of recent sync failures. Calls `GET /api/errors/log` with optional filters. Show filter controls at the top: a network dropdown (all four networks plus 'All'), an error code filter input, and a date range picker (startDate/endDate). The table columns are: timestamp (`createdAt`), network (`networkId`), error message, error category (the human-readable one like 'transient', 'auth', 'data', 'infrastructure'), attempt number, and backoff delay. Paginate using the `nextCursor` field — show a 'Load more' button when `hasMore` is true. When filters change, reset to the first page.

**Retry State Panel** — Shows per-network retry state. Calls `GET /api/errors/retry-state` (no networkId filter — fetch all). For each network, display: current `retryCount`, `nextRetryAt` (formatted as a human-readable countdown using `timeUntilNextRetrySeconds`), `lastErrorCode`, `lastSyncStatus`, and whether it's retryable (`isRetryable`). If `circuitBreakerOpen` is true, show a warning banner instead of the retry info.

**Error Summary Widget** — A compact stats widget for the dashboard header area. Calls `GET /api/errors/summary` with the last 7 days as the default range. Show: overall success rate as a percentage, total failures, and a small per-network breakdown table with each network's `successRate` and `failureCount`. Highlight the `mostProblematicNetwork` with a warning color. Include a date range selector that re-fetches when changed.

**Retry Config Panel** — Calls `GET /api/errors/retry-config` to show each network's current settings. Display `retryAttempts`, `timeoutSeconds`, and the computed `backoffSchedule` as a small table (attempt number, delay, jitter range). Add an edit button per network that opens an inline form to update `retryAttempts` (1–5) and `timeoutSeconds` (5–60). Submitting calls `PATCH /api/errors/retry-config`. Show validation errors inline. On 429, show a rate limit warning.

All panels should handle loading skeletons, empty states, and error states gracefully.
