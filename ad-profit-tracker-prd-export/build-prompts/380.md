We're building the ROI Calculation Engine. For this step, I need you to make sure every error state is handled gracefully — both in the API endpoints and in the ROIIndicatorSection UI.

**Server-side error handling**

For GET /api/roi/compute and GET /api/roi/breakdown: if the Firestore `getDocs` call throws, catch it and return a 500 with a generic message like 'Failed to retrieve data. Please try again.' Log the full error server-side. Never let an unhandled Firestore error bubble up as a 500 with stack trace details.

For PATCH /api/roi/thresholds: if the Firestore write fails, catch it and return 500. If the audit log write fails, log the failure but don't return 500 to the user — the threshold update itself succeeded, so don't penalize the user for an audit log issue. Return 200 with the updated thresholds.

For all four endpoints: make sure the ROI formula never produces `Infinity` or `NaN`. The null-safe check (`totalCost > 0 ? (netProfit / totalCost * 100) : null`) must be in place. Add a safety net: after computing roi, check `if (!isFinite(roi)) return null` before including it in the response.

**Client-side error handling in ROIIndicatorSection**

The ROI card has six distinct error states — make sure each one renders correctly:

- **Loading**: shimmer placeholder matching the card dimensions, directional arrow hidden. Use a CSS shimmer animation (not a spinner) so it matches the card shape.
- **Empty / no data (404 from GET /api/roi/compute)**: show 'No data for selected range' with an amber N/A badge.
- **error_404 from GET /api/dashboard/metrics**: show 'N/A' with a tooltip 'No data found for this date range'.
- **error_500**: show an error banner inside the ROI card (not a full-page error) with the message 'Unable to load ROI data' and a 'Retry' button. The retry button calls `refetch()` from `useROIMetrics`. Don't show a full-page error — the rest of the dashboard should remain usable.
- **error_401**: call `getIdToken(true)` to force a token refresh. If the refresh succeeds, automatically retry the failed request. If the refresh fails (user is truly logged out), redirect to / and show a 'Session expired' toast. This should happen transparently — the user shouldn't see a raw 401 error message.
- **error_403**: show 'Access Denied' inline within the ROI card with a link to /dashboard (in case they navigated here from a deep link).

For the per-network ROI rows (from GET /api/roi/breakdown): if this call fails, show a subtle 'Network breakdown unavailable' message in place of the rows — don't let this failure affect the main ROI card above it.

For the threshold settings panel: if GET /api/roi/thresholds returns 404 (user document not found), show a message 'Using system defaults' and pre-fill the form with the default values so the user can still save their preferences. If PATCH /api/roi/thresholds returns 429, show 'Too many updates — please wait before saving again' and disable the save button for 60 seconds.

**Toast notifications**

Make sure the following actions trigger toast notifications:
- Successful threshold save → success toast 'ROI thresholds updated'
- Failed threshold save → error toast 'Failed to save thresholds. Please try again.'
- Session expired (401 refresh failure) → warning toast 'Session expired. Please sign in again.'
- PATCH 429 rate limit → warning toast 'Too many updates. Please wait before saving again.'
