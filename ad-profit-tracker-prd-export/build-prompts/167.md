We need three API endpoints. Here's what each one does and how to build it:

**GET /api/sync/live-state** — This is the main polling endpoint that the SyncStatusPanel calls every 5 or 30 seconds. It accepts an optional `networkId` query param (must be one of `exoclick`, `rollerads`, `zeydoo`, `propush` — return 400 if it's anything else). Start by verifying the Firebase ID token from the `Authorization: Bearer <token>` header using `adminAuth.verifyIdToken()` — return 401 if it's missing or invalid, and always derive `uid` from the verified token, never from the request body or query params. Then run three Firestore queries using `adminDb`: fetch the matching `networkConfigs` documents for this user, fetch the most recent `adStats` document per network (ordered by `syncedAt DESC`, limit 1), and fetch the most recent `auditLogs` entry per network where `resourceType=='sync'` and `action=='manual_sync_triggered'` (ordered by `createdAt DESC`, limit 1). From these, compute `syncPhase` (derived from `lastSyncStatus`), `isCurrentlySyncing`, and `timeUntilNextScheduledSync`. Then compute `overallHealth`: `'healthy'` if all active networks have `lastSyncStatus` in `['success', 'never']` and no `circuitBreakerOpen`; `'degraded'` if any network has `lastSyncStatus=='failed'` but no circuit open; `'critical'` if any `circuitBreakerOpen==true`. Return the full `LiveStateResponse` shape with a `polledAt` timestamp.

**GET /api/sync/activity-feed** — Returns a paginated list of recent sync events from `auditLogs`. Verify the token the same way. Validate `limit` (integer 1–20, default 10) and optional `networkId`. Query `auditLogs` filtered by `userId==uid AND resourceType=='sync' AND action=='manual_sync_triggered'`, optionally also filtered by `resourceId==networkId`, ordered by `createdAt DESC`. Use cursor-based pagination: if a `cursor` query param is provided, use Firestore's `startAfter()` with the document snapshot for that ID. Fetch `limit + 1` documents to determine `hasMore`. For each document, derive `eventLabel` from `metadata.scheduledSync`, `metadata.triggeredBy` — map to `'Scheduled Sync'`, `'Manual Sync'`, `'Backfill'`, or `'Retry'`. Return the `ActivityFeedResponse` shape.

**PATCH /api/sync/progress** — This is an internal endpoint called by sync helper functions (like `lib/sync/syncNetworkForUser`), not directly by users. It supports dual authentication: check for an `Authorization: Bearer` header first (verify as a Firebase ID token to get `uid`), and if that's absent, check for an `x-internal-secret` header and validate it against the `INTERNAL_SYNC_SECRET` environment variable. Return 401 if neither is valid. Validate `networkId` and `phase` from the request body — return 400 if either is invalid. For user-authenticated calls, verify the `networkConfigs` document exists for that `uid + networkId` before writing. For internal-secret calls, accept a `userId` from the request body. Update the `networkConfigs` document: set `currentSyncPhase`, `currentSyncRowsFetched`, `currentSyncTotalRows`, and `updatedAt` (use `serverTimestamp()`). Do not write an audit log for this — it's a high-frequency internal state update. Return `{ networkId, phase, updatedAt }`.
