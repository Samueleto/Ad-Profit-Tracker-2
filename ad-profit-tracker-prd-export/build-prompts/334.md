# Step 334 of 437: Ensure nav components only render for authenticated users

The mobile nav components are purely client-side and presentational, but they still have a few security surfaces worth locking down.

**Auth guard on nav components** — MobileNavigationDrawer reads `Firebase Auth currentUser` to display the user's avatar and display name. Make sure it handles the case where `currentUser` is null — don't render the user info section at all if there's no authenticated user, and don't crash. The component should also listen to `onAuthStateChanged` so if the session expires while the drawer is open, it reacts immediately rather than showing stale user data. When `onAuthStateChanged` fires with a null user, redirect to `/` and show a 'Session expired' toast notification.

**Sign-out button** — The sign-out button in MobileNavigationDrawer calls `Firebase Auth signOut()`. Make sure this is the only thing it does — don't clear any other state manually, don't make any API calls. After `signOut()` resolves, the `onAuthStateChanged` listener in the auth state layer will handle the redirect automatically. Disable the sign-out button while the sign-out is in progress to prevent double-taps.

**No sensitive data in nav** — The drawer and bottom nav should only ever display the user's display name and avatar URL from `currentUser` — both of which are already public within the authenticated session. Don't read or display anything from Firestore or any API in these nav components. The Zustand store health state read by MobileBottomNavBar is already in memory from the dashboard — no additional data fetching happens here.

**Middleware dependency** — The ResponsiveAppLayout itself doesn't need to enforce auth — that's already handled by `middleware.ts` from the auth feature, which redirects unauthenticated requests before the layout ever renders. Document this dependency clearly in the layout file with a comment so future developers don't accidentally remove the middleware thinking the layout is unprotected.
