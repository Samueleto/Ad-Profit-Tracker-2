Now let's build all four API endpoints. Each one lives in its own `route.ts` file under `src/app/api/rate-limits/`.

**GET /api/rate-limits/config** (`config/route.ts`) — Start with this one since it's the simplest. Verify the Firebase ID token from the `Authorization: Bearer <token>` header using `admin.auth().verifyIdToken()` — return 401 if missing or invalid. Then read the static config from `lib/rate-limit/limiterConfig.ts` and `lib/rate-limit/userLimiterConfig.ts`. No Firestore reads needed. Build human-readable descriptions from the config values (e.g., 'ExoClick: max 10 requests/minute, 1 concurrent'). Return the `networkLimiters` and `userQuotaConfig` arrays matching the response shape.

**GET /api/rate-limits/status** (`status/route.ts`) — Verify the token and get `uid`. Validate the optional `networkId` query param against `VALID_NETWORKS` — return 400 if it's provided but invalid. For each target network (the one specified, or all four), call `getLimiter(uid, networkId)` from `limiterRegistry.ts` and extract `running`, `queued`, and `reservoir` from the Bottleneck instance's current state. Also read `circuitBreakerOpen` and `retryCount` from the `networkConfigs` Firestore collection filtered by `userId == uid`. For user quotas, call `getRemainingQuota(uid, endpoint)` for each of the five rate-limited endpoints and compute `resetAt` from the oldest timestamp in the sliding window. Compute `isThrottled` as true if `reservoir == 0` OR `circuitBreakerOpen == true`. Return the full consolidated response with `networks`, `userQuotas`, and `polledAt`.

**POST /api/rate-limits/reset** (`reset/route.ts`) — This one has dual-auth. Check for `Authorization` header first — if present, verify the Firebase token and get `uid`. If not, check for `x-internal-secret` header and compare it to `process.env.INTERNAL_SYNC_SECRET`. Return 401 if neither is valid. Validate `networkId` and `scope` from the request body — return 400 for invalid values. For user-authenticated calls, check the reset rate limit (10/hour) using the sliding window counter — return 429 if exceeded. If scope includes `'outbound'`, call `limiter.updateSettings({ reservoir: limiterConfig[networkId].reservoir })` on the Bottleneck instance to refill the token bucket. If scope includes `'user-quota'`, call `clearQuota(uid, endpoint)` for all endpoints associated with that network. For user-authenticated calls only, write an audit log entry to the `auditLogs` Firestore collection via the Admin SDK. Return the confirmation response.

**GET /api/rate-limits/violations** (`violations/route.ts`) — Verify the token and get `uid`. Validate `networkId` (if provided) and `limit` (must be 1–50, default 20). Query the `auditLogs` Firestore collection: filter `userId == uid AND resourceType == 'rate_limit' AND action == 'rate_limit_exceeded'`. If `networkId` is provided, also filter `resourceId == networkId`. Order by `createdAt DESC`. If a `cursor` query param is provided, use `startAfter` with that document ID for pagination. Fetch `limit + 1` documents to determine `hasMore`. If an `endpoint` filter is provided, apply it in-memory on `metadata.endpoint`. Map each document to the `ViolationRecord` shape. Return `violations`, `hasMore`, and `nextCursor`.
