We're building a RollerAds ad network integration. For this step, I need you to make sure every failure mode across all six endpoints is handled cleanly — with the right HTTP status, a useful error message, and no sensitive data leaking out.

For POST /api/networks/rollerads/sync, handle these cases distinctly:
- 400 with 'Invalid date range' if dateFrom/dateTo are malformed, dateFrom is after dateTo, or the range exceeds 90 days
- 400 with 'RollerAds network is disabled' if the networkConfigs document has isActive=false
- 404 with 'No RollerAds API key saved' if the apiKeys document doesn't exist for this user
- 429 with 'Rate limit exceeded — max 10 syncs per hour' if the rate limit is hit
- 502 with 'RollerAds API error: [message]' if the axios call fails or RollerAds returns a non-2xx response — include the RollerAds error message but never the API key
- 500 with 'Sync failed' for decryption failures or Firestore write failures — log the real error server-side but don't expose internals to the client

For all three GET stats endpoints (GET /api/networks/rollerads/stats, GET /api/networks/rollerads/stats/by-country, GET /api/networks/rollerads/stats/latest), handle:
- 400 with specific messages for each validation failure (missing dateFrom, invalid format, range too large, invalid limit)
- 404 from GET /api/networks/rollerads/stats/latest when no adStats documents exist yet — the client should show a 'No data yet — run a sync first' message
- 500 for Firestore read failures — log server-side, return a generic message to the client

For GET /api/networks/rollerads/raw-response:
- 400 if the date param is missing or malformed
- 404 if no adStats documents exist for that date — the client should show 'No data for this date'
- 500 for Firestore failures

For POST /api/networks/rollerads/scheduled-sync, since this is internal and called by a scheduler:
- Log all per-user failures with userId and error details, but don't abort the whole batch — continue processing other users
- Return the summary object (processed, succeeded, failed, skipped) even when some users fail
- Return 500 only for catastrophic failures like the initial Firestore query failing

On the frontend side, map each HTTP status to a user-friendly message. A 502 should say something like 'Could not reach RollerAds — check your API key or try again later', not 'Bad Gateway'. A 404 on latest stats should prompt the user to run their first sync.
