We need to make sure the EmailAlertPreferencesSection fails gracefully across all the different things that can go wrong.

**Loading errors on GET /api/notifications/preferences:**
- 401 — Attempt a Firebase ID token refresh via getIdToken(true). If the refresh succeeds, retry the request once. If it fails, redirect to the root route (/) with a 'Session expired' toast notification.
- 403 — Show an 'Access Denied' inline message within the section itself (not a full-page error). Don't redirect.
- 404 — Show a 'Preferences not found — defaults applied' message with a 'Reset to defaults' button that re-calls GET /api/notifications/preferences.
- 500 — Show an inline error banner inside the section with a 'Retry' button that re-calls GET /api/notifications/preferences. Don't crash the whole SettingsPage.

**Per-toggle save errors on PATCH /api/notifications/preferences:**
- If a toggle PATCH fails, roll back the optimistic update and show a small error indicator (e.g., a red '!' icon) next to that specific row. Don't show a full-page error — the user should be able to retry just that toggle.
- 401 — Same token refresh flow as above.
- 429 — Show a brief 'Too many changes — slow down' message near the toggle row.
- 500 — Show 'Failed to save — tap to retry' next to the row.

**Delivery email save errors on PATCH /api/notifications/preferences:**
- Validation failure (400) — Show an inline error below the email field: 'Please enter a valid email address'.
- 500 — Show 'Failed to save email address' below the field with a retry option.

**Test email errors on POST /api/email/send-test:**
- 400 (no delivery email configured) — Show an inline message: 'Add a delivery email address above before sending a test'.
- 429 — Show 'You've sent too many test emails. Try again in an hour.' below the button.
- 502 — Show 'Test email failed to send — the email provider may be temporarily unavailable. Try again shortly.'
- 500 — Show 'Something went wrong. Please try again.'

**Email log errors on GET /api/email/log:**
- If the log fails to load, show a subtle 'Could not load email history' message inside the collapsed section. Don't block the rest of the preferences UI.

All error messages should be plain English — no error codes, no stack traces, nothing technical visible to the user.
