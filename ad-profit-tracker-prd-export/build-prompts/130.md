We need TypeScript interfaces and types that represent all the data structures this feature works with. These will be used by all six endpoints and any future UI components.

Start with the AnomalyFlag type — this is the core data structure that gets stored in the anomalyFlags array on adStats documents. It should have: type (string describing the anomaly kind, e.g. 'revenue_out_of_range', 'ctr_too_high', 'duplicate_record', 'stale_data', 'missing_field', 'cross_network_mismatch'), field (which field triggered it), expectedRange (a [min, max] tuple or null), actualValue (any), severity ('warning' | 'critical'), and optionally a resolution object (resolvedAt timestamp, resolution string, note string or null, resolvedBy uid string) that gets added when PATCH /api/reconciliation/resolve is called.

For the adStats collection extensions, create an AdStatsValidation interface with the five new fields: validationStatus ('valid' | 'anomaly' | 'missing' | 'stale'), validationCheckedAt (Firestore Timestamp), anomalyFlags (AnomalyFlag[]), isReconciled (boolean), reconciledAt (Firestore Timestamp or null).

For the networkConfigs collection extensions, create a NetworkConfigValidation interface with: lastValidationStatus ('clean' | 'anomalies_detected' | 'missing_data' | 'never_run'), lastValidationAt (Firestore Timestamp or null), anomalyCount (number), validationRules (ValidationRules map or null).

Create a ValidationRules interface for the per-network thresholds: maxDailyRevenueUSD, minDailyRevenueUSD, maxDailyCostUSD, maxCtrPercent, minImpressions (all numbers), allowNullCountry (boolean). Also define a SYSTEM_DEFAULTS constant with the default values (50000, 0, 50000, 50, 0, true respectively).

Finally, create response types for each endpoint's success response shape — ReconciliationRunResult, ReconciliationStatus, AnomalyListResponse, ResolveResult, RulesResponse — matching exactly the response shapes documented in the endpoint specs. This makes the route handlers type-safe and self-documenting.
