We need clean state management for the ActivityLogTab so the component itself stays focused on rendering. Build a set of hooks that handle all the data fetching, filter state, and pagination logic.

**useAuditLogs hook** — This is the main hook. It should manage:
- The current list of log entries (accumulated across pages for 'load more' behavior)
- Active filter state: selectedActions (string array), dateRange ({ startDate, endDate } or a preset key like '7d', '30d', '90d'), searchText, and status filter
- Pagination state: the current nextCursor from the last response, hasMore boolean, and current page count
- Loading states: initialLoading (first fetch) vs. loadingMore (subsequent pages)
- Error state

When any filter changes, reset the accumulated log list and cursor back to the start and re-fetch from GET /api/audit/logs. When 'load more' is triggered, fetch the next page using the stored cursor and append results to the existing list — don't replace them.

The hook should expose: logs, initialLoading, loadingMore, hasMore, error, activeFilters, setActionFilter, setDateRange, setSearchText, setStatusFilter, resetFilters, loadMore, and refresh.

For the search filter specifically — apply it client-side against the current page's results (since Firestore doesn't support full-text search natively). The search param is still passed to the API but the UI should also filter the displayed rows locally against the metadata field stringified.

**useExportLogs hook** — Manages the export flow. Tracks isExporting and exportError. When triggered, calls GET /api/audit/logs/export with the current active filters (passed in as a parameter), receives the CSV blob, and programmatically triggers a browser file download with the filename pattern 'activity-log-{YYYY-MM-DD}.csv'. Use date-fns to format the date in the filename.

**useClearLogs hook** — Manages the clear flow. Tracks isClearing and clearError. When triggered, calls DELETE /api/audit/logs/clear. On success, returns the deletedCount so the UI can show it in a toast. The hook should not handle the confirmation dialog — that stays in the component.

All three hooks need to get a fresh Firebase ID token before each API call using firebase/auth's currentUser.getIdToken(). Don't cache the token between calls — always fetch a fresh one so it doesn't expire mid-session.

Keep all business logic (token fetching, query param building, cursor management, CSV download triggering) inside these hooks. The ActivityLogTab component should only call hook methods and render the returned state.
