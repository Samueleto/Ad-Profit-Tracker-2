Now we need to connect the notification system to the rest of the app so everything works together seamlessly.

First, add the NotificationBellIcon component to the top navigation bar of the dashboard layout. It should sit in the nav's right-side action area. The bell icon reads unreadCount from the Zustand dashboard store and renders the badge. Clicking it calls the openPanel action from the store.

Second, render the NotificationCenterPanel in the dashboard layout (not inside a specific page) so it overlays correctly regardless of which sub-page the user is on within /dashboard. The panel reads isOpen from the store and renders with a fixed/absolute position over the page content with a backdrop. This means the panel and bell icon are available on every route under /dashboard without any per-page setup.

Third, wire up the navigation actions inside the panel. When a user clicks a sync_failure or circuit_breaker_opened notification, the panel should close (call closePanel) and then scroll to or highlight the ManualRefreshPanel section on /dashboard. Use a URL hash or a scroll-to-element approach — coordinate with however ManualRefreshPanel identifies itself in the DOM. When a user clicks a schedule_failed or reconciliation_anomaly notification, close the panel and use Next.js router.push('/reports').

Fourth, connect the POST /api/notifications/create endpoint for internal use. Other features in the app (sync jobs, export completions, schedule deliveries) will call this endpoint server-side using the x-internal-secret header to create notifications. Make sure the INTERNAL_SYNC_SECRET environment variable is documented in .env.example so other developers know to set it. You don't need to build the callers here — just make sure the endpoint is reachable and the secret is configured.

Fifth, wire up the notification preferences endpoints. The GET /api/notifications/preferences and PATCH /api/notifications/preferences endpoints exist but don't have a UI yet — that's fine for now. Just make sure they're accessible and that the NotificationCenterPanel respects the preferences: if a notification type has enabled=false in the user's notificationPreferences (from the users document), the POST /api/notifications/create endpoint already handles not writing it, so no client-side filtering is needed.

Finally, make sure the dashboard layout's mount effect reads unreadNotificationCount from the users Firestore document (as set up in state management) so the badge is populated before the user ever opens the panel.
