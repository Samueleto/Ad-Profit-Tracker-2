We need to keep the pages clean by pulling all the data-fetching and config state into dedicated hooks. Here's what to build:

**useReportConfig hook**

This manages the entire left-panel configuration as a single state object. It should track:
- `metrics`: string[] — selected metric keys, defaults to all 8
- `dateRangePreset`: one of the 6 preset strings, defaults to 'last_30'
- `dateFrom` / `dateTo`: strings or null (only set when preset is 'custom')
- `networks`: string[] — selected networks, defaults to all 4
- `groupBy`: 'daily' | 'country' | 'network', defaults to 'daily'
- `dataQuality`: 'all' | 'anomalies' | 'clean', defaults to 'all'

Expose individual setters for each field plus a `resetConfig` function that restores defaults. Also expose a derived `isConfigValid` boolean — true when metrics is non-empty, networks is non-empty, and the date range is valid (custom ranges must have both dates set and be within 90 days). The Export and Save buttons should read `isConfigValid` to decide whether to enable themselves.

Also expose a `configAsApiParams` derived value that transforms the current config into the query params shape expected by `GET /api/filters/stats` and `POST /api/reports/run`.

**useReportPreview hook**

This handles the live preview. It takes the current config from `useReportConfig` and:
- Debounces config changes by 300ms before firing
- Calls `GET /api/filters/stats` with the current config params
- Tracks state: `rows`, `loading`, `error`, `errorStatus` (the HTTP status code so the page can render the right error state)
- Exposes a `refetch` function for the Retry button
- On a successful response, also triggers a call to `GET /api/export/preview` to update the row count badges — store those badge counts separately as `previewCounts`

**useRunReport hook**

This handles the 'Run Report' button action. It calls `POST /api/reports/run` with the full config. Tracks: `rows`, `summary`, `loading`, `error`, `hasMore`, `nextCursor`. Expose a `loadMore` function that re-calls the endpoint with the next cursor appended. When a run completes successfully, the preview table should switch to showing the run results instead of the live preview results — manage this with a `mode` flag ('preview' | 'run') in the parent page.

**useSavedReports hook**

Manages the saved reports sidebar. On mount, calls `GET /api/reports` and stores the list. Exposes:
- `reports`, `loading`, `error`
- `saveReport(name, config)` — calls `POST /api/reports/save`, appends the returned report to the local list on success, shows a toast
- `renameReport(id, name)` — calls `PATCH /api/reports/:id`, updates the local list optimistically
- `deleteReport(id)` — calls `DELETE /api/reports/:id`, removes from local list optimistically (restore on failure)
- `loadReport(report)` — takes a saved report object and calls the `useReportConfig` setter to apply all its fields to the config panel, then triggers a preview refresh

All API calls should attach the Firebase ID token as a Bearer token in the Authorization header. Create a shared `getAuthHeader()` utility that calls `firebase.auth().currentUser?.getIdToken()` and returns the header object — use this in every hook rather than duplicating the token logic.

Keep all these hooks in a `features/reports/hooks/` folder and compose them together in the CustomReportBuilderPage.
