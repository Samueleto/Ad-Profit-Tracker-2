We need TypeScript types for the PDF export endpoint's request body and for the internal data structures we'll compute during PDF generation. This keeps the endpoint code clean and catches mistakes at compile time.

Create a types file (e.g., `lib/export/pdfExportTypes.ts` or alongside the route) with the following:

**PdfExportRequest** — The validated request body shape:
- `dateFrom: string` (YYYY-MM-DD)
- `dateTo: string` (YYYY-MM-DD)
- `sections: PdfSection[]` where `PdfSection` is a union type of the 9 allowed string literals: `'cover_page' | 'executive_summary' | 'daily_trend' | 'geo_breakdown' | 'exoclick' | 'rollerads' | 'zeydoo' | 'propush' | 'activity_log'`
- `orientation: 'portrait' | 'landscape'`
- `paperSize: 'a4' | 'letter'`
- `filename?: string`

**AdStatRecord** — Shape of a document from the `adStats` Firestore collection. Include the fields the PDF endpoint actually uses: `userId`, `date` (string YYYY-MM-DD), `networkId` (one of the four network strings), `country` (ISO code), `revenue` (number), `cost` (number), `impressions` (number), `clicks` (number). Add any other fields already on the collection.

**AuditLogRecord** — Shape of a document from the `auditLogs` collection: `userId`, `createdAt` (Firestore Timestamp or string), `action`, `resourceType`, `resourceId`, `metadata` (object), `status`.

**ExportKpis** — The computed summary values: `totalRevenue`, `totalCost`, `netProfit`, `roi` (number or null).

Also export the `ALLOWED_SECTIONS` constant array — this will be used in the endpoint for validation so the allowed values are defined in one place.

If the Excel export feature already defines some of these types (like AdStatRecord or AuditLogRecord), import and reuse them rather than duplicating.
