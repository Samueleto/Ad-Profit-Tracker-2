We need four API route handlers for the RBAC system. All four live under src/app/api/rbac/ and all use the verifyAuthToken() helper and Firestore instance from the shared lib.

**GET /api/rbac/permissions** — Returns the full role permission matrix for the caller's workspace. Verify the token, get the uid, read the user's Firestore document to get their workspaceId (return 404 if none). Query the workspacePermissions collection for a document where workspaceId matches. If no document exists, return system defaults with isCustomized: false — don't return 404. If a document exists, merge its rolePermissions with system defaults using the mergeWithDefaults helper (fills in any missing keys). Always compute owner permissions as all-true inline — they're never stored. Return the full RolePermissionsMatrix plus customizedRoles array, systemDefaults, and updatedAt. Shape the response as PermissionsMatrixResponse.

**PATCH /api/rbac/permissions** — Updates permissions for a specific role. Verify token, read the caller's workspaceRole from their users document. Enforce the hierarchy: owner can edit admin/member/viewer/analyst; admin can only edit member/viewer; return 403 otherwise. Validate that the role in the request body is one of EDITABLE_ROLES (reject 'owner' with 400). Validate all permission keys against PERMISSION_KEYS. Check a rate limit of 20 updates per hour per user (you can use a simple Firestore counter or an in-memory approach — just make sure it returns 429 when exceeded). Use Firestore set() with merge: true to upsert the workspacePermissions document, updating only the specified role's keys using dot-notation (e.g., rolePermissions.admin.canExportData). If resetToDefaults is true, overwrite the entire role's map with system defaults and remove it from customizedRoles; otherwise add the role to customizedRoles if not already there. Set updatedAt to serverTimestamp() and updatedByUid to the caller's uid. After saving, batch-update effectivePermissions on all users documents where workspaceId matches and workspaceRole matches the changed role. Write an audit log entry to the auditLogs collection. Return the full updated matrix.

**GET /api/rbac/my-permissions** — Returns the caller's own resolved permissions. Verify token, read the user's document. If workspaceRole is 'owner', return all permissions as true immediately without reading workspacePermissions. Otherwise check if effectivePermissions is cached and permissionsUpdatedAt is within the last 5 minutes — if so, return the cached value (fast path). On cache miss, read workspacePermissions, merge with defaults, resolve the user's role-specific map. Kick off a background write to update effectivePermissions and permissionsUpdatedAt on the user's document (don't await it — don't block the response). Return the resolved permissions as MyPermissionsResponse.

**POST /api/rbac/check** — A server-side permission gate. Verify token. Validate that the requested permission is in PERMISSION_KEYS (400 if not). If targetUid is provided, verify the caller is owner or admin (read their workspaceRole) — return 403 if not. Read the target user's document (use targetUid if provided, otherwise the caller's uid). If their role is 'owner', return granted: true with reason 'owner_all_access'. Check effectivePermissions cache (5-minute freshness). On stale cache, resolve from workspacePermissions. Return PermissionCheckResponse with granted boolean, the permission checked, workspaceRole, uid, and reason string.
