We're building a role-based access control UI. For this step, I need you to add a 'Permissions' tab to the existing TeamManagementPage at /settings/team, and build the RolePermissionsEditModal that opens from it.

**PermissionsTab** — This is a new tab inside the existing TeamManagementPage, not a new route. Add it alongside whatever tabs already exist there.

At the very top, render a role badge legend that maps color codes to role names: indigo for Owner, blue for Admin, gray for Member, teal for Viewer. Keep it compact — just colored chips with labels.

Below that, render the main permission matrix table. Columns are the four roles (Owner, Admin, Member, Viewer) with color-coded headers matching the legend. Rows are the ten permission categories: Dashboard View, Export Data, Manage API Keys, Sync Networks, View Reports, Create Reports, Manage Team, Manage Billing, View Audit Logs, Manage Benchmarks. Each cell shows a green checkmark or red X. Make the first column (permission names) sticky so it stays visible when scrolling horizontally on smaller screens. Keep row height compact — this table is dense but should be scannable at a glance. Load this data from GET /api/rbac/permissions.

Below the matrix, add a 'Role Descriptions' accordion that's collapsed by default. When expanded, it shows a plain-language description of each role's intended use case (Owner = full control, Admin = manages team and settings, Member = day-to-day usage, Viewer = read-only access).

Below that, add an 'Effective Permissions' section that calls GET /api/rbac/my-permissions and shows the current user's own resolved permission set. Use a two-column card layout — each permission listed with a granted/denied indicator. This helps members understand exactly what they can and can't do.

If the current user's role is Owner or Admin, show a 'Customize Role Permissions' button that opens the RolePermissionsEditModal. Also show a 'Copy Role' dropdown that lets Owners duplicate an existing role's permission set as a starting point for editing.

Loading state: show shimmer rows in the matrix body while GET /api/rbac/permissions is in flight — render the 4 column headers immediately so the layout doesn't jump.

**RolePermissionsEditModal** — A centered modal (max-w-lg) with a backdrop overlay. It opens when the 'Customize Role Permissions' button is tapped.

At the top, show a segmented control role selector with options: Admin, Member, Viewer. Owner is intentionally excluded — Owner permissions are immutable. The selected role determines which permission set is being edited.

Below that, a scrollable list of all permission keys grouped into five categories: Data Access, Network Management, Team Management, Export & Reports, Settings. Each row shows the permission name, a short description of what it controls, and a toggle switch. Pre-populate toggle values from the GET /api/rbac/permissions response when the modal opens.

If a change would remove a permission that active workspace members currently rely on, show a warning banner inside the modal before they save.

At the bottom: 'Reset to Defaults' button on the left (treat as destructive — style accordingly). 'Cancel' and 'Save Changes' buttons on the right. Save calls PATCH /api/rbac/permissions with the updated permission map for the selected role. On success, show a 'Permissions updated' toast, close the modal, and refresh the parent matrix.

Saving state: disable all toggles and show a spinner on the Save button. Loading state (while GET /api/rbac/permissions resolves on open): show 8 shimmer permission rows in the modal body.
