We're building a user settings and preferences feature for a Next.js + Firebase app. For this step, I need you to extend the existing SettingsPage at /settings by adding a 'General Preferences' card section below the existing API key cards.

The SettingsPage shell is a Server Component. Update it to use Promise.all to fetch both /api/keys/status and /api/settings/preferences in parallel on page load. Pass the preferences data down as props to a new Client Component called PreferencesCard.

The PreferencesCard Client Component should render a card with a 'General Preferences' heading and a visual divider, consistent with the existing API key card style. Inside it, build out four preference controls:

1. **Timezone** — A searchable combobox (not a plain select) that lets users search through IANA timezone strings like 'America/New_York', 'Europe/London', etc. This needs to be discoverable, so filtering as the user types is important. Pre-fill with the saved timezone value, defaulting to 'UTC' if none is set.

2. **Display Currency** — A standard dropdown with common ISO 4217 currency codes: USD, EUR, GBP, JPY, and others. Pre-fill with the saved value, defaulting to 'USD'.

3. **Default Date Range** — A dropdown with exactly these options: Last 7 days, Last 14 days, Last 30 days, This month. Map these to the enum values `last_7_days`, `last_14_days`, `last_30_days`, `this_month`. Pre-fill from saved value, defaulting to `last_7_days`.

4. **Notification Preferences** — Two toggle switches: 'Daily summary email' and 'Weekly report email'. Both default to off. Pre-fill from the saved `notifications.dailySummaryEmail` and `notifications.weeklyReportEmail` boolean values.

Each control should call PATCH /api/settings/preferences immediately on blur (for text/combobox inputs) or on change (for dropdowns and toggles) — no waiting for the Save button for individual fields. At the bottom of the card, include a 'Save Preferences' button that also triggers a full save of the current state.

Handle these UI states:
- **Loading**: Show a skeleton card with 4 rows of shimmer placeholders that match the height and layout of the real controls.
- **Empty / first-time user**: Show all controls with system defaults pre-selected and a subtle 'Defaults applied' label near the top of the card.
- **Submitting**: The 'Save Preferences' button shows a spinner and all controls are disabled while the PATCH request is in flight.
- **Success**: Show a toast notification saying 'Preferences saved' and re-enable all controls.
- **Error states**: Handled separately in error_handling — for now, wire up the error paths to pass errors up to the error handling layer.

The PreferencesCard should accept the preferences data as a prop from the Server Component so the initial render is populated without a client-side fetch waterfall.
