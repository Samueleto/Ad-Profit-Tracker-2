We're building multi-user account support for a workspace-based app. For this step, I need you to build three pieces of UI: a TeamManagementPage integrated as a tab into the existing SettingsPage, an InviteMemberModal triggered from that tab, and a standalone AcceptInvitationPage.

**TeamManagementPage — add as a new 'Team' tab on the existing SettingsPage at /settings**

At the top, add a collapsible 'Workspace Settings' section (collapsed by default, toggled with a chevron). Inside it, show the workspace name as an inline-editable field — clicking the name lets the owner type a new one and save it via PATCH /api/team/workspace. Non-owners see it read-only.

Below that, render a dense member roster table with these columns: avatar, display name, email, a role badge (indigo for Owner, blue for Admin, gray for Member), joined date, and a three-dot action menu per row. The action menu options depend on the viewer's role: Owners see 'Change Role' and 'Remove Member' for everyone except themselves, plus 'Transfer Ownership' for Admin rows. Admins see 'Change Role' (to Member only) and 'Remove Member' for Member rows — they cannot act on other Admins or the Owner. Members see no action menus at all (read-only roster). Role changes call PATCH /api/team/members/:memberId/role. Removal calls DELETE /api/team/members/:memberId.

Below the roster, show a 'Pending Invitations' section with a slightly lighter background to visually separate it. Columns: email, invited by, sent date, expiry, and a 'Revoke' button (Admin/Owner only) that calls DELETE /api/team/invitations/:invitationId.

At the bottom, non-owners see a 'Leave Workspace' button that opens a confirmation dialog before calling DELETE /api/team/members/:memberId (using their own member ID).

An 'Invite Member' button (top right of the roster section) opens the InviteMemberModal. On open, load data from GET /api/team/members and GET /api/team/invitations, and GET /api/team/workspace for the workspace name.

Handle these states: loading shows 4 shimmer rows in the member table and 2 in the pending invitations section; empty (sole member) shows 'You are the only member. Invite your team to collaborate.' with a prominent Invite Member CTA; error_403 shows an 'Access Denied' card with a link to /dashboard; error_404 shows 'Workspace not found' with a link to /dashboard; error_500 shows an inline error banner with a Retry button that re-fetches both endpoints.

**InviteMemberModal — triggered from the 'Invite Member' button on TeamManagementPage**

This is a centered modal (max-w-md) with a backdrop overlay. Show the workspace name at the top so the inviter knows which workspace they're inviting to. The form has three fields: an email input (validate RFC 5322 format on blur), a segmented role selector (Admin / Member — Owner is not an option) with brief permission descriptions as muted helper text below each option, and an optional personal message textarea capped at 200 characters.

The 'Send Invitation' button calls POST /api/team/invitations. On success, close the modal, refresh the pending invitations list on TeamManagementPage, and show a toast: 'Invitation sent to [email]'. A ghost-styled 'Cancel' button closes without action.

Handle inline errors: a 409 response shows 'This person is already a member or has a pending invitation' below the email field. A 429 shows 'Too many invitations sent. Try again in an hour.' A 403 shows an error message inside the modal with a close button. During submission, disable all inputs and show a spinner on the Send button.

**AcceptInvitationPage — standalone page at /invite/accept**

This is a public page (no auth required to view). On mount, read the `token` query param from the URL and call GET /api/team/invitations/validate. While that resolves, show a shimmer placeholder inside a centered card (max-w-sm) on a clean background with no sidebar or nav frame.

If the token is valid, show a card with: the workspace name (prominent), the inviting user's name, the role being offered with a badge and short description, and two CTAs — a filled 'Accept Invitation' button and a ghost 'Decline' button. Accept calls POST /api/team/invitations/accept; Decline calls POST /api/team/invitations/decline.

If the user is not authenticated, replace the two CTAs with a single 'Sign in to accept this invitation' button that redirects to / with a `returnUrl` query param pointing back to this page.

Handle these states: invalid/expired token shows 'This invitation link is invalid or has expired. Please ask your workspace admin to send a new invitation.' with no CTA; already a member shows 'You are already a member of [Workspace]' with a Go to Dashboard link; error_403 shows 'This invitation was sent to a different email address'; error_404 shows 'Invitation not found'; error_500 shows an error banner with Retry. On successful acceptance, redirect to /dashboard with a 'Welcome to [Workspace]!' toast.
