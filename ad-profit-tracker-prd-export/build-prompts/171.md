We're building the ROI Calculation Engine. Now let's build all four API endpoints. Each one lives in `src/app/api/roi/` and uses the shared utilities from `src/lib/roi/`.

**GET /api/roi/compute** — This is the core endpoint. It accepts `dateFrom`, `dateTo`, optional `networkId`, and optional `groupBy` ('total'|'daily'|'country', defaulting to 'total'). After verifying the Firebase ID token with `admin.auth().verifyIdToken()`, validate the date params (required, valid YYYY-MM-DD format, `dateFrom <= dateTo`, max 90-day range) and validate `networkId` against the allowed list `['exoclick','rollerads','zeydoo','propush']`. Check `roiCache` for a hit using the key `{uid}_roi_{dateFrom}_{dateTo}_{networkId||all}_{groupBy}` — if found, return it with an `X-Cache: HIT` header. On a miss, query the `adStats` Firestore collection filtering by `userId==uid`, `date>=dateFrom`, `date<=dateTo`, and optionally `networkId`. Use the `computeRoi` utility from `src/lib/roi/formula.ts`. For the prior-period `roiChange`, shift the date window back by the same number of days and filter the already-fetched documents in memory — no second Firestore query. For `groupBy='daily'` or `groupBy='country'`, group the documents accordingly and compute per-group ROI. Cache the result for 5 minutes and return the full `ROIComputeResponse` shape.

**GET /api/roi/thresholds** — Verify the token, read the `users` Firestore document for the authenticated `uid`, extract `preferences.roiThresholds`, merge with the defaults from `src/lib/roi/defaults.ts` for any missing fields, and return the merged thresholds along with a `usingDefaults` array listing which fields fell back to defaults.

**PATCH /api/roi/thresholds** — Verify the token, validate the request body (all fields optional but at least one required; numeric ranges as specified; threshold ordering: `criticalThreshold <= warningThreshold <= positiveThreshold`). Enforce a rate limit of 20 updates per hour per user — you can track this with a simple in-memory counter or a Firestore timestamp check. Fetch the existing `users` document, apply the partial update using Firestore dot-notation (e.g., `'preferences.roiThresholds.positiveThreshold'`), set `updatedAt` to server timestamp, and write an audit log entry to the `auditLogs` collection recording the changed fields with before/after values. Return the full updated thresholds.

**GET /api/roi/breakdown** — Verify the token, validate `dateFrom`, `dateTo`, and `dimension` ('network'|'country'|'daily'). Check `roiCache` with key `{uid}_roi_breakdown_{dateFrom}_{dateTo}_{dimension}`. On a miss, query `adStats` for the user's date range. Group documents by the requested dimension. For each group, compute ROI using `computeRoi`, assign `roiIndicator` and `colorCode`, and calculate `contributionPercent` as that group's `netProfit` divided by total `netProfit` times 100. For `dimension='country'`, sort by `netProfit` descending. Return the `ROIBreakdownResponse` shape with the `summary` totals.

All four endpoints should return 401 for invalid tokens, 400 for bad inputs, 404 when no `adStats` documents are found, and 500 for Firestore failures.
