# Step 106 of 437: Configure Firestore rules for savedReports field

================================================================================

The Custom Report Builder stores saved report configurations as an embedded array (`savedReports`) on the existing `users` Firestore collection document. We don't need a new collection — reports live inside the user's existing document. But we do need to make sure the Firestore security rules allow the right operations on this field, and that the `auditLogs` collection (written to by all five endpoints) is properly secured.

Update your `firestore.rules` file with the following logic:

**users collection** — Users can read and update their own document (`request.auth.uid == userId`). The `createdAt` field on any embedded report object must be immutable once set — if a report already exists in the array, its `createdAt` cannot be changed. Deletes on the users document itself should not be allowed (reports use soft-delete semantics — the DELETE endpoint uses `arrayRemove` to pull the report out of the array, not delete the whole document). Only authenticated users can create their own users document.

**auditLogs collection** — Only the Firebase Admin SDK (server-side) should write to this collection. Client-side reads and writes should be blocked entirely. Since all writes come from the Admin SDK which bypasses security rules, you just need to ensure no client can read or write audit logs directly: deny all client reads and writes.

Also set up a Firestore **composite index** if needed: the `POST /api/reports/run` endpoint queries the `adStats` collection filtering by `userId`, `date` (range), and optionally `networkId`. If this index doesn't already exist from another feature, add it to `firestore.indexes.json`: fields `userId ASC`, `date ASC`, `networkId ASC`. This prevents Firestore from throwing index errors when the run endpoint fires its query.

Deploy the updated rules and indexes with `firebase deploy --only firestore`.
