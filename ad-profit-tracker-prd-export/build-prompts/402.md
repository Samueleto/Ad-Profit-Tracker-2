# Step 402 of 437: Handle notification errors gracefully in the panel

We need to make sure the NotificationCenterPanel fails gracefully in every scenario without leaving the user confused or stuck.

For the initial load (GET /api/notifications on panel open): show 5 shimmer placeholder rows while loading. If the request succeeds, render the notifications. If it returns 500, show an inline error banner inside the panel — not a full-page error — with the message 'Something went wrong loading your notifications' and a Retry button that re-calls GET /api/notifications. If it returns 401, close the panel immediately, call getIdToken(true) from firebase/auth to force a token refresh, and if that succeeds retry the request silently. If the refresh itself fails (user is truly signed out), redirect to / and show a toast: 'Session expired. Please sign in again.' If it returns 403, show 'Access Denied' inside the panel with a close button. If it returns 404, show 'No notifications found' with a close button.

For dismiss (PATCH /api/notifications/dismiss/:id): the dismiss is optimistic — the row disappears immediately. If the API returns an error, re-insert the notification at its original position in the list and show a brief toast: 'Couldn't dismiss notification. Try again.' Don't show a full error state for a single dismiss failure. If it returns 404 (already dismissed), just leave the optimistic removal in place — the end state is correct. If it returns 429, show a toast: 'Too many dismissals. Please wait a moment.'

For mark all read (PATCH /api/notifications/mark-all-read): optimistic update sets all to read. On failure, roll back the read states and show a toast: 'Couldn't mark notifications as read. Try again.' On 429, show: 'Too many requests. Please wait.'

For clear all (DELETE /api/notifications/clear): this is destructive, so be careful. The optimistic clear empties the list. On failure, restore the full notifications array from a snapshot taken just before the clear and show a toast: 'Couldn't clear notifications. Your notifications have been restored.' On 429, show the rate limit message and don't execute the clear.

For load more (GET /api/notifications with cursor): if this fails, show a small inline error below the last notification row with a Retry button. Don't clear the existing notifications — just fail the append gracefully.

For all 401 responses on any endpoint: always attempt getIdToken(true) first before giving up. Only redirect to / if the refresh itself fails. This handles the common case of a token expiring while the panel is open.

Make sure error toasts are non-blocking (they appear at the bottom of the screen, auto-dismiss after 4 seconds, and don't prevent the user from continuing to use the panel). Use a consistent toast component that's already in the project if one exists.
