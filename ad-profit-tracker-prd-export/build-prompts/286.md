We're building the Help & Documentation Center on Next.js + Firebase. For this step, set up the state management layer that keeps the two pages — HelpCenterPage and HelpArticlePage — clean and free of tangled data-fetching logic.

**HelpCenterPage state**

Create a hook (something like `useHelpCenter`) that manages:
- The full list of articles and category group counts, loaded from GET /api/help/articles on mount
- The currently selected category filter (null means 'All'). When the user clicks a category card, update this and re-call GET /api/help/articles with the category param
- The current search query string. When the query changes, debounce 300ms then call GET /api/help/search. When the query is cleared, go back to showing the full article list
- Search results (separate from the main article list so clearing search restores the previous state)
- The user's workspaceRole, loaded from GET /api/rbac/my-permissions on mount
- Role-specific articles, loaded from GET /api/help/articles?role=[workspaceRole] once the role is known
- Loading states: a general `isLoading` for the initial page load, `isSearching` while a search is in progress, and `isLoadingMore` for pagination
- Pagination state: `hasMore` and `nextCursor` from the API response, and a `loadMore()` function that appends the next page to the existing article list (don't replace it)
- Error state with the HTTP status code so the page can render the right error UI

The Popular Articles list can be derived from the loaded articles sorted by viewCount descending — no extra API call needed.

**HelpArticlePage state**

Create a hook (something like `useHelpArticle`) that takes an articleId and manages:
- The article data, loaded from GET /api/help/articles/:articleId on mount
- Related articles, loaded from GET /api/help/articles?category=[article.category]&limit=4 once the main article loads
- Loading state for the article and separately for related articles
- Error state with HTTP status code (the page needs to distinguish 403 vs 404 vs 500)
- Feedback state: whether the user has already submitted feedback for this article (track this in component state so the buttons disable after submission), and a `submitFeedback(rating)` function that calls POST /api/help/feedback
- The view increment: fire PATCH /api/help/articles/:articleId/view on mount, completely fire-and-forget — no state needed for this

**TOC state**

Create a small hook (something like `useTableOfContents`) that:
- Parses H2 and H3 headings from the rendered article DOM after the markdown renders
- Tracks which heading is currently in the viewport using IntersectionObserver
- Returns the list of headings with their ids and nesting level, and the id of the currently active heading
- Cleans up the IntersectionObserver on unmount

Keep all Firebase ID token fetching (getIdToken()) inside a shared utility so both hooks use the same pattern for attaching the Bearer token to API calls. Don't duplicate that logic in each hook.
