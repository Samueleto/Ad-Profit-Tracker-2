We're building the state management layer for the PreferencesCard Client Component. Since this is a focused, self-contained form within the SettingsPage, we don't need a global store — local React state with a custom hook is the right approach here.

Create a custom hook called `usePreferences` that the PreferencesCard will use. It should accept the initial preferences data (passed as props from the Server Component) and manage everything from there.

Here's what the hook needs to handle:

**Local state** — Track the current values for all four preference fields: timezone (string), currency (string), defaultDateRange (string), and notifications (object with dailySummaryEmail and weeklyReportEmail booleans). Initialize all of these from the props passed in, falling back to defaults (UTC, USD, last_7_days, both notifications false) if any value is missing.

**Save status** — Track a `saveStatus` state that can be one of: `idle`, `saving`, `success`, or `error`. This drives the button spinner, disabled state, toast display, and error banners.

**Auto-save on change** — Expose an `updatePreference` function that takes a field name and new value, updates local state immediately (optimistic), and then calls PATCH /api/settings/preferences with just that changed field. This is what the dropdowns and toggles call on change. Don't debounce toggles and dropdowns — fire immediately. For the timezone combobox, debounce the PATCH call by 400ms after the user stops typing so you're not firing on every keystroke.

**Save all** — Expose a `saveAll` function that sends the full current preferences object to PATCH /api/settings/preferences. This is what the 'Save Preferences' button calls. While this is in flight, set saveStatus to `saving`.

**Conflict handling** — If `updatePreference` is called while a save is already in flight, queue the update rather than dropping it. The simplest approach is to track a `pendingUpdate` ref and flush it after the in-flight request completes.

**Token retrieval** — The hook needs to get the current Firebase ID token to attach to API calls. Use `firebase/auth`'s `getAuth().currentUser.getIdToken()` before each request. Don't cache the token — always fetch a fresh one so it auto-refreshes if near expiry.

**Return values** — The hook should return: the current preference values, `saveStatus`, `updatePreference`, `saveAll`, and any `error` details for the error handling layer to consume.
