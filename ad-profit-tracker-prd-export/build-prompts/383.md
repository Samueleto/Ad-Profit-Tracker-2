The OnboardingWizardPage has several distinct error scenarios that need to be handled gracefully so users don't get stuck or confused.

**401 — Session expired:** If any API call (GET /api/auth/get-user or PATCH /api/auth/update-profile) returns a 401, first attempt to refresh the Firebase ID token silently using firebase/auth's token refresh. If the refresh succeeds, retry the original request once. If the refresh fails (the session is truly expired), redirect the user to the login page (/) and show a 'Session expired, please sign in again' toast notification.

**403 — Access denied:** If a 403 comes back from any endpoint, don't show a generic error. Instead, replace the card content with an 'Access Denied' message and a link to /dashboard. This shouldn't happen in normal usage but is a good safety net.

**404 — User document not found:** If GET /api/auth/get-user returns a 404, it means the user's Firestore document doesn't exist yet (can happen if the sync after registration failed). Automatically call POST /api/auth/sync-user to recreate the document, then retry GET /api/auth/get-user once. If the retry also fails, show the 500 error state. Don't show the 404 error to the user — handle it silently.

**500 — Server error:** Show an inline error banner inside the card (not a full-page error) with a human-readable message like 'Something went wrong. Please try again.' and a 'Retry' button that re-attempts the failed request. The user should not lose their current step position.

**Step 2 submission failure:** If the PATCH /api/auth/update-profile call fails during Step 2, show the error banner inside the card, re-enable the form fields, and reset the Next button from spinner back to its normal state. The user's entered values should be preserved so they don't have to retype everything.

**Network errors:** If a fetch fails due to no network connection (as opposed to a server error response), show a friendly 'Check your internet connection and try again' message with a Retry button.

**Completion PATCH failure:** If the final PATCH (writing onboardingCompletedAt) fails, show the error banner on Step 4 with a Retry button. Don't redirect to /dashboard until this write succeeds — otherwise the user will be stuck in an onboarding loop on their next login.
