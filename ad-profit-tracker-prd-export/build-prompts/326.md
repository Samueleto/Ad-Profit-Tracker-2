We're building a Cloud Run deployment pipeline for a Next.js + Firebase app. For this step, I need you to make sure the three backend endpoints — GET /api/health, GET /api/ready, and GET /api/deployment/status — are properly wired into the application so Cloud Run can use them as intended.

First, make sure GET /api/health and GET /api/ready are reachable without any authentication middleware or route guards in front of them. These are public liveness and readiness probes that Cloud Run's infrastructure calls automatically — if they're accidentally protected, Cloud Run will think the instance is unhealthy and refuse to route traffic to it. Double-check that no global auth middleware is intercepting these routes before the handlers run.

Next, configure Cloud Run to actually use these probes. In your Cloud Run service configuration (either via the GCP console or a service YAML file), set the liveness probe to call GET /api/health and the startup/readiness probe to call GET /api/ready. The health endpoint returns HTTP 200 for healthy/degraded and 503 for unhealthy — Cloud Run uses these status codes to decide whether to keep the instance alive. The ready endpoint returns 200 only when all required environment variables (FIREBASE_PROJECT_ID, INTERNAL_SYNC_SECRET, EMAIL_PROVIDER_API_KEY, EMAIL_FROM_ADDRESS) are present and Firebase Admin SDK is initialized — Cloud Run uses this to decide when to start sending traffic.

For GET /api/deployment/status, this endpoint is for internal admin tooling only. Make sure it's wired up so that any admin dashboard or internal tooling page that needs deployment context can call it with a Firebase ID token in the Authorization header. This endpoint reads Cloud Run runtime environment variables (CLOUD_RUN_SERVICE_NAME, CLOUD_RUN_REGION, K_REVISION, K_SERVICE, K_CONFIGURATION) and returns them — so those variables need to be present in the Cloud Run service's environment configuration. K_REVISION, K_SERVICE, and K_CONFIGURATION are injected automatically by Cloud Run at runtime, but CLOUD_RUN_SERVICE_NAME and CLOUD_RUN_REGION need to be set explicitly in your deployment config.

Finally, make sure the environment variables DEPLOYMENT_ENV (or NODE_ENV) is set correctly per environment (e.g., 'production', 'staging') so the deployment/status endpoint can derive and return the right environment label.
