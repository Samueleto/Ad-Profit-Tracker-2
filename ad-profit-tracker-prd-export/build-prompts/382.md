The ThemeSettingsSection has several distinct error states that need to be handled gracefully. Let's make sure none of them leave the user confused or stuck.

**Loading state** — While GET /api/settings/preferences is in flight, show shimmer placeholder elements in place of the segmented control and the accent color swatches. The live preview strip and the Save button can be visible but disabled during this time. Don't show an empty or broken layout.

**401 — Session expired** — If GET or PATCH returns a 401, attempt `getIdToken(true)` to refresh the Firebase token and retry once. If the retry also returns 401, redirect the user to `/` and show a toast notification saying 'Session expired. Please sign in again.' Don't show this error inline — the redirect is the right UX here.

**403 — Access denied** — Show an inline message within the ThemeSettingsSection that says something like 'You don't have permission to view appearance settings.' Don't redirect. Don't show the controls at all in this state.

**404 — Settings not found** — This means the user's preferences document doesn't exist yet. Don't show an error. Instead, show the controls pre-populated with defaults (Light theme, indigo accent) and add a small, subtle note below the controls: 'No saved settings found — your first save will create them.' The Save button should still work and will create the settings via the PATCH endpoint.

**500 — Server error on GET** — Show an inline error banner at the top of the ThemeSettingsSection with a message like 'Failed to load appearance settings.' Include a 'Retry' button that re-calls GET /api/settings/preferences. Don't show the controls in a broken half-loaded state.

**Save failure (PATCH error)** — If the PATCH fails for any reason other than 401 (which is handled above), show an inline error message below the Save button: 'Failed to save. Please try again.' The controls should stay in their current state — don't revert the optimistic update, since that would be jarring. Let the user try saving again.

**Invalid custom hex** — If the user types an invalid hex value in the custom input, show a small validation message below the input: 'Enter a valid hex color (e.g. #ff6600).' Don't apply the invalid value to the preview or the CSS property. The Save button should be disabled while the hex input is invalid.

**Network offline** — If the fetch or save fails due to a network error (not an HTTP error code), show a toast: 'No internet connection. Check your network and try again.'
