Now we need to connect the sync state management hooks into the existing app so users can actually see and interact with sync status. This feature has no dedicated pages of its own — it surfaces sync information inside whatever dashboard or settings area already exists in the app.

Find the existing dashboard or network settings area where users manage their ad network connections (the place where networkConfigs and apiKeys are already shown). In that area, wire in the `useSyncStatus` hook and display a sync status panel for each of the four networks: exoclick, rollerads, zeydoo, propush. For each network, show: the `lastSyncStatus` badge (color-coded — green for success, red for failed, grey for never), the `lastSyncedAt` timestamp formatted as a human-readable relative time (e.g., '2 hours ago'), the `latestDataDate` so users know how fresh their data is, and the `nextScheduledSync` estimate.

For any network where `lastSyncStatus` is 'failed', show a 'Retry' button that calls `useRetrySync` with that network's ID. While `isRetrying` is true for that network, disable the button and show a spinner. After a successful retry, the status badge should update automatically (because the hook triggers a refresh).

Wire in the `useSyncHistory` hook to show a sync history log below the status panel. Default to showing all networks (no filter). Add a dropdown or tab set to filter by a specific network. Show each history entry with: the network name, status badge, `rowsFetched`, `latencyMs`, whether it was a `scheduledSync` or manual retry, and the `createdAt` timestamp. Add a 'Load More' button at the bottom that calls `loadMore()` — hide it when `hasMore` is false.

Make sure the Cloud Scheduler health endpoint (GET /api/scheduled/health) is not wired into the client UI at all — it's internal only and should never be called from the browser. Similarly, POST /api/scheduled/daily-sync is internal only and must not have any client-side trigger button.

If the app has a notifications or toast system, use it to surface retry results — success toast on a successful retry, error toast on failure.
