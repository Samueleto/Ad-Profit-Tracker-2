The API Explorer is a read-only reference tool, but it deals with sensitive data — raw API responses from ad networks that are fetched using secret API keys stored in the apiKeys and networkConfigs schemas. We need to make sure none of that sensitive data leaks to the client.

The most important rule: all calls to the ad network APIs must happen server-side. The endpoints /api/networks/exoclick/raw-response, /api/networks/rollerads/raw-response, /api/networks/zeydoo/raw-response, /api/networks/propush/raw-response, and /api/networks/config/test-connection should be Next.js API routes (or Firebase callable functions) that read the API keys from server-side environment variables or Firebase secrets — never from anything accessible to the browser. The client only ever sees the response data, never the keys used to fetch it.

Verify that:
1. The raw-response endpoints check that the requesting user is authenticated (using Firebase Auth session verification server-side) before doing anything. Return 401 if not authenticated.
2. The test-connection endpoint (/api/networks/config/test-connection) also requires authentication and should be rate-limited or at minimum logged — it triggers a live external API call, which could be abused.
3. The JSON responses returned to the client should be the raw ad network data only. Strip out or never include any fields that contain API keys, secrets, or internal credentials before sending the response.
4. The apiKeys and networkConfigs data used to check 'is this network configured' on the client side should only expose a boolean (configured: true/false) — not the actual key values. Make sure the configuration check endpoint or Firestore security rules enforce this.
5. Since this tab is inside the authenticated FinancialMetricsDashboardPage, confirm the parent page's Firebase Auth guard is in place and that there's no way to directly access the ApiExplorerTab component or its data endpoints without being logged in.
