We need node-cache set up as a singleton so all three API endpoints share the same in-memory cache instance. In Next.js, API routes can be re-imported on every request, so we need to be careful to only create one cache instance for the lifetime of the server process.

Install `node-cache` via npm or yarn.

In `src/lib/cache/cacheProvider.ts`, create and export a single node-cache instance. Use a module-level singleton pattern — check if the instance already exists on the global object (a common Next.js pattern to survive hot reloads in development) before creating a new one. Configure it with `stdTTL` set to 3600 (one hour default), `checkperiod` set to 120 (check for expired keys every 2 minutes), and `useClones` set to false for better performance since we're storing serialized data.

Also set up the `cacheKeys.ts` utility with helper functions:
- `buildStatsKey(uid: string, network: string, params: object): string` — creates a namespaced key like `{uid}_stats_{network}_{hash of params}`
- `getUserKeys(uid: string): string[]` — returns all cache keys that belong to a specific user by filtering on the UID prefix

These helpers will be used by all three endpoints to ensure users can only see and invalidate their own cached data.

Finally, make sure the Firebase Admin SDK is already initialized somewhere in the project (likely `src/lib/firebase/admin.ts`) since all three endpoints will need it to verify ID tokens. If it's not set up yet, initialize it using the `FIREBASE_SERVICE_ACCOUNT` environment variable.
