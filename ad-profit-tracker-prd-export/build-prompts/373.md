The DateRangeToolbar needs to handle several distinct error states without breaking the rest of the dashboard. The toolbar should always remain functional — errors in the data availability dot or API calls should never disable the preset controls.

Here's how each error state should behave:

**401 — Session expired** — If GET /api/stats/dates returns 401, attempt `getIdToken(true)` to force a token refresh and retry the request once. If the retry also returns 401 (refresh failed), redirect to / and show a 'Session expired' toast notification. The toolbar should not show an error state during the brief refresh attempt — keep the spinner going.

**403 — Access denied** — Show an 'Access Denied' message inline within the toolbar itself (not a toast, not a redirect). The preset controls should remain visible and interactive, but the data availability dot should be replaced with the access denied message. This is an unusual state — log it so you can investigate.

**404 — No data for range** — The data availability dot shows gray with a 'No data for this range' label. This is not an error in the traditional sense — it's a valid response meaning the selected range has no records. The toolbar stays fully functional.

**500 — Server error** — The data availability dot shows an amber warning icon. Do not show a full error message — the toolbar should remain usable. The user can still switch presets and try again, which will re-trigger the GET /api/stats/dates call.

**Network failure** — If the fetch itself fails (no response at all), treat it like a 500: show the amber warning icon on the dot. Don't crash the toolbar.

**Custom date validation error** — If the user selects a 'To' date before the 'From' date, or clears one of the inputs, show an inline validation message below the date inputs: 'Please select a valid date range'. Disable the Apply button. This is separate from the 90-day cap error ('Maximum range is 90 days') — both messages should be distinct.

**Loading state** — While GET /api/stats/dates is in flight, the data availability dot shows a small gray spinner. The preset controls are never disabled during loading.

All error states for the data availability dot should update the `dataAvailability` field in the dashboardDateRangeStore so other parts of the dashboard can react if needed.
