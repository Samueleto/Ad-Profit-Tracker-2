We're building a scheduled report delivery feature for an ad analytics app. For this step, I need you to build the ScheduleReportModal — a centered modal dialog (max-w-md) that opens when a user clicks the clock/calendar icon on a saved report row in the CustomReportBuilderPage's saved reports sidebar panel. This modal is the entire UI for creating, editing, and deleting scheduled email delivery for a specific saved report.

Here's how the modal should be laid out and behave:

**Header:** Show the saved report's name at the top as read-only context. This is just informational — users can't change it here.

**Frequency selector:** A segmented control with three options — Daily, Weekly, and Monthly. When Weekly is selected, show a day-of-week picker (Mon–Sun, 0–6). When Monthly is selected, show a day-of-month picker (1–28). Below the frequency selector, show a muted 'Next delivery' preview label that computes and displays the next run timestamp based on the selected frequency, delivery hour, and timezone (e.g., 'Tomorrow at 8:00 AM EST' or 'Next Monday at 8:00 AM').

**Delivery email field:** Pre-fill this with the user's Firebase Auth email. Make it editable so they can send to any valid email address.

**Date range and format row:** Put these side by side in a two-column layout. The date range is a dropdown with options: Yesterday, Last 7 days, Last 30 days, Last 90 days, This month — mapping to the values 'yesterday', 'last_7', 'last_30', 'last_90', 'this_month'. The format toggle lets users pick Excel (.xlsx) or PDF.

**Active toggle:** At the bottom of the form, a simple on/off toggle to enable or disable the schedule without deleting it.

**Advanced options:** Collapse a timezone override field under a chevron (progressive disclosure). This lets users override their default timezone with any IANA timezone string.

**Footer buttons:** Destructive 'Delete Schedule' button at the bottom-left (only show this when an existing schedule is loaded — not for new schedules). 'Save Schedule' button at the bottom-right. Also include a 'Send Test Now' button that triggers an immediate delivery preview.

**Data loading:** When the modal opens, call GET /api/schedules with the reportId as a query param to check if a schedule already exists. If one does, pre-populate all form fields with the existing schedule's values. If none exists, show defaults: Daily frequency, user's email, Last 7 days, Excel format, active=true.

**Saving:** The 'Save Schedule' button calls POST /api/schedules/create for new schedules, or PATCH /api/schedules/:id for existing ones. Use the schedule's id from the loaded data to know which path to take.

**Deleting:** The 'Delete Schedule' button should show a confirmation dialog before calling DELETE /api/schedules/:id. Don't delete without confirmation — this is destructive.

**Test send:** The 'Send Test Now' button calls POST /api/schedules/:id/send-now. This is only available when an existing schedule is loaded (has an id).

**States to handle:**
- Loading: show 5 shimmer placeholder rows while GET /api/schedules resolves
- Saving: show a spinner on the Save button and disable all inputs
- Empty (no existing schedule): show the default form values described above
- Success: show a 'Schedule saved' toast and close the modal
- Error states: handled in the error_handling step, but wire up the state slots now
