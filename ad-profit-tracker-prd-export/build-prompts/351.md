All eight stats endpoints need to be properly secured. Let's make sure nothing slips through.

**Authentication on every endpoint** — Every single endpoint (GET /api/stats/snapshot, GET /api/stats/trend, GET /api/stats/summary, GET /api/stats/geo-breakdown, GET /api/stats/coverage, POST /api/stats/backfill, DELETE /api/stats/snapshot, GET /api/stats/dates) must verify the Firebase ID token using admin.auth().verifyIdToken() before doing anything else. If the token is missing, malformed, or expired, return 401 immediately — don't proceed to any database reads or writes. The uid must always come from the verified token, never from a query param or request body.

**User scoping on all Firestore queries** — Every query against the adStats, networkConfigs, apiKeys, and auditLogs collections must include a userId == uid filter using the uid from the verified token. This is the most critical security control — without it, a user could read or delete another user's data. Double-check that this filter is present on every getDocs call across all eight endpoints, including the audit log queries in GET /api/stats/coverage.

**Rate limiting** — POST /api/stats/backfill is capped at 3 requests per hour per user (backfills are expensive multi-day operations). DELETE /api/stats/snapshot is capped at 5 deletes per hour per user (deletions are irreversible). Implement these rate limits using a server-side counter — store the count and window start time in Firestore or in-memory (Firestore is safer for serverless). Return 429 with a Retry-After header when the limit is exceeded. The other six GET endpoints don't need rate limiting beyond what Firebase/Next.js provides by default, but make sure they can't be used to enumerate other users' data.

**Input validation** — All date params (date, dateFrom, dateTo) must be validated as proper YYYY-MM-DD strings before being used in Firestore queries. The max 90-day range must be enforced on all endpoints that accept a date range. networkId must be validated against the allowed list ['exoclick', 'rollerads', 'zeydoo', 'propush'] wherever it's accepted. metric must be validated against its allowed values. limit on geo-breakdown must be an integer between 1 and 100. Return 400 for any invalid input — never pass unvalidated user input into a Firestore query.

**Audit logging for destructive operations** — POST /api/stats/backfill and DELETE /api/stats/snapshot both write to the auditLogs collection. Make sure these writes are scoped to the authenticated uid and include enough metadata (deletedCount, date, networkId, triggeredBy) to reconstruct what happened. The audit log write should happen even if the main operation partially fails.

**API key safety** — POST /api/stats/backfill calls the shared syncNetworkForUser helper which decrypts API keys from the apiKeys collection. Make sure the decrypted key values never appear in logs, error messages, or API responses. The endpoint should only return rowsFetched counts and status strings — never key material.
