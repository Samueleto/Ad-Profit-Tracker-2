We're building state management for the geographic performance breakdown feature. The goal is to keep the GeoBreakdownSection and GeoCountryDrilldownModal in sync with the parent dashboard's shared state, while managing their own local UI state cleanly.

**Shared date range** — GeoBreakdownSection must NOT own its own date range. It should read the date range directly from the existing FinancialMetricsDashboardPage shared state (whatever mechanism the dashboard already uses — context, Zustand, Redux, or prop drilling). Before making any API call, cap the date range at 90 days client-side and reject anything longer.

**GeoBreakdownSection local state** — This component needs to track: the selected metric toggle value (Revenue / Cost / Profit, default Revenue), the selected Top N value (10 / 20 / 50, default 10), the country search query string, which country rows have their Network Breakdown sub-row expanded (a set of country codes), whether the GEO Insights strip is expanded, the raw geo table data from GET /api/stats/geo-breakdown, the ROI color enrichment data from GET /api/roi/breakdown with dimension='country', loading/error states for both API calls, and which country was clicked (to pass to the modal).

When the date range changes in the parent dashboard, GeoBreakdownSection should automatically re-fetch both GET /api/stats/geo-breakdown and GET /api/roi/breakdown. Both calls should fire in parallel (Promise.all or equivalent) so the table doesn't load in two visual stages. After fetching, merge the colorCode from the ROI breakdown response into each country row before rendering.

The country search filter and Top N selector should operate purely on the already-fetched data — no re-fetch needed when these change. The metric toggle changes the sort order of the already-fetched data client-side.

**GeoCountryDrilldownModal local state** — This component needs to track: the selected country code and name (passed in as props from GeoBreakdownSection), loading/error states, and the fetched daily snapshot data from GET /api/stats/snapshot. When the modal opens (country code changes from null to a value), it should immediately fetch data from GET /api/stats/snapshot with the country filter and the active date range. The modal should not cache data between opens — always re-fetch when a new country is selected.

For the daily trend chart, you'll need to aggregate the snapshot data into an array of { date, netProfit } objects sorted by date for recharts to consume.

Make sure all API calls attach the Firebase ID token as a Bearer token in the Authorization header. Fetch the token using Firebase Auth's getIdToken() before each call.
