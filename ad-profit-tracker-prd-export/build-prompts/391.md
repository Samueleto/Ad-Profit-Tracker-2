We need consistent, user-friendly error handling across all eight endpoints and their corresponding frontend hooks.

**Server-side error responses** — Each endpoint should return structured error objects, not raw strings. Use a consistent shape like { error: string, code: string } so the frontend can handle them predictably. For example: { error: 'No data found for this date', code: 'NOT_FOUND' } or { error: 'Date range cannot exceed 90 days', code: 'INVALID_RANGE' }. This applies to all 400, 401, 404, 429, and 500 responses across GET /api/stats/snapshot, GET /api/stats/trend, GET /api/stats/summary, GET /api/stats/geo-breakdown, GET /api/stats/coverage, POST /api/stats/backfill, DELETE /api/stats/snapshot, and GET /api/stats/dates.

**Firestore failure handling** — All getDocs calls can fail. Wrap every Firestore read in a try/catch. If a Firestore read fails, return 500 with a generic message — don't expose internal error details like collection names or document IDs in the response. Log the full error server-side for debugging. For POST /api/stats/backfill, use Promise.allSettled() so one network's failure doesn't block the others — collect each network's error into the failed array in the response.

**Frontend hook error states** — Each hook (useSnapshot, useTrend, useSummary, useGeoBreakdown, useCoverage, useDates, useBackfill, useDeleteSnapshot) should catch fetch errors and populate their error state. Distinguish between these cases so the UI can respond appropriately:
- 401 errors → the user's session has expired. The hook should signal this so the app can redirect to login.
- 404 errors → no data exists for the requested params. This is not a crash — show an empty state, not an error banner.
- 429 errors → rate limit hit. Show a message with how long to wait (parse the Retry-After header if present).
- 400 errors → bad params, likely a bug. Show a generic error.
- 500 errors → server problem. Show a 'something went wrong, try again' message with a retry option.
- Network errors (fetch failed entirely) → show 'check your connection' with a retry.

**Empty states vs errors** — Make sure 404 responses from GET /api/stats/snapshot, GET /api/stats/trend, GET /api/stats/summary, GET /api/stats/geo-breakdown, and GET /api/stats/dates are treated as empty states in the UI, not error states. The user simply hasn't synced data for that date range yet — that's normal and expected, not a failure.

**Backfill partial failures** — POST /api/stats/backfill can partially succeed (some networks succeed, others fail). The hook should surface this clearly — don't treat a partial failure as a total failure. Show which networks succeeded (triggered array), which were skipped (no API key), and which failed (failed array with errorMessage).

**Delete confirmation** — After DELETE /api/stats/snapshot succeeds, the useDeleteSnapshot hook should surface the deletedCount so the UI can show 'Deleted 47 records for 2024-01-15'. If the delete returns 404, show 'No data found to delete' rather than a generic error.
