We need to configure Firestore so our team queries run efficiently and the data is locked down correctly. This covers two things: composite indexes for the queries we'll be running, and security rules for the workspaceInvitations collection.

Indexes — The team feature runs several Firestore queries that need composite indexes to work:
1. Fetching all workspace members: query on the users collection filtering by workspaceId. Add a composite index on users: workspaceId (ascending) + workspaceJoinedAt (ascending) — this powers GET /api/team/members.
2. Fetching pending invitations for a workspace: query on workspaceInvitations filtering by workspaceId and status. Add a composite index on workspaceInvitations: workspaceId (ascending) + status (ascending) + createdAt (descending).
3. Looking up an invitation by tokenHash: add a single-field index on workspaceInvitations.tokenHash (this is used when validating and accepting invitations).
4. Looking up invitations by invitedEmail + workspaceId: add a composite index on workspaceInvitations: workspaceId (ascending) + invitedEmail (ascending) + status (ascending) — used for duplicate detection when sending invitations.

Security rules — Update your Firestore security rules for the workspaceInvitations collection:
- Read: any authenticated user can read an invitation document (needed so the AcceptInvitationPage can validate a token without being a workspace member yet)
- Create: any authenticated user can create an invitation (the server-side API enforces role checks — the rules just need to allow the Admin SDK to write)
- Update: only the document owner (matching invitedByUid or the accepting user) can update — but since all writes go through the Admin SDK on the server, you can scope this to admin writes only using request.auth != null as a baseline
- The createdAt field should be immutable — reject updates that try to change it
- Delete: authenticated users only (revocation goes through the API which uses Admin SDK)

Note: since all mutations go through Firebase Admin SDK on the server, the security rules are a safety net rather than the primary enforcement layer. The real role checks happen in the API routes.
