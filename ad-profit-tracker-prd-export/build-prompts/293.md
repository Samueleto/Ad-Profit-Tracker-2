Now that the DateRangeToolbar and the Zustand store exist, we need to make sure every active tab and section of FinancialMetricsDashboardPage actually responds when the date range changes.

Here's what needs to be wired up:

**Dashboard sections re-fetching** — Every section on FinancialMetricsDashboardPage that displays time-sensitive data (KPI cards, charts, tables, etc.) should subscribe to `startDate` and `endDate` from the dashboardDateRangeStore. Whenever either value changes, those sections should automatically re-fetch their data with the new range. Use a `useEffect` that depends on `startDate` and `endDate` in each section, or a shared custom hook (e.g., `useDashboardDateRange`) that returns the current range and triggers re-fetches.

**Data availability dot** — The DateRangeToolbar should call GET /api/stats/dates whenever the active date range changes (both preset and custom). Pass the current startDate and endDate as query params. Attach the Firebase Bearer token. On response, call `setDataAvailability` on the store with the appropriate status. Debounce this call by ~300ms so rapidly switching presets doesn't fire multiple requests.

**Last synced badge** — Whichever dashboard section receives a `lastSynced` timestamp in its API response should call `setLastSynced` on the store so the DateRangeToolbar badge stays current. If multiple sections return different timestamps, always keep the most recent one.

**Apply button flow** — For the Custom preset specifically, make sure the Apply button in DateRangeToolbar calls `setCustomRange` on the store only after passing the 90-day validation check client-side. The store update should then cascade to all subscribed sections automatically.

**Tab switching** — If FinancialMetricsDashboardPage has multiple tabs, make sure switching tabs doesn't reset the date range. The store should persist the selection across tab changes within the same session.
