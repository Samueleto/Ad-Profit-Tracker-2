We need state management for two distinct concerns: the benchmarking tab's data and the settings modal's form state.

**Benchmarking Tab State** — Create a custom hook (something like `useBenchmarkingData`) that manages everything the PerformanceBenchmarkingTab needs. It should track:
- The currently selected metric (default 'roi'), with a setter that triggers a re-fetch
- The comparison data from GET /api/networks/comparison
- The benchmark performance data from GET /api/benchmarks/performance (both historical and industry types)
- The performance score from GET /api/benchmarks/score
- Loading, error, and empty states for each data source
- A `hasLoaded` flag so the tab knows whether to show the lazy-load placeholder or actual content

The hook should accept the date range (dateFrom, dateTo) from the parent dashboard's shared state as parameters — when those change, it should re-fetch. It should NOT fetch until the tab is first activated (the caller controls this by only mounting the hook when the tab is active, or by passing an `enabled` flag).

When the selected metric changes, re-fetch GET /api/benchmarks/performance with the new metric value. The comparison data from GET /api/networks/comparison doesn't need to re-fetch on metric change — it returns all networks regardless.

For the parallel fetch, use Promise.all so both requests fire simultaneously. Store the results separately so a failure in one doesn't wipe out the other.

Also track per-network `networkStatus` (lastSyncStatus, circuitBreakerOpen) so the tab can show stale-data badges. This likely comes from the comparison endpoint response — make sure it's surfaced.

**Benchmark Settings Modal State** — Create a separate hook (something like `useBenchmarkSettings`) that manages the modal's form. It should track:
- The current saved settings (fetched from GET /api/benchmarks/settings on modal open)
- The in-progress form values (a local copy the user is editing)
- A `isDirty` flag indicating whether any field has changed from the saved state
- Per-metric validation errors
- Loading state (fetching current settings)
- Saving state (PATCH in progress)
- A `lastUpdatedAt` timestamp from the fetched settings

The hook should expose: a function to update a single metric's customTarget or useDefault toggle, a function to reset a single metric to its system default, a function to reset all metrics, a submit function that calls PATCH /api/benchmarks/settings with only the changed fields, and a reset function that discards local changes and reverts to the last saved state (used by the Cancel button).

Validation should run on every field change, not just on submit — so the user sees inline errors immediately. The submit function should also validate before firing the PATCH.

Both hooks should attach the Firebase ID token as a Bearer token in the Authorization header on every API call. Pull the token from the existing Firebase Auth context — don't re-implement auth here.
