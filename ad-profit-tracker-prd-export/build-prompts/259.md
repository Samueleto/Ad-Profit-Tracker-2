# Step 259 of 437: Set up state management for team data

We need state management to keep the TeamManagementPage, InviteMemberModal, and AcceptInvitationPage from doing raw data fetching inline. The goal is to keep all the API calls and derived state in one place so the UI components just read data and call actions.

**Team Members state**

Create a hook (something like `useTeamMembers`) that manages the full lifecycle of the member roster. It should: fetch members from GET /api/team/members on mount (attaching the Firebase ID token as a Bearer token in the Authorization header), expose the members array, a loading boolean, and any error. It should also expose a `refresh` function so the TeamManagementPage can re-fetch after a role change or removal. Internally, sort members by workspaceJoinedAt ascending so the owner always appears first.

**Workspace state**

Create a hook (something like `useWorkspace`) that fetches workspace metadata from GET /api/team/workspace. It should expose workspaceName, ownerUid, memberCount, currentUserRole, and a loading state. It should also expose an `updateWorkspaceName` function that calls PATCH /api/team/workspace and optimistically updates the local name before the server responds — rolling back if the request fails.

**Pending Invitations state**

Create a hook (something like `useTeamInvitations`) that fetches pending invitations from GET /api/team/invitations. Expose the invitations array, loading state, error, and a `refresh` function. The InviteMemberModal should call this refresh after a successful POST /api/team/invitations so the list updates without a full page reload.

**Invitation Validation state (for AcceptInvitationPage)**

Create a hook (something like `useInvitationValidation`) that takes the token from the URL query param and calls GET /api/team/invitations/validate on mount. It should expose: the invitation details (workspaceName, invitedByName, role), a loading state, and an error type (so the page can render the right error state — invalid token, already member, etc.). Also expose `acceptInvitation` and `declineInvitation` functions that call POST /api/team/invitations/accept and POST /api/team/invitations/decline respectively, each returning a promise so the page can handle the redirect on success.

**Auth token helper**

All four hooks need to attach the current user's Firebase ID token to every request. Create a shared utility function (something like `getAuthHeaders`) that calls `getIdToken()` from firebase/auth on the current user and returns `{ Authorization: 'Bearer [token]' }`. If there's no current user, it should throw so the hooks can catch it and set an auth error state. This keeps token-fetching logic in one place rather than duplicated across every hook.

Make sure none of these hooks store sensitive data (like raw tokens) in component state — tokens should only live in memory during the fetch call.
