The client-side gates we built are for UX — the real security lives in the API. Let's make sure all four RBAC endpoints are locked down correctly.

**GET /api/rbac/permissions** — Every request must include a valid Firebase ID token in the Authorization header. Verify it with `admin.auth().verifyIdToken(token)` — never trust a uid from the request body or query params. The workspaceId must be derived from the verified uid's users Firestore document, never from the request. Any authenticated workspace member can read the permission matrix (this is intentional — members need to understand what roles can do). Return 401 for invalid/expired tokens, 404 if the user has no workspace.

**PATCH /api/rbac/permissions** — Same token verification. After verifying the token, read the actor's `workspaceRole` from their users document. Enforce the hierarchy strictly: only 'owner' can edit admin/member/viewer/analyst permissions; 'admin' can only edit 'member' and 'viewer'. If the request targets 'owner' as the role to edit, return 400 immediately — Owner permissions are immutable and this should never be allowed. Validate every permission key in the incoming map against the exact list of 9 allowed keys (canViewDashboard, canExportData, canManageApiKeys, canSyncNetworks, canViewReports, canCreateReports, canManageTeam, canViewAuditLogs, canManageBenchmarks) — reject any unknown keys with 400. Rate limit to 20 permission updates per hour per user and return 429 if exceeded. After a successful update, batch-update effectivePermissions on all users with the affected role so the cache stays consistent.

**GET /api/rbac/my-permissions** — Token verification required. Users can only read their own permissions — the uid always comes from the verified token, never from a query param. If the user is an Owner, short-circuit and return all permissions as true without reading workspacePermissions. Use the 5-minute cache on effectivePermissions to avoid unnecessary Firestore reads, but update the cache in the background after a miss.

**POST /api/rbac/check** — Token verification required. Validate the permission key against the allowed list. If targetUid is provided, verify the actor is Owner or Admin before allowing a cross-user check — return 403 otherwise. This endpoint is used as an internal gate by other API routes, so it needs to be fast and reliable. Never expose the reason for denial beyond what's needed (don't leak whether a user exists via 404 vs 403 differences).

**Firestore security rules** — Make sure the workspacePermissions collection rules are tight: any authenticated user can read (to support the matrix view), but only the workspace owner (where `request.auth.uid == resource.data.workspaceId`) can write. The users collection effectivePermissions field should only be writable by the server (Admin SDK bypasses rules, but make sure client-side writes can't touch these fields). Double-check that createdAt is immutable in both collections.
