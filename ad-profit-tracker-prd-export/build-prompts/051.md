We need three things set up before writing any sync logic:

1. Firebase Admin SDK — This is the backbone of the whole feature. Every endpoint uses it to read from and write to Firestore (networkConfigs, adStats, apiKeys, and auditLogs collections), and two of the endpoints (sync-status and sync-history and retry-failed) also use it to verify Firebase ID tokens. Make sure firebase-admin is initialized as a singleton — initialize it once and reuse the instance across all route handlers. In Next.js this means checking if an app is already initialized before calling initializeApp(). Store the service account credentials in environment variables (FIREBASE_SERVICE_ACCOUNT_JSON or individual FIREBASE_PROJECT_ID, FIREBASE_CLIENT_EMAIL, FIREBASE_PRIVATE_KEY vars) — never hardcode them.

2. axios — We'll use axios to call the four ad network APIs. No global configuration needed here, but set up a reasonable default timeout (30 seconds) so a slow network API doesn't hang the entire sync. Each network client wrapper will create its own axios instance with the right base URL.

3. cryptr — This handles decrypting the API keys that are stored encrypted in Firestore. Initialize cryptr with the ENCRYPTION_SECRET environment variable. Create a thin wrapper at src/lib/sync/cryptoHelper.ts that exports a single decryptApiKey(encryptedKey: string): string function. This wrapper should never log the decrypted value — treat it as a secret at all times.

Also set up the required environment variables. The ones this feature needs are: INTERNAL_SYNC_SECRET (the shared secret that protects the daily-sync and health endpoints from being called by anyone other than Cloud Scheduler), ENCRYPTION_SECRET (for cryptr), and any network-specific API base URLs if they're configurable. Add these to your .env.local for development and document them in a .env.example file.
