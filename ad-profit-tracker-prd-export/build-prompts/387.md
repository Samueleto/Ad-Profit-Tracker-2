We're building an ad network analytics dashboard. For this step, I need you to make sure every error scenario across all six ExoClick endpoints and all five frontend views fails gracefully with clear, helpful feedback.

**Backend error handling:**

For `POST /api/networks/exoclick/sync`: If the ExoClick API is unreachable or returns an error, return 502 with a message like 'ExoClick API returned an error — please try again later'. Before returning, update the `networkConfigs` document with `lastSyncStatus='failed'` and `lastSyncError` set to the error message. If decryption fails (wrong secret, corrupted key), return 500 — never expose the decryption error details in the response. If the Firestore batch write fails partway through, log the failure and return 500 with how many rows were written before the failure.

For `GET /api/networks/exoclick/stats/latest`: If no documents exist yet, return a clean 404 with `{ error: 'no_data', message: 'No ExoClick data found. Run a sync to get started.' }` — not a generic 500.

For `GET /api/networks/exoclick/raw-response`: If no records exist for the requested date, return 404 with `{ error: 'no_data_for_date', message: 'No data found for this date. Try syncing this date range first.' }`.

For all endpoints: wrap Firestore reads in try/catch and return 500 with a generic message if they fail — don't expose Firestore error details to the client.

**Frontend error handling:**

`ExoClickSummaryCard`: If `useExoClickLatest` returns a 404, show the empty state prompt (not an error). If it returns any other error, show an inline error message with a 'Retry' button that calls `refetch()`.

`ExoClickStatsView` and `ExoClickCountryBreakdown`: If the fetch fails, show an error banner above the chart/table with a 'Retry' button. Keep any previously loaded data visible rather than replacing it with an error screen — the user may just have a momentary network blip.

`ExoClickSyncPanel`: Map the backend's HTTP status codes to user-friendly messages:
- 400 (invalid dates) → 'Please check your date range — dates must be valid and within 90 days'
- 400 (network disabled) → 'ExoClick is currently disabled. Enable it in your network settings.'
- 404 (no API key) → 'No ExoClick API key found. Add your API key in the network settings first.'
- 429 (rate limited) → 'You've hit the sync limit (10/hour). Please wait before syncing again.'
- 502 (ExoClick API error) → 'ExoClick's API is currently unavailable. Try again in a few minutes.'
- 500 → 'Something went wrong on our end. Please try again.'

`ExoClickApiExplorer`: If the fetch returns 404, show 'No data for this date — try syncing this date range first' with a button that navigates to the Sync tab. If it returns any other error, show a generic error with a retry option.

**Auth expiry handling**: If any hook receives a 401 response, redirect the user to the login page. Don't show a generic error — a 401 means the session expired.
