# Step 393 of 437: Handle reconciliation errors gracefully

================================================================================

**Category:** Error Handling

The reconciliation feature involves long-running operations and complex validation logic, so there are many ways things can go wrong. We need to make sure every failure mode is handled gracefully.

For POST /api/reconciliation/run: this is the most complex endpoint and the most likely to fail. If the run partially succeeds (some networks validated, others failed), the response should still return 200 with the partial results — don't fail the whole run because one network had a Firestore read error. On the frontend, if a run returns with some networks errored, show a warning banner: 'Reconciliation completed with errors — [network names] could not be checked.' If the endpoint returns 429, show a message: 'You've run too many reconciliations recently. Try again in [time until reset].' If it returns 500, show a retry button.

For GET /api/reconciliation/status: if this fails, the overview panel should show a degraded state rather than a blank screen — show each network card with a 'Status unavailable' placeholder and a retry button. Don't block the rest of the page from loading.

For GET /api/reconciliation/anomalies: if the initial load fails, show a full-panel error with a retry button. If 'Load More' fails, show a small inline error below the list ('Failed to load more — retry') without clearing the already-loaded records.

For PATCH /api/reconciliation/resolve: this is a write operation, so failures need careful handling. If the batch write fails, roll back the optimistic removal from the anomaly list (put the records back). Show a clear error: 'Failed to resolve anomalies — your changes were not saved.' If only some documents were skipped (the skipped count > 0 in the response), show an informational message: 'X records were resolved. Y records could not be updated.'

For GET /api/reconciliation/rules and PATCH /api/reconciliation/rules: if loading rules fails, show the system defaults as a read-only fallback with a warning that custom rules couldn't be loaded. If saving rules fails, keep the form in its edited state (don't reset the user's changes) and show an error with a retry option.

For 401 responses across all endpoints: redirect to the login page and show a message: 'Your session has expired. Please sign in again.' Preserve the current URL so the user is returned to the reconciliation page after re-authenticating.

For 400 validation errors: surface the specific field errors from the response body rather than a generic 'bad request' message. For example, if PATCH /api/reconciliation/rules returns a 400 because maxCtrPercent is out of range, highlight that specific field in the rules editor form.
