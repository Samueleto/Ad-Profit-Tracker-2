We're building the ROI Calculation Engine. For this step, I need you to make sure all four ROI endpoints and the client-side token handling are properly secured.

**Server-side security for all four endpoints**

Every endpoint — GET /api/roi/compute, GET /api/roi/thresholds, PATCH /api/roi/thresholds, and GET /api/roi/breakdown — must verify the Firebase ID token as the very first thing it does, before any database reads or writes. Use `admin.auth().verifyIdToken(token)` and extract `uid` from the verified token result. Never trust a `userId` passed in the request body or query params — always use the uid from the verified token.

For the Firestore queries in GET /api/roi/compute and GET /api/roi/breakdown, make sure the `userId == uid` filter is always applied. It should be impossible for one user to retrieve another user's adStats data, even if they craft a request manually.

For PATCH /api/roi/thresholds, the Firestore write must use dot-notation partial updates scoped to `users/{uid}/preferences/roiThresholds` — it should never be possible to overwrite another user's document. Double-check that the audit log entry written to the auditLogs collection includes the uid and the before/after values for each changed field.

For the node-cache keys in GET /api/roi/compute and GET /api/roi/breakdown, confirm the cache key format is `{uid}_roi_{dateFrom}_{dateTo}_{networkId||all}_{groupBy}` and `{uid}_roi_breakdown_{dateFrom}_{dateTo}_{dimension}` respectively — the uid prefix is critical so one user can never get a cache hit from another user's cached data.

**Rate limiting on PATCH /api/roi/thresholds**

The PATCH endpoint needs a rate limit of 20 updates per hour per user. Implement this using a Firestore counter or an in-memory store keyed by uid. Return 429 with a clear message if the limit is exceeded. Make sure the rate limit check happens after auth verification but before any Firestore reads or writes.

**Client-side token handling**

In the `getAuthHeaders()` utility built in the state management step, make sure it always calls `getIdToken()` fresh (not a cached copy) so the token is never expired when sent. If `getIdToken()` throws or returns null, the utility should throw an error that the hooks can catch and handle. Never store the raw ID token in localStorage or sessionStorage — let the Firebase SDK manage token storage.

**Input validation on GET /api/roi/compute and GET /api/roi/breakdown**

Both endpoints accept date range params. Make sure the server validates: dateFrom and dateTo are present, match YYYY-MM-DD format, dateFrom <= dateTo, and the range doesn't exceed 90 days. For GET /api/roi/compute, also validate that networkId (if provided) is one of ['exoclick','rollerads','zeydoo','propush'] and groupBy is one of ['total','daily','country']. For GET /api/roi/breakdown, validate that dimension is one of ['network','country','daily']. Return 400 with a descriptive message for any validation failure — never let invalid params reach the Firestore query.

**Prevent data leakage in responses**

Make sure none of the four endpoints return internal Firestore document IDs, other users' data, or the raw Firebase Admin SDK error messages. Catch all errors and return generic 500 messages to the client while logging the full error server-side.
