The team management feature has a lot of moving parts and several distinct error states. We need to make sure every failure mode is handled gracefully so users aren't left staring at a broken page.

**401 handling — token refresh flow**

When any API call in the team feature returns a 401, the app should automatically try to refresh the Firebase ID token by calling `getIdToken(true)` (force refresh). If the refresh succeeds, retry the original request once with the new token. If the refresh fails (the user's session is truly expired), redirect to / and show a 'Session expired. Please sign in again.' toast. This logic should live in a shared error handler used by all four hooks (useTeamMembers, useWorkspace, useTeamInvitations, useInvitationValidation) so it's not duplicated.

**TeamManagementPage error states**

- error_401: attempt token refresh as above; if it fails, redirect to / with toast
- error_403: replace the page content with an 'Access Denied — you do not have permission to manage this workspace' card with a link to /dashboard. Don't show a partial table.
- error_404: show 'Workspace not found' with a link to /dashboard
- error_500 on initial load: show an inline error banner at the top of the Team tab with a 'Retry' button that re-calls GET /api/team/members and GET /api/team/invitations simultaneously
- error_500 on a mutation (role change, removal, revoke): show a toast notification ('Something went wrong. Please try again.') and do NOT update the local state — the roster should remain as it was before the failed action

**InviteMemberModal error states**

- 409 (duplicate): show 'This person is already a member or has a pending invitation' as inline text below the email field — not a toast, since the user needs to see it while the field is still focused
- 429 (rate limit): show 'Too many invitations sent. Try again in an hour.' as an inline banner inside the modal
- 403: show 'You do not have permission to invite members' inside the modal with a close button
- 404: show 'Workspace not found' inside the modal
- 500: show an inline error banner inside the modal with a 'Retry' button that re-submits the form
- Network error (fetch failed entirely): show 'Check your connection and try again' inside the modal

**AcceptInvitationPage error states**

- Invalid/expired token (404 or 410 from GET /api/team/invitations/validate): show 'This invitation link is invalid or has expired. Please ask your workspace admin to send a new invitation.' with no CTA — don't show the Accept/Decline buttons
- Already a member (specific error from validate): show 'You are already a member of [Workspace]' with a 'Go to Dashboard' link
- error_403 on accept (email mismatch): show 'This invitation was sent to a different email address. Please sign in with the correct account.' with a 'Sign in with a different account' link
- error_500 on accept or decline: show an error banner with a Retry button that re-calls the same action
- Network error: show 'Check your connection and try again' with a Retry button

**General principles**

Mutations that fail should never leave the UI in an inconsistent state — if a role change fails, the badge should still show the old role. If a removal fails, the member should still appear in the roster. Use optimistic updates sparingly and only where you can reliably roll back. For all error messages, use plain language — never show raw error codes or Firestore error messages to the user.
