Now we need to connect the ScheduleReportModal to the rest of the app. The modal doesn't have its own route — it renders as an overlay on the /reports page, triggered from the CustomReportBuilderPage's saved reports sidebar panel.

Here's what needs to be wired up:

**Trigger button:** In the CustomReportBuilderPage's saved reports sidebar, add a clock/calendar icon button to each saved report row. This button should be subtle — don't let it compete with the primary actions on the row. When clicked, it should open the ScheduleReportModal and pass the saved report's id and name into it.

**Modal mounting:** The ScheduleReportModal should be mounted at the CustomReportBuilderPage level (or in a portal at the app root if you prefer), not inside the sidebar list item. This avoids z-index and overflow clipping issues. Pass the selected report's id and name as props, and an onClose callback that clears the selected report and closes the modal.

**Opening flow:** When the clock icon is clicked, set the selected report in local state (reportId + reportName), which causes the modal to render. The modal's useScheduleForm hook should call fetchSchedule(reportId) on mount to check for an existing schedule.

**Closing flow:** The modal closes when: the user clicks the backdrop, presses Escape, the save succeeds, or the delete succeeds. On close, clear the selected report state so the modal unmounts cleanly.

**Success feedback:** When a schedule is saved (POST or PATCH succeeds), show a toast notification — something like 'Schedule saved' with the report name. When a schedule is deleted, show 'Schedule deleted'. When a test email is sent, show 'Test email sent to [email]'. Use whatever toast system is already in the app.

**Visual indicator on the row:** After a schedule is saved, show a small visual indicator on the saved report row (like a filled clock icon or a colored dot) to let users know that report has an active schedule. You can derive this from whether the GET /api/schedules response returned a schedule for that reportId. If the user has multiple saved reports open, you may want to fetch all schedules once (GET /api/schedules without a reportId filter) when the sidebar loads, and use that to mark which rows have schedules.

**No new routes needed:** This is all happening on /reports. Don't create a new page or route for this feature.
