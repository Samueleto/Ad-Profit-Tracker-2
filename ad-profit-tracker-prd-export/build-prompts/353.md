The reconciliation feature touches sensitive financial data — ad revenue, costs, and anomaly flags — so we need to make sure every layer is properly secured.

On the API side, every one of the six endpoints (POST /api/reconciliation/run, GET /api/reconciliation/status, GET /api/reconciliation/anomalies, PATCH /api/reconciliation/resolve, GET /api/reconciliation/rules, PATCH /api/reconciliation/rules) must verify the Firebase ID token using admin.auth().verifyIdToken() before doing anything else. The uid must always come from the verified token — never from the request body or query params. Double-check that none of the endpoints accept a userId parameter from the client.

For the Firestore security rules on the adStats collection, make sure the rules enforce that users can only read and write documents where userId matches their own uid. The validationStatus, anomalyFlags, validationCheckedAt, isReconciled, and reconciledAt fields are written by the Cloud Functions / API routes using the Admin SDK (which bypasses client-side rules), so the client-side rules don't need to allow writes to these fields directly — but they should not allow clients to write them either.

For the networkConfigs collection, the same ownership rules apply. The validationRules map, lastValidationStatus, lastValidationAt, and anomalyCount fields should only be writable server-side via the Admin SDK.

Rate limits are already defined in the endpoint specs — enforce them: 10 reconciliation runs/hour per user on POST /api/reconciliation/run, 20 resolve operations/hour on PATCH /api/reconciliation/resolve, and 20 rule updates/hour on PATCH /api/reconciliation/rules. Implement these using a simple counter stored in Firestore or Firebase RTDB keyed by uid + hour bucket. Return 429 with a clear message when exceeded.

In PATCH /api/reconciliation/resolve, the spec says to verify userId==uid on each document before updating and to silently skip documents that don't belong to the user (rather than returning a 403 for individual docs). Make sure this is implemented correctly — a user should never be able to resolve another user's anomalies, even if they somehow obtain the document IDs.

On the frontend, make sure the Firebase ID token is refreshed before each API call using getIdToken(true) if the token is close to expiry (Firebase tokens expire after 1 hour). Never store the token in localStorage — use the Firebase Auth SDK's in-memory token management. Make sure anomaly flag data (which could contain field values from ad network APIs) is rendered as text only — never passed to eval() or dangerouslySetInnerHTML.
