We're building a real-time sync status monitoring panel that lives inside the existing dashboard. For this step, set up the folder structure so everything has a home before we start writing logic.

Create a sync-status feature area. You'll need folders for:
- A components folder — this is where the SyncStatusPanel and its sub-components will live (network cards, health banner, activity feed, anomaly alert strip)
- A hooks folder — for the polling logic and data-fetching hooks
- A types folder — for the TypeScript interfaces that describe network sync state, activity feed entries, and health status
- A lib or utils folder — for any helper functions like computing relative timestamps, deriving sync phase labels from status strings, and formatting latency

On the backend side, create an API routes folder at `app/api/sync/` (or `pages/api/sync/` depending on whether the project uses the Next.js App Router or Pages Router — check the existing project structure and match it). Inside that folder you'll need route files for three endpoints: `live-state`, `activity-feed`, and `progress`.

Also make sure there's a shared Firebase Admin initialization module somewhere in `lib/` (like `lib/firebaseAdmin.ts`) if one doesn't already exist — all three API routes will import from it.

This feature doesn't need its own page — the SyncStatusPanel integrates into the existing dashboard at `/dashboard`, specifically into the ManualRefreshPanel area. So no new route file is needed for the UI side.
