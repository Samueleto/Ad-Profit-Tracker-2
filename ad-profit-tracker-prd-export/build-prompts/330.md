Let's make sure the SyncStatusPanel and its supporting hooks handle authentication and authorization correctly on the client side, and that the three API endpoints are properly secured.

**Client-Side Token Handling** — Every call to GET /api/sync/live-state, GET /api/sync/activity-feed, GET /api/reconciliation/anomalies, and GET /api/reconciliation/status must include a fresh Firebase ID token in the Authorization header as `Bearer <token>`. Use `getIdToken()` from firebase/auth — call it fresh before each request (Firebase handles caching and refresh internally). Never store the raw token in component state or localStorage. If `getIdToken()` throws (e.g., user is signed out), treat it as a 401 and stop polling.

**PATCH /api/sync/progress Security** — This endpoint is internal — it should only be called by server-side sync helpers (lib/sync/syncNetworkForUser), never directly from the browser. Make sure there's no client-side code that calls PATCH /api/sync/progress. The endpoint uses either a Firebase ID token (for user-initiated syncs) or an `x-internal-secret` header (for scheduled syncs) — the client should never need to touch this endpoint.

**Polling Bounds** — The polling logic must be bounded: always clear intervals on unmount, never start a new interval if one is already running, and don't fire a new fetch if the previous one hasn't completed yet (use an in-flight flag or AbortController). Unbounded polling could cause excessive Firestore reads and unnecessary costs.

**Data Scoping** — All three GET endpoints scope their Firestore queries to the authenticated user's documents using the `uid` from the verified token. On the client, don't pass a userId in the request — the server derives it from the token. If you see any code passing userId as a query param or in the request body for these read endpoints, remove it.

**No Sensitive Data in Logs** — The `lastSyncError` and `lastErrorCode` fields returned by GET /api/sync/live-state may contain internal error details. Don't log these to the browser console in production. Use a conditional log that only runs in development mode.

**Rate Limiting Awareness** — GET /api/sync/live-state is designed for polling and has no rate limit, but the anomalies and reconciliation endpoints may not be. If those calls are included in every poll cycle, consider whether they need to be polled at the same frequency or if they can be fetched less often (e.g., every 5th poll or only on manual refresh).

**403 Handling** — If any endpoint returns 403, show the inline 'Access Denied' banner inside SyncStatusPanel and stop polling. Don't retry 403s — they indicate a permissions issue that won't resolve on its own.
