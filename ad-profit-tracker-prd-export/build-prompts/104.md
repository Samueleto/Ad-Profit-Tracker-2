The export feature reads from two existing Firestore collections — `adStats` and `auditLogs` — so we don't need to create new collections. But we do need to make sure the right composite indexes exist so the queries don't fail or time out.

Here's what the export endpoints query:

**adStats collection:**
- Filter by `userId == uid` AND `date >= dateFrom` AND `date <= dateTo`
- This needs a composite index on `userId` (ascending) + `date` (ascending)

**auditLogs collection:**
- Filter by `userId == uid` AND `createdAt >= dateFrom` AND `createdAt <= dateTo`
- This needs a composite index on `userId` (ascending) + `createdAt` (ascending)

Check your `firestore.indexes.json` file (or the Firebase Console under Firestore → Indexes) to see if these composite indexes already exist. If they don't, add them to `firestore.indexes.json` and deploy with `firebase deploy --only firestore:indexes`.

Also double-check the Firestore security rules to make sure the server-side Admin SDK reads aren't blocked — Admin SDK bypasses rules by default, so this is mainly a sanity check. The export endpoints use `getDocs` (one-time reads, not real-time listeners), which is the right approach for a batch export.

Note: the `adStats` and `auditLogs` schemas are defined elsewhere in the project. This step is just about making sure the database is ready to serve the queries the export endpoints will make.
