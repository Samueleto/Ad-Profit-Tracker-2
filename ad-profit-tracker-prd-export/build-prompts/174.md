We need three API routes for the benchmarking feature. All three live under `app/api/benchmarks/` and all three follow the same auth pattern: read the `Authorization` header, call `admin.auth().verifyIdToken(token)` to get the `uid`, and return 401 immediately if that fails. Never trust a userId from the request body or query params — always use the uid from the verified token.

**1. GET /api/benchmarks/performance** (`app/api/benchmarks/performance/route.ts`)

This is the most complex endpoint. It takes query params: `dateFrom` (YYYY-MM-DD, required), `dateTo` (YYYY-MM-DD, required), `metric` (one of the 7 MetricKeys, default `'roi'`), and optionally `networkId`. Validate all inputs — return 400 if dates are missing, malformed, out of order, or span more than 90 days. Check the node-cache singleton for key `{uid}_benchmarks_performance_{dateFrom}_{dateTo}_{metric}_{networkId||all}` — if it's there, return it immediately with an `X-Cache: HIT` header.

On cache miss, fire three things in parallel via `Promise.all()`: (a) query `adStats` for the current date range filtered to this user (and optionally this networkId), (b) query `adStats` for the 90-day historical window ending at `dateFrom`, and (c) read the user's `benchmarks` document to get their custom target for this metric. If no `adStats` docs are found for the current range, return 404.

From the current-period data, compute the actual metric value using null-safe aggregation (totalRevenue from rollerads+zeydoo+propush, totalCost from exoclick, then derive roi/ctr/cpm etc.). From the historical data, compute the per-day metric values, average them, and return null if fewer than 3 days have data. Load the effective industry benchmark: if the user has `useDefault=false` and a non-null `customTarget`, use that; otherwise use the system defaults from `lib/benchmarks/defaults.ts`. Compute the gaps, performanceRatio, and trend direction. Cache the result for 5 minutes and return the `BenchmarkPerformanceResponse` shape.

**2. GET /api/benchmarks/settings** (`app/api/benchmarks/settings/route.ts`)

Simple read endpoint. After auth, query the `benchmarks` collection for a document where `userId == uid`. If no document exists, that's fine — return the system defaults for all 7 metrics with `isCustom: false` and `useDefault: true` for each. If a document exists, merge its `metricTargets` with system defaults: for each metric, if `useDefault` is true or `customTarget` is null, the `effectiveTarget` is the system default. Return the full `BenchmarkSettingsResponse` shape including `historicalWindowDays` (default 90 if not set) and `lastUpdatedAt`.

**3. PATCH /api/benchmarks/settings** (`app/api/benchmarks/settings/route.ts`)

Upsert endpoint. After auth, validate the request body using the `BenchmarkSettingsPatchBody` type and the validation helpers from `lib/benchmarks/validation.ts`. Return 400 with specific field-level errors if anything is out of range. Return 400 if the body is empty (nothing to update). Check a rate limit of 20 updates per hour per user — you can track this in node-cache with a counter key `{uid}_settings_ratelimit` and a 1-hour TTL; return 429 if exceeded.

Use Firestore Admin SDK `set()` with `merge: true` on the benchmarks document for this uid — this creates the document if it doesn't exist. Set `updatedAt` to `serverTimestamp()`, and set `createdAt` to `serverTimestamp()` only if the document is new (check with `getDoc` first or use a transaction). Write an audit log entry to the `auditLogs` collection with `action='preferences_updated'`, `resourceType='preferences'`, `resourceId='benchmark_settings'`, and metadata listing which metrics changed. Return the full updated settings merged with system defaults.
