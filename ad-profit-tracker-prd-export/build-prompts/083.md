We need to make sure the Firestore security rules are updated to protect the preferences data we're adding to the `users` collection.

The preferences fields live as a nested object inside each user's document at `users/{uid}`. The security rules need to enforce that:
- A user can only read their own document (`request.auth.uid == uid`)
- A user can only update their own document
- Users cannot delete their document outright — deletion should be a soft delete (mark a field like `deletedAt`, don't actually remove the document)
- The `createdAt` field is immutable — once set, it cannot be changed by a client write
- Unauthenticated requests are denied entirely

Note that our API endpoints use the Firebase Admin SDK server-side, which bypasses Firestore security rules entirely. These rules are a defense-in-depth measure for any direct client SDK access. The API routes are the primary access path for this feature, but the rules should still be correct.

Also, since the preferences fields are being added to an existing `users` collection, make sure the rules don't break any existing read/write patterns already in place for that collection. Review the current rules before making changes.

If the project uses Firestore emulators for local development, make sure the updated rules are reflected in `firestore.rules` and that the emulator config in `firebase.json` points to that file.
