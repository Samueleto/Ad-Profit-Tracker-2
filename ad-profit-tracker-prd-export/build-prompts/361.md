The backend endpoints handle most of the security enforcement, but we need to make sure the frontend is doing its part correctly too. For this step, audit and harden how the ScheduleReportModal and useScheduleForm hook interact with the five schedule endpoints.

**Bearer token on every request:** Every call to POST /api/schedules/create, GET /api/schedules, PATCH /api/schedules/:id, DELETE /api/schedules/:id, and POST /api/schedules/:id/send-now must include a fresh Firebase ID token as a Bearer token in the Authorization header. Use getIdToken() from the Firebase Auth SDK — call it fresh before each request rather than caching the token in state or a variable. This ensures you're never sending an expired token.

**Token refresh on 401:** If any of the five endpoints returns a 401, the hook should call getIdToken(true) to force a token refresh and retry the request once. If the retry also returns 401, that means the session has genuinely expired — close the modal and redirect the user to / with a 'Session expired' toast. Don't retry more than once.

**No sensitive data in the request body that shouldn't be there:** The deliveryEmail in POST /api/schedules/create and PATCH /api/schedules/:id comes from the form field. Make sure the hook is sending exactly what the user typed — not pulling it from somewhere else. For POST /api/schedules/:id/send-now, confirm the hook sends NO request body — the delivery email must come from the stored schedule on the server, not from the client.

**Input sanitization before sending:** Before calling saveSchedule(), trim whitespace from the deliveryEmail field. Validate the email format client-side (basic RFC 5322 check) and show an inline error if it's invalid — don't even make the API call if the email looks wrong. This is a UX improvement, not a security replacement (the server validates too).

**Rate limit feedback:** If any endpoint returns 429, show a user-friendly message explaining the limit — for example, 'You've sent too many test emails. Please wait before trying again.' Don't just show a generic error. Map each endpoint's rate limit to a specific message: 10 creates/hour for POST /api/schedules/create, 20 updates/hour for PATCH, 20 deletes/hour for DELETE, 5 manual sends/hour for POST /api/schedules/:id/send-now.

**No schedule id in places it shouldn't be:** The schedule id (UUID) is used in PATCH /api/schedules/:id, DELETE /api/schedules/:id, and POST /api/schedules/:id/send-now. Make sure this id comes from the schedule object returned by GET /api/schedules — never from user input or URL params on the frontend.
