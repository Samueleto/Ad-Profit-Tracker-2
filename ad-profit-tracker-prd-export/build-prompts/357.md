The filter endpoints handle user data and write to user documents, so we need to make sure everything is properly secured. Here's what to verify and harden across GET /api/filters/options, GET /api/filters/stats, GET /api/filters/search, GET /api/filters/presets, POST /api/filters/presets, and DELETE /api/filters/presets/:presetId.

**Authentication on every endpoint**: Every single one of these six endpoints must call admin.auth().verifyIdToken(token) before doing anything else. The uid must always come from the verified token result — never from a query param, request body, or header that the client controls. If the token is missing, malformed, or expired, return 401 immediately.

**Input validation**: All query params and body fields must be validated server-side before use:
- dateFrom/dateTo: valid YYYY-MM-DD format, dateFrom <= dateTo, max 90-day range
- type param on /api/filters/options: must be exactly one of ['country','network','metric','all']
- networks on /api/filters/stats: must be a subset of ['exoclick','rollerads','zeydoo','propush'] — reject any unknown network IDs
- metric: must be one of the allowed metric values — reject anything else
- dataQuality: must be one of ['all','anomalies','clean']
- searchQuery / q: strip leading/trailing whitespace, enforce max 100 chars, reject empty strings after trim
- presetId in DELETE: must be a non-empty string
- POST /api/filters/presets body: name 1–50 chars, countries max 50 ISO codes, all enum fields validated

**Firestore query scoping**: Every Firestore query on adStats must include userId==uid as a filter. Never query adStats without this constraint. For preset reads and writes, always read the users document for the authenticated uid — never accept a userId from the client.

**Cache key namespacing**: All node-cache keys must be prefixed with the uid (e.g., '{uid}_filter_options_...') so one user's cached results can never be served to another user.

**Preset abuse prevention**: POST /api/filters/presets must enforce the 20-preset-per-user cap by reading the current array length before writing. Also enforce the 20 saves/hour and 20 deletes/hour rate limits. Return 429 with a clear message when exceeded.

**Token refresh on the client**: In the FilterToolbar, if any API call returns 401, attempt a Firebase ID token refresh via getIdToken(true). If the refresh succeeds, retry the request once with the new token. If the refresh fails, redirect to / and show a 'Session expired' toast. For 403 responses, show 'Access Denied' inline within the toolbar row — don't redirect.

**No sensitive data in responses**: Double-check that none of the filter endpoint responses include Firebase Admin credentials, internal Firestore document IDs beyond what's needed, or any other user's data.
