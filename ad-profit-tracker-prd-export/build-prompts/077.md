We're using Firebase Admin SDK on the server side for all RBAC operations — verifying ID tokens, reading and writing Firestore documents, and doing batch updates when permissions change. This step makes sure the Admin SDK is properly initialized and ready to use across all four RBAC API routes.

First, check if firebase-admin is already installed in the project. If not, add it. Then create (or extend) a shared Firebase Admin initialization file at src/lib/firebase-admin.ts. This file should initialize the Admin SDK once using a singleton pattern — calling initializeApp() multiple times in a Next.js dev environment causes errors due to hot reloading, so guard the initialization with a check like getApps().length === 0 before calling initializeApp().

The Admin SDK needs credentials. Use a service account: store the service account JSON values as environment variables (FIREBASE_PROJECT_ID, FIREBASE_CLIENT_EMAIL, FIREBASE_PRIVATE_KEY) in .env.local. Never commit the actual service account file. The FIREBASE_PRIVATE_KEY value often has escaped newlines — make sure to handle that when reading it (replace \\n with \n).

Export two things from this file: the initialized admin app, and a Firestore instance (admin.firestore()). All four RBAC API routes — GET /api/rbac/permissions, PATCH /api/rbac/permissions, GET /api/rbac/my-permissions, and POST /api/rbac/check — will import the Firestore instance from here.

Also create a helper function in src/lib/rbac/ called verifyAuthToken(request) that extracts the Bearer token from the Authorization header and calls admin.auth().verifyIdToken(token). It should return the decoded token (with uid) on success, or throw a structured error that the API routes can catch and return as a 401. This avoids repeating the same token verification logic in all four endpoints.
