Now we need to connect the three reconciliation views into the existing app navigation and make sure they're accessible from the right places.

The reconciliation feature should live under a route like /reconciliation (or /dashboard/reconciliation if the app uses a nested dashboard layout). Create a page at that route that renders all three views in a tabbed layout: a 'Overview' tab showing ReconciliationOverviewPanel, an 'Anomalies' tab showing AnomalyListView, and a 'Rules' tab showing ValidationRulesEditor. The tab state should be reflected in the URL as a query param (e.g., ?tab=anomalies) so users can deep-link directly to the anomaly list.

In the app's main navigation sidebar or header (wherever other sections like sync status or network settings live), add a 'Reconciliation' link that goes to this page. If there are any unresolved anomalies, show a badge with the count next to the nav item — pull this from the anomalyCount values across all networks using GET /api/reconciliation/status.

Wire up cross-feature connections:
- If the app has an existing network sync status page or dashboard, add a small 'View Anomalies' link next to each network that deep-links to /reconciliation?tab=anomalies&networkId=<networkId>.
- After a successful reconciliation run from the overview panel, if anomalies were found, show a prompt: 'X anomalies detected — view them now' that navigates to the anomalies tab.
- After resolving anomalies, if the anomaly count drops to zero for a network, show a success toast: 'All anomalies resolved for [network name]'.

Make sure the reconciliation page is protected — only authenticated users should be able to reach it. If the app already has an auth guard on the dashboard layout, the reconciliation route will inherit it. If not, add a redirect to the login page for unauthenticated users.

Finally, make sure the page title and meta tags are set correctly for the reconciliation route so it's identifiable in browser tabs and history.
