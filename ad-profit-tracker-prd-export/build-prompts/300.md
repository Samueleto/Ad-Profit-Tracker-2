We need to connect the OnboardingWizardPage to the rest of the app so new users are automatically routed through it and can't accidentally skip it or revisit it after completion.

**Middleware gating:** In your Next.js middleware (middleware.ts at the project root), add logic that runs on every protected route. After confirming the user is authenticated, check their users document for onboardingCompletedAt. If it's null or absent, redirect them to /onboarding. If it's set, allow them through. Also add the reverse: if an authenticated user with onboardingCompletedAt already set tries to visit /onboarding, redirect them to /dashboard immediately — they've already completed it.

**Post-login redirect:** After a user logs in for the first time (or any login where onboardingCompletedAt is not set), they should land on /onboarding rather than /dashboard. Wire this into wherever your login success handler currently redirects — instead of always going to /dashboard, check the user's onboardingCompletedAt and route accordingly.

**Step 3 routing:** The 'Go to Settings' button on Step 3 of the wizard should route to /settings. Make sure this also triggers the completion PATCH (writing onboardingCompletedAt) before navigating — the user has seen enough of the wizard to be considered onboarded even if they're heading to Settings to set up API keys right away.

**Step 4 routing:** The 'Go to Dashboard' button routes to /dashboard. This should only be clickable after the completion PATCH has succeeded — don't navigate until the write is confirmed.

**Skip link behavior:** The 'Skip for now' link on Step 3 advances to Step 4 (the Done screen) rather than immediately navigating away. The actual navigation happens from Step 4's 'Go to Dashboard' button, which is when the PATCH is written. This way the user always sees the Done screen before leaving the wizard.
