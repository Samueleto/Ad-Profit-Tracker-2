# Step 377 of 437: Handle sync failures and surface errors clearly

This feature involves a lot of moving parts — four external network APIs, Firestore reads and writes, decryption, and parallel fan-out. We need to make sure failures at any layer are caught, logged, and surfaced appropriately without cascading into a total system failure.

On the server side, inside POST /api/scheduled/daily-sync, the parallel fan-out using `Promise.all()` must never let one network's failure crash the entire sync job. Wrap each individual user+network sync task in a try/catch so a failure for one combination (say, exoclick for user A) doesn't prevent zeydoo for user B from completing. After all tasks settle, collect the full summary and return it with the counts of succeeded, failed, and skipped combinations. The endpoint should still return 200 even if some networks failed — the `networksFailed` count in the response tells the caller what happened.

For transient API failures (network timeouts, 5xx from external APIs), the exponential backoff retry logic must kick in: wait 1 second before the first retry, 2 seconds before the second, 4 seconds before the third. After exhausting `retryAttempts` (default 3, from the networkConfigs document), mark the sync as failed, update `lastSyncStatus='failed'` and `lastSyncError` with the error message in the networkConfigs document, and write a failure audit log entry to the auditLogs collection. Never retry on 4xx errors from external APIs — those are permanent failures (bad API key, invalid request) and retrying won't help.

For POST /api/scheduled/retry-failed, if the external network API is unreachable or returns an error after retries, return 502 with a clear message. If Firestore writes fail during the retry, return 500. Make sure the networkConfigs document is always updated (to either 'success' or 'failed') even if the audit log write fails — the status update is more important than the log.

On the client side, the `useRetrySync` hook should distinguish between error types and surface appropriate messages: 429 → 'Rate limit reached — try again in an hour', 404 → 'No API key saved for this network — add one in settings', 502 → 'The network API is currently unreachable — try again later', 400 with 'No failed sync to retry' → 'This network doesn't have a failed sync to retry', 500 → 'Something went wrong on our end — please try again'.

For GET /api/scheduled/health, if Firestore is unreachable or an env var is missing, return 503 (not 500) with the specific failing check identified in the `checks` object so monitoring systems can pinpoint the problem.

Make sure no raw error stack traces or internal error messages are ever returned to the client — log them server-side and return sanitized messages.
