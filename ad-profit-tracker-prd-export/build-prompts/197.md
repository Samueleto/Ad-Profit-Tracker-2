We need several reusable components that the ActivityLogTab will be built from. Build these before the page so the page can import them by name.

**ActionTypeBadge** — A small badge component that takes an `action` string (one of the `AUDIT_ACTIONS` values) and renders it as a color-coded label. Use semantic colors: green for success-oriented actions (`api_key_saved`, `network_connection_tested`), red for destructive actions (`api_key_deleted`, `account_deleted`), blue for config actions (`network_config_updated`, `network_config_reordered`, `network_config_reset`, `preferences_updated`, `profile_updated`), and amber for sync actions (`manual_sync_triggered`). The label text should be human-readable (e.g., 'API Key Saved' not 'api_key_saved'). Also accept a `status` prop — if status is 'failure', add a subtle error indicator regardless of action type.

**LogTableRow** — A single table row for the audit log. Takes an `AuditLog` object as a prop. Shows five columns: timestamp (formatted as relative time like '2 hours ago' using `date-fns/formatDistanceToNow`, with the absolute formatted date shown on hover via a tooltip), action type (using `ActionTypeBadge`), resource affected (combine `resourceType` and `resourceId` into something readable like 'ExoClick API Key' or 'Sync'), a details summary (pull the most relevant field from `metadata` — for `changedFields`, show something like 'syncSchedule: daily → hourly'; for other metadata, show a short JSON summary), and an expand chevron. When the row is expanded (toggle on click), show the full `metadata` JSON in a formatted, syntax-highlighted code block inline below the row — no modal, just an accordion. The expanded state is local to the row.

**AuditFilterBar** — The filter controls that sit above the table. Contains: a multi-select dropdown for action types (using the `AUDIT_ACTIONS` list as options), a date range selector with preset options (Last 7 days, Last 30 days, Last 90 days, Custom), a status filter (All / Success / Failure), and a free-text search input (debounced 300ms, max 100 chars). Emits an `onFiltersChange` callback with the current `LogFilters` object whenever any filter changes. Shows a 'Clear filters' link when any filter is active.

**ClearLogsDialog** — A confirmation dialog for the destructive 'Clear All Logs' action. Two-step: first click shows the dialog with a warning message and a 'Yes, delete all logs' button. That button should require the user to type 'DELETE' to confirm (or at minimum, show a second confirmation step). On confirm, calls a provided `onConfirm` callback. Shows a loading state while the delete is in progress. This component is used only in ActivityLogTab but is complex enough to live in its own file.
