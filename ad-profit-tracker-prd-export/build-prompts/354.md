Let's make sure the Financial Metrics Dashboard is properly secured end to end.

**Route-level protection** — Confirm that the Next.js middleware checks for a valid Firebase session cookie or token before serving /dashboard. The check should happen server-side in middleware.ts so the page HTML is never sent to unauthenticated users. Don't rely solely on client-side redirects.

**Token handling on the client** — In useDashboardMetrics, every call to GET /api/dashboard/metrics and POST /api/cache/invalidate must include the Firebase ID token as a Bearer token in the Authorization header. Use getIdToken(false) for normal requests (uses the cached token, avoids unnecessary network calls). Only call getIdToken(true) to force-refresh when you receive a 401 back — and only retry once. If the force-refresh also fails, treat it as a session expiry: clear local state and redirect to / with a 'Session expired' toast message.

**Server-side token verification** — The GET /api/dashboard/metrics endpoint already verifies the token via admin.auth().verifyIdToken(). Confirm this is the first thing the handler does — before any Firestore reads, before any cache lookups. The uid used for all Firestore queries and cache key namespacing must come exclusively from the verified token, never from query params or request body.

**Date range cap** — The 90-day cap must be enforced on both the client (in useDashboardMetrics, before the fetch fires) and the server (in GET /api/dashboard/metrics, returning 400 if exceeded). Client-side validation is for UX; server-side is the real security boundary.

**Cache key namespacing** — Confirm that the node-cache keys in GET /api/dashboard/metrics are namespaced by uid (format: '{uid}_dashboard_metrics_{dateFrom}_{dateTo}'). This prevents one user from ever reading another user's cached dashboard data.

**No sensitive data in responses** — Review the GET /api/dashboard/metrics response shape and confirm it contains no API keys, Firebase service account details, raw Firestore document IDs beyond what's needed, or other user's data. The perNetwork array should only include the authenticated user's network configs.

**POST /api/cache/invalidate protection** — This endpoint should also verify the Firebase ID token and only invalidate cache keys belonging to the authenticated user (scoped by uid). It should not accept a uid in the request body.
