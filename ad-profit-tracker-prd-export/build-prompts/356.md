The benchmarking feature touches three endpoints and one Firestore collection. Let's make sure every layer is properly secured.

**Client-Side Token Attachment** — Every API call from PerformanceBenchmarkingTab and BenchmarkSettingsModal must include the Firebase ID token as a Bearer token in the Authorization header. Pull the token using `getIdToken()` from the existing Firebase Auth context. Don't hardcode tokens or store them in localStorage — always fetch fresh from the auth context before each request.

**Token Refresh on 401** — If any API call returns a 401, the client should attempt a forced token refresh using `getIdToken(true)` (the `true` forces a refresh from Firebase rather than returning a cached token). If the refresh succeeds, retry the original request once with the new token. If the refresh fails (meaning the session is truly expired), redirect to `/` and show a 'Session expired' toast. This logic should live in a shared API utility so it applies consistently to all three endpoints — GET /api/benchmarks/performance, GET /api/benchmarks/settings, and PATCH /api/benchmarks/settings.

**Date Range Cap** — The client should enforce the 90-day maximum date range before sending any request to GET /api/benchmarks/performance. If the parent dashboard's date range exceeds 90 days, clamp it or show a warning — don't let the request go out and fail with a 400. This is a UX guard; the server still validates independently.

**Firestore Security Rules for benchmarks** — Verify the Firestore security rules for the benchmarks collection are correctly applied: authenticated users can create their own document (userId matches their auth UID), read only their own document, update only their own document, and documents cannot be hard-deleted (soft delete only). The createdAt and userId fields must be immutable — the rules should reject any write that tries to change them after creation. Test these rules using the Firebase Rules Playground before deploying.

**Server-Side Enforcement** — Confirm that all three endpoint handlers (GET /api/benchmarks/performance, GET /api/benchmarks/settings, PATCH /api/benchmarks/settings) use `admin.auth().verifyIdToken(token)` and extract the userId exclusively from the verified token — never from the request body or query params. The benchmarks Firestore queries must always include a `userId == uid` filter so a user can never read or write another user's benchmark configuration.

**Rate Limiting on PATCH** — PATCH /api/benchmarks/settings has a 20 updates/hour per user rate limit. Make sure the client handles 429 responses gracefully — show a 'Too many updates, please wait before saving again' message in the modal rather than a generic error.

**Audit Log** — PATCH /api/benchmarks/settings writes to the auditLogs Firestore collection. Confirm the audit log entry includes the userId, changed metrics, and old/new historicalWindowDays values. The auditLogs collection should have security rules that prevent any client from reading or writing audit logs directly — only the server-side Admin SDK should touch it.
