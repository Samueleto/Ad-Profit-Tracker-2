Now we need to connect the FilterToolbar's Zustand state to the actual data fetching across the FinancialMetricsDashboardPage. When filter state changes in the store, the relevant dashboard sections should automatically re-fetch with the new filter params.

Here's what needs to be wired up:

**FilterToolbar → GET /api/filters/options**: When the country combobox is opened or focused, fire a request to GET /api/filters/options?type=country with the current dateFrom and dateTo from the store. If the active date range changes while the dropdown is open, re-fetch the country list. Pass the Firebase ID token as a Bearer token in the Authorization header.

**Filter state → GET /api/filters/stats**: Any dashboard section that currently fetches aggregated stats should now pass the committed filter values (selectedNetworks, selectedCountries, selectedMetric, dataQuality) as query params to GET /api/filters/stats. Wire this so that whenever selectedNetworks, selectedCountries, selectedMetric, or dataQuality changes in the store, those sections automatically re-fetch. Use the groupBy param to match each section's existing grouping (daily for time-series charts, country for geo tables, network for network breakdowns).

**Free-text search → client-side filtering**: The searchQuery field in the store should be used to filter already-loaded table rows client-side — no additional API call needed. Any table component on the dashboard that shows country or network rows should subscribe to searchQuery from the store and apply a case-insensitive filter against the row's country name or network name before rendering.

**Preset controls → GET/POST/DELETE /api/filters/presets**: Wire up preset loading, saving, and deletion. On the FilterToolbar (or a small 'Saved Filters' dropdown near the toolbar), add a way to:
- Load presets: call GET /api/filters/presets on mount and show the list. When a preset is selected, call applyFilters() with the preset's values written into the store.
- Save current filters: show a small 'Save as preset' input that calls POST /api/filters/presets with the current committed filter state and a user-provided name.
- Delete a preset: call DELETE /api/filters/presets/:presetId and remove it from the local list.

**Date range changes**: When dateFrom or dateTo changes in the store (from the existing DateRangeToolbar), the country list in the combobox should be invalidated and re-fetched on next open, since available countries depend on the date range.

Make sure all fetch calls include the Firebase ID token from the current auth session as a Bearer token. Use the existing auth utility pattern from the project.
