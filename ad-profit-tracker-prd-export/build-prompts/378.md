We're building an API rate limiting and throttling system. For this step, I need you to make sure errors from all four rate limit endpoints are handled gracefully — both on the server side and in the client-side hooks.

On the server side, each endpoint should return consistent, structured error responses. Every error response should include a `code` string (like `'INVALID_NETWORK_ID'`, `'UNAUTHENTICATED'`, `'RATE_LIMIT_EXCEEDED'`, `'LIMITER_REGISTRY_UNAVAILABLE'`, `'FIRESTORE_READ_FAILURE'`), a human-readable `message`, and where relevant, extra context (like `retryAfterSeconds` on a 429). Make sure 500 errors from Firestore failures in `GET /api/rate-limits/status` and `GET /api/rate-limits/violations` are caught and returned as structured 500s — don't let unhandled promise rejections crash the handler. Similarly, if the Bottleneck limiter registry is unavailable in `GET /api/rate-limits/status` or `POST /api/rate-limits/reset`, return a 500 with code `'LIMITER_REGISTRY_UNAVAILABLE'` rather than throwing.

For `POST /api/rate-limits/reset`, if the audit log write to Firestore fails, don't fail the whole request — the reset itself may have succeeded. Log the audit log failure server-side and return a successful response with a warning flag like `{ ..., auditLogWarning: 'Audit log write failed' }` so callers know the reset happened but wasn't logged.

On the client side, update the four hooks from the state management step to handle each error shape:
- 401 errors should trigger a sign-out or token refresh, not just set an error state. If the Firebase token has expired, try refreshing it once before giving up.
- 429 errors from `useRateLimitReset` should surface the `retryAfterSeconds` value so the UI can show a countdown or disabled state with a timer.
- 500 errors should set the `error` state with the `code` and `message` so the UI can show a specific message ("Rate limit service temporarily unavailable" is better than "Something went wrong").
- Network failures (fetch throws) should be caught and treated as a retriable error — don't crash the hook.

For the polling in `useRateLimitStatus`, if a poll fails, don't stop polling — just set the error state and keep trying. If three consecutive polls fail, increase the poll interval (exponential backoff) to avoid hammering a struggling server.
