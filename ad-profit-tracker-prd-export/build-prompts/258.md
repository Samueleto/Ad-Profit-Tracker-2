We're building the state management layer for the Excel export feature. The ExportModal needs to read from the existing Zustand store (for the dashboard date range) and we need to manage the modal's own open/closed state and export status.

First, check the existing Zustand store for the dashboard. It should already have dateFrom and dateTo fields — the ExportModal reads these directly. Do not add a separate date range to the export store. If for some reason the store doesn't have these, add them now.

Next, add export-related state to the Zustand store (or a dedicated export slice if the project uses slice-based organization). The export state should track:
- isExportModalOpen: boolean — whether the modal is visible
- exportStatus: 'idle' | 'loading' | 'exporting' | 'success' | 'error' — the current state of the export flow
- exportError: string | null — an error message to display if something goes wrong
- previewData: the response shape from GET /api/export/preview (sheets row counts, hasData, totalRows) or null if not yet loaded
- previewLoading: boolean — whether the preview request is in flight

Add actions:
- openExportModal() — sets isExportModalOpen to true and resets exportStatus to 'idle'
- closeExportModal() — sets isExportModalOpen to false and clears any error state
- setExportStatus(status) — updates exportStatus
- setExportError(message) — sets exportError
- setPreviewData(data) — stores the preview response
- setPreviewLoading(loading) — toggles the preview loading flag

The ExportModal component should read isExportModalOpen, exportStatus, exportError, previewData, previewLoading, dateFrom, and dateTo from this store. It should call openExportModal/closeExportModal for the open/close triggers. The export flow (calling the API, handling the response) should live in the component or a custom hook — not in the Zustand store itself, since it involves async side effects and file downloads.

If the project uses React Query or SWR for data fetching, use that for the GET /api/export/preview call (with the dateFrom+dateTo as the query key) rather than storing previewData in Zustand. In that case, only the modal open/close state and exportStatus need to live in Zustand.
