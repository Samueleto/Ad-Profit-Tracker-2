We're building an ad network sync dashboard. For this step, I need you to make sure all seven error handling endpoints are properly secured.

**Authentication on every endpoint** — All seven endpoints (`GET /api/errors/retry-state`, `GET /api/errors/circuit-breaker/status`, `POST /api/errors/circuit-breaker/reset`, `GET /api/errors/log`, `GET /api/errors/summary`, `GET /api/errors/retry-config`, `PATCH /api/errors/retry-config`) must verify the Firebase ID token via `admin.auth().verifyIdToken(token)` before doing anything else. The `uid` must always come from the verified token — never from the request body or query params. Return 401 immediately if the token is missing, malformed, or expired.

**Data scoping** — Every Firestore query must filter by `userId == uid` from the verified token. This applies to both `networkConfigs` and `auditLogs` queries. A user must never be able to read or modify another user's circuit breaker state, retry config, or error logs. Double-check that the `GET /api/errors/log` query filters `auditLogs` by both `userId==uid` AND `resourceType=='sync'` — without the resourceType filter, users could potentially see audit log entries from other features.

**Input validation** — For every endpoint that accepts a `networkId`, validate it against the exact list `['exoclick', 'rollerads', 'zeydoo', 'propush']` — return 400 for anything else. For `GET /api/errors/log`, validate that `limit` is an integer between 1 and 50, and that `startDate`/`endDate` are valid `YYYY-MM-DD` strings (reject things like `../../etc`). For `PATCH /api/errors/retry-config`, validate `retryAttempts` is an integer 1–5 and `timeoutSeconds` is an integer 5–60. For `POST /api/errors/circuit-breaker/reset`, confirm the document's `userId` field matches `uid` before writing (defense in depth beyond the query filter).

**Rate limiting** — `POST /api/errors/circuit-breaker/reset` is limited to 10 resets/hour per user. `PATCH /api/errors/retry-config` is limited to 20 updates/hour per user. Implement these using a Firestore counter or an in-memory rate limiter (like `rate-limiter-flexible` if available in the project). Return 429 with a `Retry-After` header indicating when the limit resets.

**Response sanitization** — Make sure none of the error endpoints leak sensitive data. The `GET /api/errors/log` response must not include any API keys, encrypted credential blobs, or raw Firestore document IDs beyond what's needed for pagination cursors. The `lastSyncError` field in `GET /api/errors/retry-state` may contain raw error messages from external APIs — strip or truncate anything that looks like a credential (anything matching patterns like `key=`, `token=`, `secret=`, `password=`).

**Audit logging** — `POST /api/errors/circuit-breaker/reset` and `PATCH /api/errors/retry-config` both write to the `auditLogs` collection via the Admin SDK. Make sure these writes happen after the main operation succeeds — a failed audit log write should not roll back the main operation, but should be logged server-side.
