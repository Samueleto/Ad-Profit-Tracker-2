The dashboard's API endpoint runs two Firestore queries that need to be fast and correctly scoped. We're not creating new collections here — adStats and networkConfigs should already exist — but we need to make sure the right composite indexes are in place so the queries don't fail or time out.

For the adStats collection, the endpoint queries: `userId == uid AND date >= dateFrom AND date <= dateTo`. This is a range query on date filtered by userId, which requires a composite index. Make sure there's a composite index on (userId ASC, date ASC) for the adStats collection. If it doesn't exist, add it to your firestore.indexes.json and deploy it. Without this index, Firestore will reject the query with an error and provide a link to create it — but it's better to have it ready before the endpoint goes live.

For the networkConfigs collection, the query is just `userId == uid` — a simple equality filter that doesn't need a composite index beyond the default single-field indexes Firestore provides automatically.

Also check your Firestore security rules. The dashboard API endpoint uses the Firebase Admin SDK, which bypasses security rules entirely — so the rules don't affect the API. But if any client-side code ever reads these collections directly, the rules should ensure users can only read their own documents (userId == request.auth.uid).

If you have a firestore.indexes.json file, add the composite index entry there and deploy with `firebase deploy --only firestore:indexes`. If you don't have one yet, create it with just the adStats index.

Finally, double-check that the adStats documents actually have a date field in YYYY-MM-DD string format (not a Firestore Timestamp) — the endpoint filters by string comparison, so the format needs to be consistent.
