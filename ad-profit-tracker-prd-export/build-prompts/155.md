We need two API route handlers in a single file at `src/app/api/settings/preferences/route.ts` — one for reading preferences and one for updating them.

**GET /api/settings/preferences** — This reads the current user's preferences from Firestore.

Start by extracting the Bearer token from the `Authorization` header. If it's missing, return 401 immediately. Verify it using `adminAuth.verifyIdToken(token)` from the Firebase Admin SDK — if verification fails (expired, malformed, revoked), return 401. Once you have the verified UID, fetch the `users/{uid}` document from Firestore using `adminDb`. Extract the `preferences` sub-object from the document. If the document doesn't exist (shouldn't happen in normal flow, but handle it), return 404 and optionally trigger a user sync. If `preferences` is undefined or null (first-time user who hasn't set preferences yet), return the `DEFAULT_PREFERENCES` constant with a 200 status — don't return 404 for this case. Otherwise return the preferences object. The response shape should match the `UserPreferences` interface: `{ timezone, currency, defaultDateRange, notifications: { dailySummaryEmail, weeklyReportEmail }, updatedAt }`.

**PATCH /api/settings/preferences** — This updates one or more preference fields.

Same auth flow as GET — extract and verify the Bearer token, return 401 if invalid. Parse the request body as JSON. Validate each provided field using the helpers from the schema_models step: timezone must be a valid IANA string, currency must be a valid ISO 4217 code, defaultDateRange must be one of the four enum values, notification fields must be booleans. If any field fails validation, return 400 with a clear error message that names the specific field and what's wrong (e.g., `{ error: 'Invalid timezone: "Fake/Zone" is not a recognized IANA timezone' }`). Don't update anything if any field is invalid.

If validation passes, use `buildPreferencesUpdateObject()` to build the dot-notation Firestore update object (this automatically adds `preferences.updatedAt` as a server timestamp). Call `users/{uid}.update(updateObject)` — using dot-notation paths means we only touch the fields we're updating, leaving everything else in the document untouched. After the write, fetch the updated document and return the full preferences object with 200 status.

Both handlers should catch unexpected errors and return 500 with a generic message — don't leak Firestore error details to the client.
