We need to define the types and Zustand store that the DateRangeToolbar and the rest of the dashboard will share. This is the contract between the toolbar and every other component that depends on the selected date range.

First, in the types file, define these TypeScript types:

- A PresetOption type that covers the six preset values: 'last7', 'last14', 'last30', 'last90', 'thisMonth', and 'custom'.
- A DataAvailabilityStatus type: 'loading', 'complete', 'partial', 'none', 'error'.
- A DateRange interface with: preset (PresetOption), fromDate (string in YYYY-MM-DD format), toDate (string in YYYY-MM-DD format), and isCustom (boolean).
- A DateRangeState interface that extends DateRange and adds: dataAvailability (DataAvailabilityStatus), lastSynced (string or null — ISO timestamp from the dashboard's shared state), and any pending custom input values the user is typing before they hit Apply.

Then, in the Zustand store file, create a useDateRangeStore hook using Zustand's create(). The store should hold the full DateRangeState and expose these actions:
- setPreset(preset: PresetOption) — calculates the fromDate and toDate for the given preset using date-fns (e.g., last 7 days = today minus 6 days through today), sets isCustom to false, and immediately updates the store.
- setCustomRange(from: string, to: string) — sets the pending custom dates but does NOT update the active range yet (waits for Apply).
- applyCustomRange() — validates that the range is 90 days or fewer using date-fns differenceInDays, then writes the custom dates to the active range if valid.
- setDataAvailability(status: DataAvailabilityStatus) — called after GET /api/stats/dates resolves.
- setLastSynced(timestamp: string) — updates the last synced badge.

Initialize the store with a sensible default — Last 30 Days is a good starting point. Use date-fns to calculate the initial fromDate and toDate at store creation time so the dashboard has a valid range immediately on load.
