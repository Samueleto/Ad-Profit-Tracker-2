We need four API route handlers under `app/api/auth/`. All of them follow the same auth pattern: extract the Bearer token from the `Authorization` header, verify it with `adminAuth.verifyIdToken(token)`, and reject with 401 if it's missing or invalid. Let's build each one:

**POST `/api/auth/sync-user` (`app/api/auth/sync-user/route.ts`):**
This is called immediately after a successful client-side Firebase Auth sign-in. It doesn't need a request body — all the info comes from the verified ID token. After verifying the token, check if `users/{uid}` exists in Firestore. If it doesn't exist, create the document with all fields from the `users` schema: `uid`, `email`, `displayName`, `photoURL`, `authProvider` (derive this from `token.firebase.sign_in_provider` — map `'google.com'` to `'google'` and `'password'` to `'email'`), `role: 'user'`, `createdAt: FieldValue.serverTimestamp()`, and `lastLoginAt: FieldValue.serverTimestamp()`. If the document already exists, only update `lastLoginAt`. Return the full user document data as a `UserProfileResponse` (timestamps serialized to ISO strings).

**GET `/api/auth/get-user` (`app/api/auth/get-user/route.ts`):**
Verify the token, then fetch `users/{uid}` from Firestore. If the document doesn't exist, return 404 — the client should call `sync-user` when it gets a 404 here. Return the full profile including `preferences` as a `UserProfileResponse`.

**PATCH `/api/auth/update-profile` (`app/api/auth/update-profile/route.ts`):**
Verify the token, then parse the request body. Only allow `displayName` (string, max 100 characters) and `preferences` (object with only `defaultDateRange` and `timezone` keys). Reject any other fields with a 400. Do a merge-update on `users/{uid}` with the validated fields plus `updatedAt: FieldValue.serverTimestamp()`. Return the updated `uid`, `displayName`, `preferences`, and `lastLoginAt`.

**DELETE `/api/auth/delete-account` (`app/api/auth/delete-account/route.ts`):**
Verify the token. Delete the `users/{uid}` Firestore document. Then delete the Firebase Auth user via `adminAuth.deleteUser(uid)`. Do both operations and handle partial failure — if Firestore deletion succeeds but Auth deletion fails (or vice versa), log the inconsistency clearly so it can be cleaned up manually. Return `{ deleted: true }` on full success.

Create a shared helper function (e.g., `lib/firebase-admin/verify-token.ts`) that extracts and verifies the Bearer token, so you're not duplicating that logic across all four routes.
