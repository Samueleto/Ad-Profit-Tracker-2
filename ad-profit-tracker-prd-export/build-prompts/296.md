The tab should only be visible to authenticated users — if someone lands on /settings/team without being logged in, the existing middleware should handle the redirect.

**Wire up the action menu callbacks**

In TeamManagementPage, connect the three-dot action menu items to the right API calls:
- 'Change Role' should open a small inline role picker (or a simple confirmation dialog) and call PATCH /api/team/members/:memberId/role with the selected role. On success, call the `refresh()` function from useTeamMembers so the roster updates.
- 'Remove Member' should open a confirmation dialog ('Are you sure you want to remove [name] from the workspace?') and call DELETE /api/team/members/:memberId. On success, refresh the roster.
- 'Transfer Ownership' (Owner only, on Admin rows) should open a confirmation dialog with a warning that this action is irreversible, then call the appropriate endpoint. On success, refresh both the roster and workspace metadata (since the current user's role changes).
- 'Leave Workspace' (non-owners) should open a confirmation dialog and call DELETE /api/team/members/:memberId using the current user's own UID. On success, redirect to /dashboard.

**Wire up the InviteMemberModal**

The 'Invite Member' button on TeamManagementPage should open InviteMemberModal. Pass the workspace name (from useWorkspace) into the modal so it can display it at the top. After a successful invite, the modal should call the `refresh()` function from useTeamInvitations so the pending invitations section updates immediately.

**Wire up the Revoke button**

The 'Revoke' button in the pending invitations section should call DELETE /api/team/invitations/:invitationId. On success, call `refresh()` from useTeamInvitations.

**Wire up AcceptInvitationPage post-acceptance redirect**

After a successful POST /api/team/invitations/accept, redirect the user to /dashboard. Use Next.js router's `push` or `replace` (use `replace` so they can't go back to the accept page). Pass a query param or use a toast library to show 'Welcome to [Workspace]!' on the dashboard after redirect.

After a successful POST /api/team/invitations/decline, redirect to a simple 'You've declined the invitation' confirmation screen or back to / — whichever makes more sense for the app's existing flow.

**Wire up the unauthenticated accept flow**

On AcceptInvitationPage, if the user is not signed in (check Firebase Auth's current user), the Accept button should redirect to / with a `returnUrl` query param set to the current page URL including the token. After the user signs in, the app should read the `returnUrl` and redirect back to /invite/accept?token=... so they can complete the acceptance. Make sure the existing login flow at / supports this `returnUrl` param.
