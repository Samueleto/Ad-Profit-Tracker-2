The team management feature touches sensitive workspace data and role assignments, so we need to make sure the security is solid on both the client and server sides.

**Client-side: never trust the role from local state for access decisions**

The role-based UI rendering in TeamManagementPage (showing/hiding action menus, the Invite button, the Revoke button) is purely cosmetic — it's there for usability, not security. Make sure the app never uses the locally cached `currentUserRole` to decide whether to *send* a request. Every mutation (role change, removal, invite, revoke) should always be sent to the server, which will enforce the role check server-side. If the server returns a 403, handle it gracefully (see error_handling step) rather than trying to predict it client-side.

**Client-side: Firebase ID token freshness**

The `getAuthHeaders` utility calls `getIdToken()` — make sure it calls `getIdToken(false)` by default (uses the cached token if not expired) but the error handling layer calls `getIdToken(true)` (force refresh) when a 401 is received. This avoids unnecessary token refreshes on every request while still recovering from expired tokens automatically. Never store the raw ID token in localStorage, sessionStorage, or React state — only use it in memory during the fetch call.

**Client-side: AcceptInvitationPage token handling**

The invitation token in the URL query param should never be logged, stored in analytics events, or passed to any third-party service. When the accept or decline action completes (success or failure), consider replacing the URL in the browser history (using `router.replace`) to remove the token from the address bar so it's not visible if the user shares their screen or the URL gets captured in browser history sync.

**Client-side: role assignment in InviteMemberModal**

The role selector should only offer 'admin' and 'member' — never 'owner'. Make sure the value sent in the POST /api/team/invitations body is taken from the controlled form state and validated client-side before sending. If somehow 'owner' ends up in the payload (e.g., via browser devtools), the server will reject it with a 400, but we should not send it in the first place.

**Server-side: verify all role checks happen on the server**

For each of the four endpoints — GET /api/team/members, GET /api/team/workspace, PATCH /api/team/workspace, and PATCH /api/team/members/:memberId/role — confirm that:
- The Firebase ID token is verified via `admin.auth().verifyIdToken(token)` before any Firestore read or write
- The `uid` used for all permission checks comes exclusively from the verified token, never from the request body or query params
- PATCH /api/team/workspace checks `workspaceRole === 'owner'` before allowing a rename
- PATCH /api/team/members/:memberId/role enforces the full role hierarchy: owner can change any non-owner, admin can only change members, nobody can change the owner's role
- DELETE /api/team/members/:memberId verifies the actor and target are in the same workspace before proceeding
- POST /api/team/invitations verifies the inviting user is admin or owner
- GET /api/team/invitations/validate, POST /api/team/invitations/accept, and POST /api/team/invitations/decline verify the token hash server-side and check expiry

**Firestore security rules for workspaceInvitations**

The workspaceInvitations collection rules should ensure: authenticated users can create invitations (the server endpoint handles the deeper permission check), authenticated users can read invitations for their own workspace, only the inviting user or workspace owner can update/revoke an invitation, and `createdAt` is immutable. Make sure these rules are deployed and tested.
