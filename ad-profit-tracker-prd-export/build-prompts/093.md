We're extending the existing `networkConfigs` Firestore collection with new fields for retry state and circuit breaker tracking. We're not creating a new collection — we're adding fields to documents that already exist.

The six new fields to add to each `networkConfigs` document are:
- `retryCount` (number) — how many consecutive failures have happened; resets to 0 on success
- `nextRetryAt` (Firestore Timestamp or null) — when the next automatic retry is scheduled, computed using exponential backoff
- `circuitBreakerOpen` (boolean, default `false`) — when true, all sync attempts for this network are skipped
- `circuitBreakerOpenedAt` (Firestore Timestamp or null) — when the circuit breaker was last tripped; used to compute the 1-hour auto-reset
- `lastErrorCode` (string or null) — the semantic error code from the last failure: one of `NETWORK_TIMEOUT`, `AUTH_FAILURE`, `RATE_LIMITED`, `PARSE_ERROR`, `FIRESTORE_WRITE_FAILURE`, `DECRYPTION_FAILURE`
- `totalFailureCount` (number) — lifetime failure count, never reset, used for monitoring

Write a one-time migration script (or Firestore Admin script) that finds all existing `networkConfigs` documents and sets these fields to their defaults if they're missing: `retryCount: 0`, `nextRetryAt: null`, `circuitBreakerOpen: false`, `circuitBreakerOpenedAt: null`, `lastErrorCode: null`, `totalFailureCount: 0`. This ensures existing documents are compatible with the new API routes.

Update the Firestore security rules to cover these new fields. The rules for `networkConfigs` should be:
- Create: user must be authenticated (`request.auth != null`)
- Read: user must be authenticated
- Update: only the document owner can update (`resource.data.userId == request.auth.uid`)
- Delete: only the document owner can delete
- The `createdAt` field should be immutable (reject updates that change it)

Deploy the updated security rules using `firebase deploy --only firestore:rules`.
