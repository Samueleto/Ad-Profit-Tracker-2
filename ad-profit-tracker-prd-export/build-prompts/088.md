We're using four Firestore collections for the Zeydoo integration: `adStats`, `apiKeys`, `networkConfigs`, and `auditLogs`. We need the right indexes so our queries run fast, and the right security rules so users can only access their own data.

For **Firestore indexes**, create composite indexes for the queries our endpoints will run:
- `adStats`: index on `(userId, networkId, date)` — used by the stats, by-country, latest, and raw-response endpoints to filter by user + network + date range
- `adStats`: index on `(userId, networkId, date DESC)` — used by the latest endpoint to find the most recent date
- `networkConfigs`: index on `(networkId, isActive, syncSchedule)` — used by the scheduled-sync endpoint's collection-group query to find all active Zeydoo configs

Add these to `firestore.indexes.json` and deploy them.

For **Firestore security rules**, the key principle is that all sensitive reads and writes for this feature happen via the Firebase Admin SDK on the server (which bypasses client rules entirely). However, set up rules as a safety net:
- `adStats`: authenticated users can read their own documents (`userId == request.auth.uid`), no client writes
- `apiKeys`: no client reads or writes — server-only via Admin SDK
- `networkConfigs`: authenticated users can read their own documents, no client writes
- `auditLogs`: no client reads or writes — server-only

This defense-in-depth approach means even if a bug accidentally exposes a client-side path, the rules block it.
