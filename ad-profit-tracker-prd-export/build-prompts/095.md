This feature reads from three Firestore collections — `networkConfigs`, `adStats`, and `auditLogs` — and writes to `networkConfigs`. We need to make sure the right indexes exist so the queries in our three API routes run efficiently and don't time out.

Here's what each API route queries:

**GET /api/sync/live-state** runs these queries:
- `networkConfigs` filtered by `userId == uid` (and optionally `networkId`)
- `adStats` filtered by `userId == uid AND networkId == <network>`, ordered by `syncedAt DESC`, limit 1 — do this for each of the 4 networks
- `auditLogs` filtered by `userId == uid AND resourceType == 'sync' AND action == 'manual_sync_triggered' AND resourceId == <networkId>`, ordered by `createdAt DESC`, limit 1 — once per network

**GET /api/sync/activity-feed** queries:
- `auditLogs` filtered by `userId == uid AND resourceType == 'sync' AND action == 'manual_sync_triggered'`, optionally also filtered by `resourceId == networkId`, ordered by `createdAt DESC`, with cursor-based pagination

**PATCH /api/sync/progress** writes to:
- `networkConfigs` document for a specific `userId + networkId` — a targeted document update, no index needed

Create a `firestore.indexes.json` file (or update the existing one) with composite indexes for:
1. `adStats`: `userId ASC, networkId ASC, syncedAt DESC`
2. `auditLogs`: `userId ASC, resourceType ASC, action ASC, createdAt DESC`
3. `auditLogs`: `userId ASC, resourceType ASC, action ASC, resourceId ASC, createdAt DESC` (for the per-network filter)

Also update `firestore.rules` to ensure: authenticated users can read their own documents in all three collections (scoped by `userId == request.auth.uid`), and can only write to `networkConfigs` documents that belong to them. The `adStats` and `auditLogs` collections should be read-only from the client — all writes go through the server-side admin SDK.
