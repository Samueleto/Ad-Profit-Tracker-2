We need the Firestore `auditLogs` collection properly indexed and secured before we build the endpoints on top of it.

**Indexes** — The audit log queries need composite indexes to run efficiently. Create these in your `firestore.indexes.json`:

1. `(userId ASC, createdAt DESC)` — This is the primary query pattern. Every read from `GET /api/audit/logs` and `GET /api/audit/logs/export` filters by `userId` and sorts by `createdAt DESC`. This index is required.
2. `(userId ASC, action ASC, createdAt DESC)` — For when users filter by action type.
3. `(userId ASC, status ASC, createdAt DESC)` — For filtering by success/failure status.
4. `(userId ASC, resourceType ASC, createdAt DESC)` — For filtering by resource type.

Deploy these with `firebase deploy --only firestore:indexes`.

**Security Rules** — Update `firestore.rules` for the `auditLogs` collection:
- **Create**: Allow only if `request.auth != null` AND `request.resource.data.userId == request.auth.uid`. In practice, writes go through the Admin SDK (which bypasses rules), but this is a safety net.
- **Read**: Allow only if `request.auth != null` AND `resource.data.userId == request.auth.uid`. Users can only read their own logs.
- **Update**: Allow only if `request.auth != null` AND `resource.data.userId == request.auth.uid`. Also enforce that `createdAt` and `userId` fields cannot be changed (immutable fields).
- **Delete**: Deny direct client deletes — deletion only happens through the Admin SDK in `DELETE /api/audit/logs/clear`. This prevents clients from bypassing the rate limit and audit trail on the clear action.

Deploy rules with `firebase deploy --only firestore:rules`.

Also make sure the `auditLogs` collection itself doesn't need to be manually created — Firestore creates collections automatically on first write, so no setup needed there.
