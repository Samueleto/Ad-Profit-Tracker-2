We need TypeScript types and shared utility functions that all seven API routes will use. Getting these right before building the routes will save a lot of repetition.

**Types** — Create a types file at `src/lib/errors/types.ts` with:
- `NetworkId` type: a union of the four supported network strings (`'exoclick' | 'rollerads' | 'zeydoo' | 'propush'`)
- `ErrorCode` type: a union of the six semantic error codes (`'NETWORK_TIMEOUT' | 'AUTH_FAILURE' | 'RATE_LIMITED' | 'PARSE_ERROR' | 'FIRESTORE_WRITE_FAILURE' | 'DECRYPTION_FAILURE'`)
- `ErrorCategory` type: `'transient' | 'auth' | 'data' | 'infrastructure'`
- `NetworkRetryState` interface matching the full shape returned by `GET /api/errors/retry-state` (all the fields listed in that endpoint's success response)
- `CircuitBreakerStatus` interface matching the shape returned by `GET /api/errors/circuit-breaker/status`
- `ErrorLogEntry` interface matching the shape returned by `GET /api/errors/log`
- `RetryConfig` interface matching the shape returned by `GET /api/errors/retry-config`
- `BackoffScheduleEntry` interface: `{ attempt: number, delayMs: number, jitterRangeMs: [number, number] }`

**Backoff utility** (`src/lib/errors/backoff.ts`) — Export a `computeBackoffSchedule(retryAttempts: number)` function that returns an array of `BackoffScheduleEntry`. Use `baseDelayMs = 1000`, `maxDelayMs = 30000`, formula: `min(baseDelayMs * 2^(attempt-1), maxDelayMs)`, jitter range ±20% of computed delay. This is used by `GET /api/errors/retry-config`.

**Error classification utility** (`src/lib/errors/classify.ts`) — Export two functions:
- `isRetryable(errorCode: ErrorCode | null): boolean` — returns true for `NETWORK_TIMEOUT`, `RATE_LIMITED`, `PARSE_ERROR`; false for others
- `getErrorCategory(errorCode: ErrorCode | null): ErrorCategory` — maps `NETWORK_TIMEOUT`/`RATE_LIMITED` → `'transient'`, `AUTH_FAILURE` → `'auth'`, `PARSE_ERROR` → `'data'`, `FIRESTORE_WRITE_FAILURE`/`DECRYPTION_FAILURE` → `'infrastructure'`

**Rate limit utility** (`src/lib/errors/rateLimit.ts`) — Export a simple in-memory rate limiter (or use a Firestore-backed one if the project already has rate limiting infrastructure). It should expose `checkRateLimit(uid: string, action: string, maxPerHour: number): Promise<boolean>` — returns `true` if the request is allowed, `false` if the limit is exceeded. The circuit breaker reset endpoint uses 10/hour and the retry config update uses 20/hour.
