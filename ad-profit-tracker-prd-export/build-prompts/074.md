We need two Firebase SDK setups before building anything else — one for the server side (Admin SDK for reading and writing Firestore from API routes) and one for the client side (Firebase Auth for getting ID tokens to attach to API requests).

On the server side, set up the Firebase Admin SDK in a shared module (something like src/lib/firebase-admin.ts). Initialize it once using a singleton pattern so it doesn't re-initialize on every hot reload in development. Pull the service account credentials from environment variables — never hardcode them. You'll need FIREBASE_PROJECT_ID, FIREBASE_CLIENT_EMAIL, and FIREBASE_PRIVATE_KEY at minimum. Export the initialized admin.auth() and admin.firestore() instances so all seven notification API routes can import them cleanly.

Also set up the INTERNAL_SYNC_SECRET environment variable. This is used by POST /api/notifications/create to allow server-side Cloud Functions (sync, export, schedule jobs) to create notifications without a user's Firebase ID token. Add it to your .env.local and document it clearly — it should be a long random string.

On the client side, make sure firebase/auth is available and that there's a utility function (or hook) that returns the current user's ID token — something like getAuthToken() that calls currentUser.getIdToken(). This will be used by all frontend calls to the notification endpoints to attach the Bearer token in the Authorization header. If the project already has a Firebase client setup, just make sure getIdToken() is accessible from the notifications feature.

Double-check that all environment variables are added to .env.local (for development) and documented for production deployment.
