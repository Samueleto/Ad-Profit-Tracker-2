# Step 107 of 437: Configure Firestore rules for scheduled reports

The scheduled reports data lives as an embedded array inside the existing users Firestore documents — we're not creating a new collection. So the database setup here is about making sure the Firestore security rules properly protect the scheduledReports field, and that we have the right indexes for any queries we'll run.

Firestore security rules — Update your firestore.rules file to cover the users document mutations we'll be doing. The rules should enforce:
- Only the authenticated user can read their own users document (match /databases/{database}/documents/users/{userId} — allow read if request.auth.uid == userId)
- Only the authenticated user can update their own users document
- The createdAt field inside any scheduledReports array item must be immutable — if the document already exists and has scheduledReports, the createdAt values in existing items should not change
- Unauthenticated requests are denied entirely

Note: since all our schedule mutations go through the Firebase Admin SDK on the server side (which bypasses client-side rules), these rules are a defense-in-depth measure for any direct client SDK calls. The real enforcement happens server-side via verified ID tokens.

Firestore indexes — The GET /api/schedules endpoint reads the users document and filters/sorts the scheduledReports array in application code (not via Firestore queries), so no composite indexes are needed for the schedules feature itself. However, if the auditLogs collection doesn't already have an index on (userId + createdAt), add that now — we'll be writing audit log entries from all five schedule endpoints and may want to query them later.

Add the auditLogs index to firestore.indexes.json: a composite index on the auditLogs collection with fields userId (ascending) and createdAt (descending).

Deploy the updated rules and indexes using the Firebase CLI.
