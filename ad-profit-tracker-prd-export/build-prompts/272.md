We're building the state layer for the Financial Metrics Dashboard. The goal is to keep all data fetching and derived state out of the page component itself, so FinancialMetricsDashboardPage just reads state and calls actions.

Create a custom hook called useDashboardMetrics that manages everything the dashboard needs:

**Date range state** — Track the selected date range as { dateFrom: string, dateTo: string, preset: string }. Initialize to Last 7 days (compute the dates using date-fns on mount). Expose a setDateRange action that accepts either a preset label or a custom { dateFrom, dateTo } pair. When the preset is 'Custom', validate that the range doesn't exceed 90 days before updating state — if it does, set a validation error instead.

**Fetch logic** — Whenever dateFrom or dateTo changes, fetch GET /api/dashboard/metrics?dateFrom=...&dateTo=... with the user's Firebase ID token as a Bearer token. Get the token using getIdToken() from firebase/auth — call getIdToken(false) first (uses cached token), and only call getIdToken(true) (force refresh) if you get a 401 back. Store the full response payload: kpis, dailySeries, topCountries, perNetwork, cachedAt, dateFrom, dateTo.

**Loading and error state** — Track a status field that can be: 'idle', 'loading', 'success', 'empty', 'error'. Set it to 'loading' immediately when a fetch starts. On a successful response with data, set 'success'. If the response has no adStats data (empty arrays), set 'empty'. On error responses, store the HTTP status code separately so the page can render the right error UI for 401, 403, 404, and 500.

**Refresh action** — Expose a refresh() function that calls POST /api/cache/invalidate (with Bearer token) and then re-fetches GET /api/dashboard/metrics. This is what the 'Refresh Data' button calls.

**Freshness derived state** — Derive a freshness object from cachedAt: if cachedAt is null, it's 'live'. If cachedAt is a timestamp, compute how many minutes ago it was using date-fns differenceInMinutes and expose { type: 'cached', minutesAgo: number }.

**Abort on unmount / stale fetches** — Use an AbortController so that if the user changes the date range quickly, the previous in-flight request is cancelled and its result doesn't overwrite the newer one.

The hook should return: { status, errorStatus, kpis, dailySeries, topCountries, perNetwork, freshness, dateRange, setDateRange, refresh, dateRangeValidationError }.

Wire this hook into FinancialMetricsDashboardPage so the page reads all its data from the hook rather than fetching directly.
