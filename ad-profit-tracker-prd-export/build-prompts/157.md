We need four API endpoints for the audit logging system. Here's what each one does and how to build it:

**POST /api/audit/log** — This is the write endpoint. Other endpoints in the app call this after mutations to record what happened. It should: verify the Firebase ID token from the `Authorization: Bearer <token>` header using `admin.auth().verifyIdToken()`; validate that `action` is one of the `AUDIT_ACTIONS` values, `resourceType` is one of `AUDIT_RESOURCE_TYPES`, and `status` is 'success' or 'failure' — return 400 with a clear message if not; sanitize the `metadata` object by stripping any keys that contain the words 'key', 'token', 'secret', or 'password' (case-insensitive) to prevent accidental credential logging; build the Firestore document with `userId` from the verified token (never from the request body), plus `action`, `resourceType`, `resourceId`, `metadata` (sanitized), `status`, `errorMessage`, `ipAddress` (from `x-forwarded-for` header or `req.socket.remoteAddress`), `userAgent` (from `User-Agent` header), and `createdAt: admin.firestore.FieldValue.serverTimestamp()`; write it to the `auditLogs` collection; return `{ logged: true, logId }`. Rate limit at 60 requests/min per user.

Also update the `src/lib/auditLogger.ts` helper to be a simple wrapper that calls this endpoint internally — other parts of the app should use this helper rather than calling the endpoint directly.

**GET /api/audit/logs** — The paginated read endpoint for the ActivityLogTab. Verify the token. Validate query params: `action` values must be from `AUDIT_ACTIONS`; dates must parse as valid ISO 8601; `limit` must be 1–100 (default 25); `status` must be 'success' or 'failure' if provided. Build a Firestore query on `auditLogs` always filtering `userId == uid` from the token. Apply `action` filter using Firestore's `in` operator (max 10 values). Apply `createdAt` date range filters. Apply `status` filter. Order by `createdAt DESC`. If a `cursor` document ID is provided, fetch that document first and use `startAfter()` for pagination. Fetch `limit + 1` results to detect if there's a next page. Map documents to the `AuditLog` shape (serialize `createdAt` as ISO string). Return `{ logs, total, hasMore, nextCursor }`. Note: the `search` param is applied as client-side filtering on the returned page results (Firestore doesn't support full-text search) — filter the results array by checking if the stringified `metadata` contains the search term. Rate limit at 60/min.

**GET /api/audit/logs/export** — Same auth and validation as the logs endpoint, but no pagination. Query all matching documents up to a hard cap of 10,000. Map each document to a flat CSV row: `id, action, resourceType, resourceId, status, errorMessage, metadata (JSON stringified), ipAddress, userAgent, createdAt (ISO string)`. Build a CSV string with a header row. Return with `Content-Type: text/csv` and `Content-Disposition: attachment; filename='activity-log-{YYYY-MM-DD}.csv'`. Rate limit at 5 requests/hour per user (exports are expensive).

**DELETE /api/audit/logs/clear** — Verify the token. If `olderThan` query param is provided, validate it's a valid ISO date. Query `auditLogs` where `userId == uid`, optionally filtered by `createdAt < olderThan`. Delete in batches of 500 (Firestore batch limit) using the Admin SDK. After all deletes, write one final audit log entry directly via the Admin SDK (not via the POST endpoint — that would cause recursion) recording `action: 'audit_log_cleared'`, `resourceType: 'audit_log'`, `metadata: { deletedCount, olderThan }`, `status: 'success'`. Return `{ deleted: true, deletedCount }`. Rate limit at 3 requests/day per user.
