Let's make sure the notification system is properly secured at every layer.

On all six user-facing endpoints — GET /api/notifications, PATCH /api/notifications/dismiss/:id, PATCH /api/notifications/mark-all-read, DELETE /api/notifications/clear, GET /api/notifications/preferences, and PATCH /api/notifications/preferences — verify the Firebase ID token on every single request using admin.auth().verifyIdToken(token) from firebase-admin. Never trust a userId from the request body or query params for these endpoints. Always derive uid from the verified token result. Return 401 immediately if the token is missing, malformed, or expired.

For POST /api/notifications/create, enforce the dual-auth pattern carefully. If an Authorization header is present, verify it as a Firebase ID token and use the uid from the token — ignore any userId in the request body. If an x-internal-secret header is present instead, compare it against the INTERNAL_SYNC_SECRET environment variable using a constant-time comparison (to prevent timing attacks) — only then trust the userId from the request body. If neither is valid, return 401. Never allow both paths to be active simultaneously on the same request.

Enforce rate limits on the mutating endpoints: 60 dismissals/hour for PATCH /api/notifications/dismiss/:id, 30 mark-all-read actions/hour for PATCH /api/notifications/mark-all-read, 10 clear-all actions/hour for DELETE /api/notifications/clear, and 20 preference updates/hour for PATCH /api/notifications/preferences. Use a server-side rate limiting approach keyed by uid (not IP, since users may share IPs). Return 429 with a Retry-After header when exceeded.

On the Firestore security rules side, make sure the users document rules enforce that a user can only read and write their own document: request.auth.uid == userId. The notifications array is embedded in the users document, so this rule covers all notification data. The createdAt field on individual notification objects should be treated as immutable — don't allow client-side writes to set or change it (all writes go through the Admin SDK server-side anyway, but the rules should reflect this).

For DELETE /api/notifications/clear and PATCH /api/notifications/preferences, write an audit log entry to the auditLogs Firestore collection via the Admin SDK. The audit log should capture: uid, action ('clear_all' or 'preferences_updated'), resourceType ('notification'), metadata (cleared count or changed preference types), timestamp, and status ('success' or 'failure'). This gives you a paper trail for destructive actions.

On the client side, make sure the NotificationCenterPanel always calls getIdToken() from firebase/auth before each API request — don't cache the token string in component state or the Zustand store, as tokens expire after an hour. If getIdToken() returns a stale token, Firebase will automatically refresh it, so calling it fresh each time is the right pattern.
