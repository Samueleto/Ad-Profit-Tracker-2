We're building a Next.js + Firebase app. For this step, I need you to make sure every error scenario in the preferences feature fails gracefully with a clear, helpful user experience.

Here are all the error states to handle, and exactly what should happen in each:

**401 — Session expired**: When either GET or PATCH /api/settings/preferences returns 401, first attempt to refresh the Firebase ID token silently using `getAuth().currentUser.getIdToken(true)` (the `true` forces a refresh). If the refresh succeeds, retry the original request once. If the refresh fails or the retry also returns 401, redirect the user to / (the home/login page) and show a toast saying 'Session expired. Please sign in again.'

**403 — Access denied**: Show an inline error banner inside the PreferencesCard (not a full-page error) with the message 'Access Denied' and a link to /dashboard. Don't redirect automatically.

**404 on GET — No preferences document**: This is a normal first-time user state, not an error. Silently fall back to defaults (UTC, USD, last_7_days, both notifications false) and show the 'Defaults applied' label. Allow the user to save normally — the PATCH endpoint handles upsert.

**400 — Validation error on PATCH**: Show an inline error message directly below the specific field that failed. For example, if the timezone is invalid, show a small red message under the timezone combobox. Don't show a generic banner — be specific. Re-enable the controls so the user can correct the value.

**429 — Rate limited**: Show a toast saying 'Too many changes — please wait a moment' and re-enable controls after 5 seconds.

**500 — Server error on GET**: Show an inline error banner inside the PreferencesCard (where the controls would be) with a 'Retry' button. Clicking Retry re-fetches GET /api/settings/preferences.

**500 — Server error on PATCH**: Show a toast saying 'Failed to save preferences. Please try again.' Re-enable controls immediately so the user can retry. Don't revert the local state — keep what the user entered so they don't lose their changes.

**Network failure (fetch throws)**: Treat this the same as a 500 — show the retry toast for PATCH, show the inline retry banner for GET.

For the Server Component's parallel fetch (Promise.all), if the preferences fetch fails, don't let it break the whole SettingsPage. Catch the error, pass null as the initial preferences, and let PreferencesCard handle it client-side with the 500 error state and Retry button.
