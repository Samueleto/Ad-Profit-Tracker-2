We're building the ScheduleReportModal — a centered modal dialog (max-w-md) that lets users configure automated email delivery for a saved report. This modal is triggered by a clock/calendar icon button on each saved report row in the CustomReportBuilderPage's saved reports sidebar panel. It's the only UI component for this feature, but it's complex enough to warrant its own file.

The modal receives a reportId and reportName as props. When it opens, it calls GET /api/schedules?reportId={reportId} to check if a schedule already exists. While that loads, show 5 shimmer placeholder rows where the form fields will be.

Once loaded, show the form. If a schedule exists, pre-populate all fields with the existing values. If not, use defaults: frequency='daily', deliveryEmail=the user's Firebase Auth email, dateRangePreset='last_7', format='excel', isActive=true.

The form layout (compact vertical stack):
1. Report name shown at the top as read-only text — just context for the user
2. Frequency selector — render as a segmented control with three options: Daily, Weekly, Monthly. When Weekly is selected, show a day-of-week picker (Mon–Sun, 0-indexed). When Monthly is selected, show a day-of-month picker (1–28). These sub-pickers should appear/disappear smoothly with the selection.
3. Below the frequency selector, show a 'Next delivery' preview in muted text — compute this client-side from the current form values using the same logic as computeNextRunAt (you can call the API or replicate the logic client-side for instant feedback). Format it as something like 'Tomorrow at 8:00 AM' or 'Next Monday at 8:00 AM'.
4. Delivery email field — text input, pre-filled with user's email, validates format on blur
5. A two-column row: left is the date range dropdown ('Yesterday', 'Last 7 days', 'Last 30 days', 'Last 90 days', 'This month'), right is the format toggle (Excel / PDF)
6. Active toggle — a labeled switch at the bottom of the form
7. A collapsed 'Advanced' section under a chevron — inside, show a timezone override dropdown (IANA timezone selector, defaults to the user's timezone from their profile)

At the bottom of the modal:
- Bottom-left: 'Delete Schedule' button — only visible when editing an existing schedule. It's styled destructively (red). Clicking it shows a confirmation dialog ('Are you sure you want to delete this schedule?'). On confirm, calls DELETE /api/schedules/:id.
- Bottom-right: 'Save Schedule' button — calls POST /api/schedules/create if no schedule exists, or PATCH /api/schedules/:id if editing. While saving, show a spinner on the button and disable all inputs.
- Somewhere accessible (maybe a link or secondary button): 'Send Test Now' — calls POST /api/schedules/:id/send-now. Only visible when an existing schedule is loaded. Show a brief loading state and then a success/error inline message.

Handle these states:
- loading: 5 shimmer rows
- saving: spinner on Save button, all inputs disabled
- empty (no existing schedule): form with defaults
- error_401: close the modal, refresh the Firebase ID token with getIdToken(true), redirect to / with a 'Session expired' toast if refresh fails
- error_403: show 'Access Denied' message inside the modal with a close button
- error_404: show 'Report not found' inside the modal with a close button
- error_500: show an inline error banner inside the modal with a Retry button
- success: show a 'Schedule saved' toast notification, then close the modal

Use the project's existing ThemeProvider tokens for all colors and spacing. The backdrop should be a semi-transparent overlay. The modal should be dismissible by clicking the backdrop or pressing Escape (unless a save is in progress).

Create this component at src/features/scheduled-reports/components/ScheduleReportModal.tsx. Also create a useSchedule hook at src/features/scheduled-reports/hooks/useSchedule.ts that encapsulates all the API calls (GET, POST, PATCH, DELETE, send-now) and exposes loading/saving/error states — the modal component should use this hook rather than calling fetch directly.
