Let's make sure the OnboardingWizardPage and the PATCH /api/auth/update-profile endpoint are properly locked down.

**Route protection:** The /onboarding route must require an authenticated Firebase user. The Next.js middleware should verify the Firebase ID token on every request to /onboarding — if there's no valid token, redirect to the login page. Don't rely solely on client-side auth checks.

**update-profile endpoint hardening:** The PATCH /api/auth/update-profile endpoint already exists, but make sure these specific onboarding fields are handled safely:
- onboardingCompletedAt must be set to serverTimestamp() server-side using the Firebase Admin SDK — the client sends the string 'serverTimestamp' as a signal, but the actual timestamp value is generated on the server. Never trust a timestamp value sent directly from the client.
- onboardingSkipped is a boolean — validate it strictly (only accept true/false, reject strings or other types).
- preferences.timezone must be validated against a list of valid IANA timezone strings on the server — don't just accept any string the client sends. Return a 400 if an unrecognized timezone is submitted.
- preferences.defaultDateRange must be one of the three allowed values: 'last7days', 'last30days', 'thisMonth'. Reject anything else with a 400.
- The endpoint must only update the document for the authenticated user's own UID — derive the UID from the verified Firebase ID token, never from the request body.

**Firestore security rules:** Confirm the users collection rules enforce that a user can only read and update their own document (user_id == AUTH_USER_ID). The onboardingCompletedAt field should not be directly writable from the client via Firestore SDK — all writes go through the API endpoint which uses the Admin SDK (bypasses client rules, but that's intentional since the server controls the timestamp).

**One-time enforcement:** The middleware check for onboardingCompletedAt is the primary guard against revisiting /onboarding. Make sure this check happens server-side in middleware, not just client-side, so it can't be bypassed by disabling JavaScript.
