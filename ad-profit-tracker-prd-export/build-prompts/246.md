We're building an in-app notification system for a financial metrics dashboard. For this step, I need you to build the NotificationCenterPanel — a right-side slide-over panel that lives on /dashboard and is triggered by a bell icon in the top navigation bar.

The bell icon should sit in the top nav and show a red badge with the unread count. The count comes from the Zustand dashboard store (more on that in state management), so it should update reactively without any extra fetching. When the count is zero, hide the badge entirely.

When the user clicks the bell, the panel slides in from the right — 360px wide on desktop, full-screen on mobile — with a backdrop overlay behind it. The panel header has a title ('Notifications'), a 'Mark All as Read' button (only active when there are unread items, calls PATCH /api/notifications/mark-all-read), and a 'Clear All' button that's styled as destructive. Clear All should require a single confirm click before calling DELETE /api/notifications/clear — use a simple inline confirmation state (button text changes to 'Confirm Clear?' on first click, executes on second click within 3 seconds, resets otherwise).

Below the header, show two tabs: 'All' and 'Unread'. Switching tabs filters the displayed list client-side — 'Unread' shows only items where isRead is false.

Each notification row is 56px tall and shows: a semantic icon on the left (sync type → arrows icon, error/failure types → red X, export types → download icon, schedule types → clock icon), a colored left border (severity: success=green, error=red, warning=amber, info=blue), a text block with a bold title and a body message truncated to one line, a relative timestamp (e.g., '3 minutes ago'), and an X dismiss button on the far right that calls PATCH /api/notifications/dismiss/:id with the notification's id. Clicking the row body expands it to show the full body text (progressive disclosure). Dismissed notifications should disappear from the list immediately (optimistic update).

Clicking a notification row should also navigate based on type: sync_failure and circuit_breaker_opened types navigate to the ManualRefreshPanel section on /dashboard (use a hash or scroll-to approach). schedule_failed and reconciliation_anomaly types navigate to /reports. export_complete and all other types do nothing on click.

The panel loads the first 20 notifications on open via GET /api/notifications. At the bottom of the list, show a 'Load more' button that calls GET /api/notifications with the cursor from the previous response. Appended results add to the existing list. Hide the button when hasMore is false.

Handle these states:
- Loading: show 5 shimmer placeholder rows matching the 56px notification row structure while the initial GET /api/notifications call resolves.
- Empty: show a friendly 'You're all caught up' message with a checkmark illustration. Bell icon shows no badge.
- error_500: show an inline error banner inside the panel with a Retry button that re-calls GET /api/notifications.
- error_403: show 'Access Denied' inside the panel with a close button.
- error_404: show 'No notifications found' with a close button.
- error_401: close the panel, attempt token refresh via getIdToken(true) from firebase/auth, and if that fails redirect to / with a 'Session expired' toast.

All API calls from this panel must include the Firebase ID token as a Bearer token in the Authorization header.
