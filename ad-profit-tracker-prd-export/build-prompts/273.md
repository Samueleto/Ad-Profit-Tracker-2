We're building the ROI Calculation Engine. For this step, I need you to set up the client-side state management that powers the ROIIndicatorSection and the threshold configuration UI.

We need two distinct pieces of state:

**1. ROI Metrics State**
Create a custom hook (something like `useROIMetrics`) that manages the data fetched from GET /api/roi/compute and GET /api/roi/breakdown. This hook should:
- Accept `dateFrom`, `dateTo`, `networkId` (optional), and `groupBy` as parameters
- Fetch from GET /api/roi/compute on mount and whenever those params change
- Expose: `roi`, `roiIndicator`, `colorCode`, `roiChange`, `roiChangeDirection`, `totalRevenue`, `totalCost`, `netProfit`, `series` (for daily/country groupings), `isLoading`, `error`, and a `refetch` function
- Debounce date range changes by ~300ms so rapid date picker interactions don't fire multiple requests
- Cache responses in memory for the session so switching back to a previously viewed date range doesn't re-fetch (the server already has node-cache, but a client-side cache avoids the round trip entirely for recently viewed ranges)

Create a second hook `useROIBreakdown` that fetches from GET /api/roi/breakdown. It takes `dateFrom`, `dateTo`, and `dimension` ('network' | 'country' | 'daily') and returns the breakdown array, summary totals, loading state, and error state. This is used by the per-network rows and the collapsible breakdown panel.

**2. ROI Thresholds State**
Create a `useROIThresholds` hook that:
- Fetches the user's current threshold config from GET /api/roi/thresholds on mount
- Exposes the thresholds object, a `usingDefaults` array, loading state, and error state
- Provides an `updateThresholds(patch)` function that calls PATCH /api/roi/thresholds with the changed fields
- Optimistically updates the local threshold state immediately on save, then rolls back if the PATCH fails
- Tracks a `isSaving` boolean so the UI can disable the save button during the request
- After a successful PATCH, shows a success toast notification
- Enforces the threshold ordering rule client-side before even sending the request: criticalThreshold <= warningThreshold <= positiveThreshold — surface a validation error immediately if violated

**Auth token attachment**
All three hooks need to attach the Firebase ID token as a Bearer token in the Authorization header. Create a shared utility (e.g., `getAuthHeaders()`) that calls `firebase.auth().currentUser?.getIdToken()` and returns the headers object. All API calls in these hooks should use this utility. If `getIdToken()` throws, the hook should set an auth error state and trigger the token refresh flow (call `getIdToken(true)`).

Keep all business logic — ROI formula display, color code mapping, threshold validation — inside these hooks or small pure utility functions, not inside the components themselves.
