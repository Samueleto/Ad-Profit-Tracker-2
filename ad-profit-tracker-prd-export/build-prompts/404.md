We need to make sure every error state in the RBAC feature is handled gracefully — both in the UI and at the API level.

**PermissionsTab error states:**
- While loading the matrix (GET /api/rbac/permissions), show shimmer rows. If the request fails with a 500, show an inline error banner with a 'Retry' button that re-calls the endpoint — don't show a blank table.
- If the response is 401, attempt a Firebase ID token refresh via `getIdToken(true)`. If the refresh succeeds, retry the request automatically. If it fails, redirect to `/` with a 'Session expired' toast notification.
- If the response is 403, show an inline 'Access Denied — insufficient permissions to view role matrix' message instead of the table.
- If the response is 404 (user has no workspace), show 'Permissions configuration not found' with a Retry button.
- The 'Effective Permissions' section (GET /api/rbac/my-permissions) should handle errors independently from the matrix — if it fails, show a small error state in just that card without breaking the rest of the tab.

**RolePermissionsEditModal error states:**
- While the modal is loading (GET /api/rbac/permissions resolving on open), show 8 shimmer permission rows.
- If the load fails with 500, show an inline error banner inside the modal with a Retry option.
- If the save (PATCH /api/rbac/permissions) fails with 403, show 'Only Owners and Admins can edit role permissions' inside the modal — don't close it.
- If the save fails with 400 (invalid permission key or owner role specified), show a specific validation message inside the modal.
- If the save fails with 429 (rate limit), show 'Too many changes — please wait before making more updates' inside the modal.
- If the save fails with 500, show an inline error banner inside the modal with a Retry option — keep the modal open and the user's draft changes intact so they don't lose their work.
- If a 401 occurs during save, close the modal, attempt `getIdToken(true)`, and if that fails redirect to `/` with 'Session expired' toast.
- On success, show a 'Permissions updated' toast, close the modal, and call `refresh()` on the parent matrix.

**API-level error handling:**
- All four endpoints (GET /api/rbac/permissions, PATCH /api/rbac/permissions, GET /api/rbac/my-permissions, POST /api/rbac/check) should return consistent error response shapes: `{ error: string, code: string }` — this makes it easy for the frontend to display the right message.
- Firestore read failures should be caught and returned as 500 with a generic message — never expose raw Firestore error details to the client.
- The batch effectivePermissions update in PATCH /api/rbac/permissions should not block the response — run it after returning 200. But if it fails, log the failure server-side so it can be investigated (stale caches are a security concern).

**Inline validation in the modal:**
- Before allowing Save, check that the user hasn't removed all permissions from a role — show an inline validation message if they try.
- If a change would remove a permission that active workspace members currently rely on, show a warning banner inside the modal (not a blocking error — let them proceed if they confirm).
