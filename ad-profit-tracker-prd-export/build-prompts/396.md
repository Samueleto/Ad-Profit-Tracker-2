# Step 396 of 437: Handle errors gracefully in benchmarking UI

The benchmarking feature has two components that make API calls, and each needs to handle failures without breaking the rest of the dashboard.

**PerformanceBenchmarkingTab Error States:**

*401 errors* — Attempt a forced Firebase ID token refresh via `getIdToken(true)`. If the refresh succeeds, retry the failed request once. If it fails, redirect to `/` and show a 'Session expired' toast. This applies to both GET /api/networks/comparison and GET /api/benchmarks/performance.

*403 errors* — Show an 'Access Denied' message inline within the tab panel (not a full-page redirect). Include a link to /dashboard so the user isn't stuck. Don't show this as a toast — it should be a persistent inline state since the user can't proceed.

*404 errors* — Show 'No data found for selected range' with a date range reset button that snaps the parent dashboard's date range back to the last 30 days (or whatever the default is). This is different from the empty state — 404 means the server confirmed no data exists, whereas the loading/empty state is pre-fetch.

*500 errors* — Show an inline error banner inside the tab panel (not a full-page error). Include a 'Retry' button that re-fires both GET /api/networks/comparison and GET /api/benchmarks/performance in parallel. The banner should be dismissible. Don't clear any previously loaded data if only one of the two parallel requests fails — show the error banner alongside whatever data did load.

*Partial failures* — Since the tab fires two requests in parallel, handle the case where one succeeds and one fails. If GET /api/networks/comparison fails but GET /api/benchmarks/performance succeeds (or vice versa), show the data you have and display a targeted error message for the failed piece rather than treating the whole tab as broken.

*Stale data badges* — These are not errors per se, but any network row where networkStatus.lastSyncStatus is 'failed' or circuitBreakerOpen is true should show an amber warning badge with a warning icon. These are non-blocking — the row still renders with whatever data is available.

**BenchmarkSettingsModal Error States:**

*401 on GET /api/benchmarks/settings* — Close the modal and redirect to `/` with a 'Session expired' toast.

*403 on GET /api/benchmarks/settings* — Show 'Access Denied' inside the modal body. Don't close the modal — let the user dismiss it.

*404 on GET /api/benchmarks/settings* — This is a valid state (user has never saved settings). Show 'Settings not found — defaults will be used' with a 'Create Settings' CTA that calls PATCH /api/benchmarks/settings with the system defaults. Don't treat this as a hard error.

*500 on GET /api/benchmarks/settings* — Show an error banner inside the modal with a 'Retry' button that re-fetches GET /api/benchmarks/settings.

*401 on PATCH /api/benchmarks/settings* — Same as above: attempt token refresh, retry once, redirect on failure.

*429 on PATCH /api/benchmarks/settings* — Show 'Too many updates. Please wait before saving again.' inside the modal. Keep the form values intact so the user doesn't lose their changes.

*500 on PATCH /api/benchmarks/settings* — Show an error banner inside the modal with a 'Retry' button. Keep the form values intact.

*Successful save* — Show a 'Settings saved' toast (non-blocking, auto-dismisses). Close the modal. Trigger a re-fetch of GET /api/benchmarks/performance in the parent tab so the updated targets are reflected immediately.

**General** — All error messages should be in plain English, not raw HTTP status codes or Firestore error strings. Log the raw error to the console for debugging but show the user something human-readable.
