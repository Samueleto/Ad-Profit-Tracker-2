Let's make sure the network configuration feature is properly secured end-to-end.

All seven endpoints — GET /api/networks/config/list, PATCH /api/networks/config/update, PATCH /api/networks/config/reorder, POST /api/networks/config/test-connection, POST /api/networks/sync-all, GET /api/networks/config/:networkId, and DELETE /api/networks/config/reset — must verify the Firebase ID token on every request using admin.auth().verifyIdToken(). The UID must always come from the verified token, never from the request body or query params. Return 401 immediately for any missing, expired, or invalid token.

For the networkConfigs Firestore collection, make sure the security rules enforce that users can only read and write documents where userId matches their own auth UID. The immutable fields — createdAt and userId — must not be writable after document creation. Soft-delete behavior should be enforced: documents should be marked as deleted rather than actually removed.

For PATCH /api/networks/config/update specifically, enforce a server-side field whitelist: only isActive, syncSchedule, endpointOverride, timeoutSeconds, retryAttempts, and displayOrder can be updated. The fields dataRole, userId, and createdAt must be silently ignored or explicitly rejected if included in the request body — they cannot be changed through this endpoint.

For POST /api/networks/config/test-connection and POST /api/networks/sync-all, the API keys fetched from the apiKeys subcollection must be decrypted server-side only using the cryptr/AES secret stored in server environment variables. The decrypted key must never appear in any API response, log output, or error message. The probe request in test-connection must be made from the server using node-fetch — never expose the key to the client.

Rate limiting: apply per-user rate limits on the mutation endpoints. POST /api/networks/sync-all should be limited to 3 requests per hour per user to prevent hammering the ad network APIs. POST /api/networks/config/test-connection should be limited to prevent abuse. PATCH endpoints should have reasonable limits (e.g. 30/min) to prevent config spam.

For the endpointOverride field, validate on write that the value is either null or a well-formed HTTPS URL. Reject plain HTTP URLs and anything that doesn't parse as a valid URL. This prevents someone from pointing the system at an internal network address or a data-exfiltration endpoint.
