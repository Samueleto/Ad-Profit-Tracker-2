We need to make sure every error scenario in the ScheduleReportModal fails gracefully and gives the user a clear path forward. For this step, wire up all the error states described in the modal's spec.

**401 — Session expired:** If any API call returns 401 and the token refresh retry also fails, close the modal immediately and redirect the user to / with a toast that says 'Session expired. Please sign in again.' Don't show an error inside the modal — just close it and redirect. This is handled in the useScheduleForm hook.

**403 — Access denied:** If any API call returns 403, show an 'Access Denied' message inside the modal body (replace the form content) with a close button. Don't show the form. This shouldn't happen in normal usage but could if permissions change mid-session.

**404 — Report not found:** If GET /api/schedules or POST /api/schedules/create returns 404 (meaning the reportId doesn't exist in the user's savedReports), show a 'Report not found' message inside the modal with a close button. Replace the form content — don't show a broken form.

**400 — Validation errors:** If POST /api/schedules/create or PATCH /api/schedules/:id returns 400, the response should include field-level errors. Display these inline next to the relevant form fields (e.g., red text under the email field saying 'Invalid email address'). Also handle the special 400 case 'Maximum 10 scheduled reports allowed' — show this as a banner at the top of the modal, not a field error.

**409 — Duplicate schedule:** If POST /api/schedules/create returns 409 ('A schedule already exists for this report'), this means the GET /api/schedules fetch missed an existing schedule. Handle this gracefully: show a banner saying 'A schedule already exists for this report. Loading it now...' and then call GET /api/schedules?reportId=... again to load and display the existing schedule for editing.

**429 — Rate limited:** Show a specific message based on which endpoint was rate limited (as described in the security step). Display this as an inline banner inside the modal, not a toast. Include how long to wait if the API provides that information.

**500 — Server error:** Show an inline error banner inside the modal with a 'Retry' button. The banner should say something like 'Something went wrong. Please try again.' Don't close the modal — let the user retry without losing their form input.

**502 — Email delivery failure (send-now only):** If POST /api/schedules/:id/send-now returns 502, show a toast saying 'Test email failed to send. Please try again.' This is a transient error — don't show it inside the modal, just a toast.

**Network errors:** If a fetch call throws (no response at all — network offline, timeout), show an inline banner inside the modal saying 'Connection error. Check your internet and retry.' with a Retry button.

**Loading error (GET /api/schedules fails):** If the initial schedule fetch fails with a 500, show the error banner inside the modal with a Retry button that re-runs fetchSchedule(). Don't show a broken empty form.

Make sure none of these error states lose the user's form input. If they've typed a new email and the save fails with a 500, their email should still be in the field when they retry.
