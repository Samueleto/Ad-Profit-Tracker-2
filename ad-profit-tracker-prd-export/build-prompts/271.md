We're building a data caching and optimization layer for a Next.js + Firebase app. For this step, I need you to wire up the client-side SWR caching behavior so it reads its configuration dynamically from the `networkConfigs` Firestore collection.

Each document in `networkConfigs` can have three caching fields: `clientCacheStaleSeconds` (how long data is considered fresh before SWR revalidates in the background), `clientCacheFocusRevalidate` (whether SWR should re-fetch when the browser tab regains focus), and `serverCacheTTLSeconds` (used server-side, but we'll read it here so the UI can display it). All three have fallback defaults: 60 seconds for stale time, true for focus revalidation, and 3600 for server TTL.

Create a custom hook — something like `useNetworkCacheConfig(networkId)` — that reads the relevant `networkConfigs` document from Firestore for a given network ID and returns the three config values with their defaults applied. This hook should itself use SWR (or a Firestore real-time listener) so the config stays fresh without a page reload.

Then create a second hook — something like `useCachedNetworkData(networkId, fetcher)` — that uses the config from the first hook to build the SWR options object dynamically. It should pass `dedupingInterval` and `refreshInterval` based on `clientCacheStaleSeconds`, and set `revalidateOnFocus` from `clientCacheFocusRevalidate`. Any data-fetching hook in the app that calls ad network APIs should use this wrapper instead of calling SWR directly, so the caching behavior is centrally controlled per network.

Also create a small utility function that formats cache TTL values into human-readable strings (e.g., 3600 → '1 hour', 60 → '1 minute') — this will be useful when displaying cache config in the UI.

Make sure the hooks handle the case where the `networkConfigs` document doesn't exist yet (fall back to defaults gracefully, don't throw).
