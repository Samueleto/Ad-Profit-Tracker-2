Now that the RBAC state hooks exist, we need to wire them into the rest of the app so that UI elements are shown or hidden based on the current user's effective permissions.

**Add the Permissions tab to TeamManagementPage** — Open the existing TeamManagementPage at /settings/team and add the PermissionsTab as a new tab alongside the existing ones. The tab label should be 'Permissions'. No new route is needed — this is purely a tab addition within the existing page.

**Gate the 'Customize Role Permissions' button** — Inside PermissionsTab, the button that opens RolePermissionsEditModal should only render if `useMyPermissions` returns a workspaceRole of 'owner' or 'admin'. Don't just hide it — conditionally render it so it's not in the DOM at all for Members and Viewers.

**Gate the 'Copy Role' dropdown** — Similarly, only render this for Owners (workspaceRole === 'owner').

**Wire up the modal's role selector restrictions** — In RolePermissionsEditModal, the segmented control should only show 'Admin' as an option if the current user is an Owner. Admins should only see 'Member' and 'Viewer'. This is display-only enforcement — the server enforces the real hierarchy — but it prevents confusing UX.

**App-wide permission gates** — Use `useMyPermissions` and the `checkPermission` utility to gate other features across the app. Specifically:
- Any 'Export Data' button or action should check `canExportData`
- Any 'Manage API Keys' section should check `canManageApiKeys`
- Any 'Sync Networks' action should check `canSyncNetworks`
- Any 'Manage Team' navigation item or action should check `canManageTeam`
- Any 'View Audit Logs' link should check `canViewAuditLogs`
- Any 'Manage Benchmarks' section should check `canManageBenchmarks`

For each of these, if the user lacks the permission, either hide the element entirely or show it in a disabled state with a tooltip like 'Your role doesn't include access to this feature' — use your judgment based on context (hiding is better for nav items, disabled+tooltip is better for inline action buttons so users understand why they can't act).

**POST /api/rbac/check integration** — For any server-side operations that are particularly sensitive (like triggering a network sync or exporting data), wire in a call to POST /api/rbac/check as a pre-flight check before the main action fires. This gives a server-side confirmation that the permission is still valid even if the client-side cache is slightly stale.
