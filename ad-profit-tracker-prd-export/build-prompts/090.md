This feature reads from and writes to four Firestore collections: networkConfigs, adStats, apiKeys, and auditLogs. We need the right indexes and security rules in place so the queries run fast and the data stays secure.

Indexes to create — these are the queries the sync endpoints will run:

For the daily-sync master endpoint, it does a collection-group query on networkConfigs filtering isActive==true AND syncSchedule=='daily'. Create a composite index on the networkConfigs collection group for these two fields.

For sync-status, it queries networkConfigs where userId==uid (simple, likely covered by default), and queries adStats where userId==uid AND networkId==network ordered by date DESC. Create a composite index on adStats for (userId, networkId, date DESC).

For sync-history, it queries auditLogs where userId==uid AND action=='manual_sync_triggered' AND resourceType=='sync', optionally also filtering resourceId==networkId, ordered by createdAt DESC. Create a composite index on auditLogs for (userId, action, resourceType, resourceId, createdAt DESC) — this covers both the filtered and unfiltered cases.

For retry-failed, it queries auditLogs for the most recent failed entry for a user+network. The index above covers this.

Security rules — since all writes in this feature go through the Firebase Admin SDK (which bypasses security rules), the rules mainly protect against direct client-side access:
- networkConfigs: authenticated users can read their own documents (userId==request.auth.uid). No client writes.
- adStats: authenticated users can read their own documents. No client writes.
- apiKeys: authenticated users can read their own documents. No client writes.
- auditLogs: authenticated users can read their own documents. No client writes.

All actual writes happen server-side via Admin SDK, so the rules just need to lock down reads appropriately.
