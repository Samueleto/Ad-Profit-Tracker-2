This feature reads from Firestore in two places: the health check endpoint (GET /api/health) does a lightweight connectivity probe against the `users` collection, and the deployment status endpoint (GET /api/deployment/status) looks up a user's document to verify their `workspaceRole` before returning sensitive infrastructure metadata.

We don't need to create any new collections or documents — both of these read from the existing `users` collection. But we do need to make sure a couple of things are in place.

First, confirm that the `users` collection exists in Firestore and that documents in it have a `workspaceRole` field. The deployment status endpoint checks this field for the values `'owner'` or `'admin'` — if your existing user documents use different field names or different role values, note that now so we can adjust the endpoint logic accordingly.

Second, make sure the Firebase Admin SDK's service account has read access to the `users` collection. In most setups this is automatic (Admin SDK bypasses security rules), but if your project has any custom IAM restrictions on Firestore, the service account running the Cloud Run service needs the `roles/datastore.user` IAM role.

Third, for the health check's 2-second timeout on the Firestore probe — Firestore's Node.js SDK doesn't have a built-in per-query timeout, so we'll implement this using `Promise.race()` with a `setTimeout` that rejects after 2000ms. Document this pattern in `src/lib/deployment/` as a reusable `withTimeout(promise, ms)` utility that the health endpoint will use.

No migrations or schema changes needed — this feature is read-only against existing data.
