We're building the state layer for the OnboardingWizardPage. The page has a few distinct pieces of state that need to be managed cleanly so the steps don't handle data fetching themselves.

**Wizard step state:** Track the current step (1–4) and whether the user chose to skip on Step 3. This can live in local React state inside the wizard — it doesn't need to persist across page refreshes (if the user refreshes mid-wizard, they just start from Step 1 again, which is fine since onboardingCompletedAt isn't written until the end).

**User profile state:** When the wizard mounts, fetch the current user's profile from GET /api/auth/get-user using the Firebase ID token. Store the result — specifically displayName, preferences.timezone, and preferences.defaultDateRange — so Step 2's form can be pre-filled. Track a loading state (shows the skeleton shimmer) and an error state.

**Step 2 form state:** Use react-hook-form to manage the profile setup form. Initialize it with the values fetched from GET /api/auth/get-user once they arrive. The form should track: displayName (string, required), timezone (string, required — default to the user's existing timezone or 'UTC' if none), and defaultDateRange (enum, default to 'last7days' if not set). Track a submitting state separately from the form's own isSubmitting so you can disable fields and show the spinner.

**Completion state:** After the final PATCH /api/auth/update-profile call succeeds (writing onboardingCompletedAt), update local state to reflect completion and trigger the redirect to /dashboard. If the user clicked 'Skip for now', make sure onboardingSkipped: true is included in that final PATCH call.

Keep all of this state close to the OnboardingWizardPage — there's no need for a global store here since onboarding is a one-time flow. The wizard component should pass down only what each step needs via props.
