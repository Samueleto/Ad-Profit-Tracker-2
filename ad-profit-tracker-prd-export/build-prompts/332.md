The ComparativeNetworkAnalysisTab inherits the FinancialMetricsDashboardPage's Firebase Auth middleware, but we need to make sure the security is airtight for the specific calls this tab makes.

**Bearer token on every request** — Every call to GET /api/networks/comparison and POST /api/sync/manual must include the Firebase ID token in the Authorization header as `Bearer <token>`. Get the token via `getIdToken()` from firebase/auth immediately before each request — don't cache the token in component state, as it can expire. The `useComparativeAnalysis` hook should always call `getIdToken()` fresh at the start of each fetch.

**Token refresh on 401** — If GET /api/networks/comparison returns a 401, the hook should call `getIdToken(true)` to force a token refresh, then retry the request exactly once with the new token. If the retry also returns 401, treat it as a session expiry: redirect the user to `/` (the root/login page) and show a 'Session expired' toast notification. Do not retry more than once — infinite retry loops are a real risk here.

**Date range cap** — Before sending the request, validate client-side that the date range between dateFrom and dateTo does not exceed 90 days. If the parent dashboard somehow passes a range larger than 90 days, clamp it or show an error rather than sending the out-of-range request. This is a defense-in-depth measure since the API also validates this, but catching it client-side gives a better user experience.

**No sensitive data in client state** — The API response from GET /api/networks/comparison should not contain any API keys, admin credentials, or raw Firestore document IDs beyond what's needed for display. When storing the response in `comparisonData` state, don't log it to the console in production builds.

**403 handling** — If the API returns 403, show 'Access Denied' inline within the tab panel (not a redirect) with a link to /dashboard. This is an authorization issue (the user is authenticated but not permitted), not an authentication issue, so don't redirect to login.

**Cache awareness** — The API uses node-cache with a 5-minute TTL and returns an `X-Cache: HIT` header on cache hits. The client doesn't need to do anything special with this, but it's worth noting that the 'Sync All' button should bypass the cache by triggering a fresh sync before re-fetching — the POST /api/sync/manual call invalidates the underlying data, so the subsequent GET /api/networks/comparison fetch should get fresh results.
