We're building the ROI Calculation Engine. For this step, I need you to connect all the pieces together so the ROIIndicatorSection works end-to-end within the existing FinancialMetricsDashboardPage, and wire up the threshold configuration so users can customize their ROI color-coding.

**Wiring ROIIndicatorSection into the dashboard**

The FinancialMetricsDashboardPage already has a date range filter and calls GET /api/dashboard/metrics. You need to:
- Pass the dashboard's current `dateFrom` and `dateTo` values down into ROIIndicatorSection as props (or read them from shared state if the dashboard uses a context/store)
- Inside ROIIndicatorSection, use `useROIMetrics` with those date range values to drive the main KPI card
- Use `useROIBreakdown` with `dimension='network'` and the same date range to drive the per-network ROI rows
- When the dashboard's date range changes, both hooks should automatically re-fetch — make sure the date range state flows down correctly so this happens without any manual wiring
- The recharts breakeven line (y=0 reference line) should be added to the existing daily profit trend chart component — find that chart in the codebase and add a `<ReferenceLine y={0} stroke='#9CA3AF' strokeDasharray='4 4' />` to it
- The collapsible ROI Breakdown panel should read `totalRevenue`, `totalCost`, and `roi` from the `kpis` data already fetched by GET /api/dashboard/metrics — don't call a separate endpoint for this panel's formula display

**Wiring threshold configuration**

Users need a way to view and edit their ROI thresholds. Add a small 'Configure Thresholds' link or gear icon within the ROIIndicatorSection that opens a settings panel or modal. This panel should:
- Use `useROIThresholds` to load and display current threshold values
- Show input fields for positiveThreshold, warningThreshold, criticalThreshold, targetRoi, and toggles for alertOnNegative and alertOnTargetMiss
- Show which fields are using system defaults (from the `usingDefaults` array) with a subtle 'default' label
- Validate threshold ordering inline as the user types (criticalThreshold <= warningThreshold <= positiveThreshold)
- On save, call `updateThresholds` from the hook — the optimistic update means the ROI card's color coding should update immediately
- After a successful save, show a success toast and close the panel
- After a failed save, show an error message and keep the panel open

**Connecting thresholds to color coding**

Right now the ROI card uses hardcoded logic (roi > 0 = green). Update it to use the thresholds from `useROIThresholds` instead: roi >= positiveThreshold = green, roi between warningThreshold and positiveThreshold = amber, roi < warningThreshold = red. This makes the color coding dynamic based on the user's saved preferences.

**Retry wiring**

The retry button on the error_500 state inside the ROI card should call the `refetch` function from `useROIMetrics`. Make sure this is wired up so clicking retry actually re-calls GET /api/dashboard/metrics (or GET /api/roi/compute — whichever the card is reading from for its primary value).
