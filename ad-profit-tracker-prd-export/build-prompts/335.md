We need to make sure both export endpoints are properly locked down and that the client is sending credentials correctly.

On the client side, every request to GET /api/export/preview and POST /api/export/excel must include a valid Firebase ID token in the Authorization header as a Bearer token. Get this token by calling firebaseAuth.currentUser?.getIdToken() — this returns a fresh token (or a cached one if it's still valid). Do this immediately before each API call, not once at component mount, so the token is always fresh.

On the server side (both endpoints), the first thing that happens is verifying the token via firebase-admin's admin.auth().verifyIdToken(token). The uid used for all Firestore queries must come exclusively from this verified token — never from the request body, query params, or any client-supplied value. This ensures users can only ever export their own data.

For the POST /api/export/excel endpoint specifically, make sure:
- The filename parameter is sanitized server-side (alphanumeric, dash, underscore only, max 100 chars) even if the client already validated it
- The sheets array is validated against the exact allowed list: ['summary', 'daily_trend', 'geo_breakdown', 'exoclick', 'rollerads', 'zeydoo', 'propush', 'activity_log'] — reject any unknown values with a 400
- The date range is validated: both dates required, valid YYYY-MM-DD format, dateFrom <= dateTo, max 90-day range
- Rate limiting is enforced: max 10 exports per hour per user — use a simple in-memory counter keyed by uid, or a Firestore-based counter if the app runs on multiple instances. Return 429 with a clear message if exceeded
- No API keys, secrets, or sensitive server-side values are included anywhere in the generated .xlsx file
- The audit log entry is written to the auditLogs Firestore collection after a successful export (action='manual_sync_triggered', resourceType='sync', resourceId='export', with metadata including sheetsIncluded, dateFrom, dateTo, rowCount, filename)

For GET /api/export/preview, the node-cache key must include the uid ('{uid}_export_preview_{dateFrom}_{dateTo}') so one user's cached preview can never be served to another user.
