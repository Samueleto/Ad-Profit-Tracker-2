We're building state management for the per-network analytics tabs. The key challenge here is coordinating lazy loading across four tabs, keeping data in sync with the parent dashboard's date range, and avoiding redundant API calls.

Create a custom hook called `useNetworkAnalytics` that manages all data for the PerNetworkAnalyticsTabsSection. Here's what it needs to track:

**Per-network data cache** ‚Äî An object keyed by networkId ('exoclick', 'rollerads', 'zeydoo', 'propush'). Each entry holds:
- `summary` ‚Äî the KPI totals from `GET /api/networks/stats` with `groupBy=total`
- `series` ‚Äî the daily time-series from `GET /api/networks/stats` with `groupBy=daily`
- `countries` ‚Äî the country breakdown from `GET /api/networks/stats` with `groupBy=country`
- `networkStatus` ‚Äî lastSyncedAt, lastSyncStatus, circuitBreakerOpen
- `loadingState` ‚Äî one of 'idle' | 'loading' | 'success' | 'error' | 'empty'
- `error` ‚Äî error details if loadingState is 'error'
- `loadedForDateRange` ‚Äî the dateFrom+dateTo string this data was loaded for, so we know when to invalidate

**Active tab** ‚Äî Track which tab is currently selected. Default to 'exoclick'.

**Load on activation** ‚Äî When a tab becomes active, check if data exists for the current date range. If not (idle or stale), fire three parallel requests to `GET /api/networks/stats` for that networkId ‚Äî one for each groupBy value (total, daily, country). Use Promise.all so all three resolve together before setting the state to 'success'. While loading, set that network's loadingState to 'loading'.

**Date range invalidation** ‚Äî Accept `dateFrom` and `dateTo` as inputs (coming from the parent dashboard's shared state). When either changes, clear the `loadedForDateRange` for all four networks and immediately re-fetch the currently active tab. Other tabs stay 'idle' until activated.

**Sync action** ‚Äî Expose a `syncNetwork(networkId)` function that calls `POST /api/sync/manual`. After a successful sync, invalidate that network's cached data and re-fetch it.

Create a separate hook called `useApiExplorer` for the ApiExplorerTab. It tracks:
- Per-network raw response data fetched from the four raw-response endpoints (`GET /api/networks/exoclick/raw-response`, etc.)
- Loading state per network accordion
- Test connection results per network (from `POST /api/networks/config/test-connection`)
- Whether each network has an API key configured (derived from the raw-response endpoint response)

Raw responses should only be fetched when the user expands that network's accordion (lazy, same pattern as the tab loading). Test connection results are ephemeral ‚Äî don't persist them, just show inline after the call.

Both hooks should attach the Firebase ID token as a Bearer token on every API call. Use `getIdToken()` from `firebase/auth` before each request. If a request returns 401, attempt a token refresh with `getIdToken(true)` and retry once before surfacing the error.

Keep all this state local to the dashboard section ‚Äî don't push it into a global store. The parent dashboard only needs to pass down `dateFrom` and `dateTo`.
