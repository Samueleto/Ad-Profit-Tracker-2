We're building a Propush ad network integration. For this step, I need you to make sure every endpoint is properly secured so users can only access their own data and the internal sync endpoint can't be called by end users.

Here's what needs to be locked down across all six endpoints:

**User-facing endpoints** (POST /api/networks/propush/sync, GET /api/networks/propush/stats, GET /api/networks/propush/stats/by-country, GET /api/networks/propush/stats/latest, GET /api/networks/propush/raw-response):
- Every one of these must verify the Firebase ID token via `admin.auth().verifyIdToken()` at the very start of the handler, before doing anything else. If the token is missing, malformed, or expired, return 401 immediately.
- After verifying the token, extract the `uid` and use it to scope ALL Firestore queries. Never trust a userId from the request body or query params — always use the uid from the verified token.
- For the sync endpoint specifically, add a rate limit check: no more than 10 sync requests per hour per user. Store the rate limit state in Firestore or a fast cache. Return 429 with a clear message if exceeded.
- The sync endpoint must also check that the user's networkConfig for Propush is active before proceeding — return 400 if it's inactive or missing.
- The API key fetched from the apiKeys collection must be decrypted server-side only and must never appear in any API response, log output, or error message.

**Internal endpoint** (POST /api/networks/propush/scheduled-sync):
- This endpoint must NOT verify a Firebase ID token — it uses a different auth mechanism.
- Validate the `x-internal-secret` header against the `INTERNAL_SYNC_SECRET` environment variable. If it's missing or wrong, return 401 immediately.
- Add a check that this endpoint is never reachable from the browser — consider IP allowlisting or at minimum ensure the secret is long and random.
- Make sure `INTERNAL_SYNC_SECRET` is set in your environment config and is never committed to source control.

**General hardening across all endpoints**:
- Validate all date inputs (YYYY-MM-DD format, max 90-day range) and return 400 with a descriptive error if they're invalid.
- Validate the `limit` param on the by-country endpoint (integer, 1–100).
- Make sure audit log entries written to the auditLogs collection via the Admin SDK include the uid, the action taken, and a timestamp — but never include the decrypted API key.
