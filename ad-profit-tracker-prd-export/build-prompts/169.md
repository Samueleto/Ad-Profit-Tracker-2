We need three API endpoints for the caching system. Here's what each one does:

**POST /api/cache/invalidate** — Lets authenticated users manually clear their cached data. First, verify the Firebase ID token from the Authorization header using the Admin SDK. Then validate the request body with `CacheInvalidateRequestSchema` — `scope` must be `'all'`, `'network'`, or `'endpoint'`; `keys` is optional. Apply a rate limit of 15 requests per hour per user (store rate limit state in a simple in-memory map keyed by UID, or use a lightweight approach that fits the project). Based on the scope: if `'all'`, use `getUserKeys(uid)` to find and delete all cache entries for this user; if `'network'`, delete all keys matching `{uid}_stats_{networkName}_*` for each name in `keys`; if `'endpoint'`, delete the exact keys provided (after verifying they belong to the user's UID namespace). Write an entry to the `auditLogs` Firestore collection with `action: 'cache_invalidated'`, the user's UID, timestamp, and details including `scope` and `invalidatedCount`. Return `{ invalidatedCount, scope }` with a 200 status.

**GET /api/cache/status** — A read-only debugging endpoint. Verify the Firebase ID token. Access the shared node-cache instance and call `.getStats()` to get hits, misses, key count, and memory size. Then call `getUserKeys(uid)` to get only the keys belonging to this user. Return `{ hits, misses, keyCount, size, keys }` — where `keyCount` and `keys` reflect only the user's entries, not the whole cache. Return 200 on success, 401 if unauthenticated.

**POST /api/cache/warm** — An internal-only endpoint for pre-populating the cache, triggered by Cloud Scheduler (not by end users). Instead of checking a Firebase token, validate the `x-internal-secret` header against an `INTERNAL_SECRET` environment variable. If it doesn't match, return 401 immediately. The request body contains `tasks: [{ endpoint: string, params: object }]`. For each task, invoke the corresponding internal data-fetching logic (for example, if `endpoint` is `'stats/summary'`, call the same core function that `GET /api/stats/summary` uses internally — don't make an HTTP call to yourself). Store the result in node-cache using the appropriate key. Track how many tasks succeed and fail. Write an `auditLogs` entry with `action: 'cache_warmed'` and `{ tasksProcessed, tasksFailed }`. Return `{ tasksProcessed, tasksFailed }` with 200.

All three endpoints should import the shared cache instance from `cacheProvider.ts` and use the key helpers from `cacheKeys.ts`.
