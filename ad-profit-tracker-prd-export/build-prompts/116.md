We need a TypeScript type for the `users` collection document and some helper utilities so every part of the app works with the same shape of user data.

**User Type (`features/auth/types/user.ts`):**
Create a `UserProfile` interface that matches the `users` collection schema exactly:
- `uid: string` — the Firestore document ID, matches Firebase Auth UID
- `email: string`
- `displayName: string | null`
- `photoURL: string | null`
- `authProvider: 'google' | 'email'`
- `createdAt: Timestamp` — Firestore Timestamp type
- `lastLoginAt: Timestamp`
- `role: 'user' | 'admin'`
- `preferences: { defaultDateRange: string; timezone: string }`

Also create a `UserProfileUpdate` type that picks only the mutable, user-editable fields: `displayName` and `preferences` (both optional). This is what `PATCH /api/auth/update-profile` accepts and what the update form will send.

**Firestore Document Helpers (`features/auth/types/user.ts` or a separate `lib/firebase/converters.ts`):**
Add a Firestore data converter for `UserProfile` so that when we read from or write to `users/{uid}`, TypeScript knows the shape. This uses Firestore's `withConverter()` pattern — implement `toFirestore` and `fromFirestore` methods.

**API Response Type:**
Create a `UserProfileResponse` type for what the API endpoints return — this is the same as `UserProfile` but with `createdAt` and `lastLoginAt` serialized as ISO strings (since Firestore Timestamps can't be JSON-serialized directly). The API routes will convert Timestamps to `.toDate().toISOString()` before returning.

Export all types from a single `features/auth/types/index.ts` barrel file so imports stay clean throughout the app.
