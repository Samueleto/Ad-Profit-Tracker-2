**Firestore Security Rules for the `users` collection:**
Write rules that enforce the following:
- Create: only allowed if the requesting user's Auth UID matches the document ID being created (`request.auth.uid == userId`). This prevents one user from creating a document on behalf of another.
- Read: only the document owner can read their own document (`request.auth.uid == resource.data.uid`)
- Update: only the document owner can update, AND the update must not change `createdAt` or `uid` (mark these as immutable in the rules). The `role` field should also be immutable via client-side rules â€” role changes should only happen server-side via the Admin SDK.
- Delete: block direct client-side deletes entirely. Deletion only happens through DELETE `/api/auth/delete-account` which uses the Admin SDK server-side.

**API endpoint hardening:**
All four endpoints (POST `/api/auth/sync-user`, GET `/api/auth/get-user`, PATCH `/api/auth/update-profile`, DELETE `/api/auth/delete-account`) use Firebase Admin SDK's `verifyIdToken()` as their auth check. Make sure:
- Every endpoint extracts the Bearer token from the Authorization header and rejects with 401 if it's missing or malformed (not just if it's invalid)
- The `uid` used for all Firestore operations comes from the verified token, never from the request body or query params â€” a user shouldn't be able to pass someone else's UID
- PATCH `/api/auth/update-profile` has a strict whitelist: only `displayName` and `preferences` (with its sub-keys `defaultDateRange` and `timezone`) are accepted. Any attempt to update `uid`, `email`, `role`, `authProvider`, or `createdAt` should return 400
- DELETE `/api/auth/delete-account` should handle partial failure gracefully â€” if the Firestore delete succeeds but `admin.auth().deleteUser()` fails (or vice versa), log the inconsistency with enough detail for manual cleanup

**Client-side hygiene:**
- Never log Firebase ID tokens to the console
- Never store ID tokens in localStorage or sessionStorage â€” always call `getIdToken()` fresh
- The Firebase client config (API key, project ID, etc.) is safe to expose client-side â€” that's by design â€” but make sure the Firebase Admin service account credentials are only in server-side environment variables and never bundled into client code
