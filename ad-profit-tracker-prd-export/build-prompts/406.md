We're building a Cloud Run deployment pipeline for a Next.js + Firebase app. For this step, I need you to make sure all three endpoints handle failures gracefully and return the right HTTP status codes — because Cloud Run's infrastructure makes decisions based on those codes.

For GET /api/health, the error handling is the most critical. If the Firestore connectivity check throws an error or exceeds the 2-second timeout, the endpoint must return HTTP 503 with { status: 'unhealthy', error: string }. The error message should be descriptive enough to diagnose (e.g., 'Firestore timeout after 2000ms') but should not expose internal stack traces or connection strings. If Firestore responds but takes more than 1 second, return HTTP 200 with status: 'degraded' and firestore: 'slow' — this tells Cloud Run the instance is alive but struggling. If memory usage exceeds 400MB (heapUsed from process.memoryUsage()), flag this in the response but still return 200 unless Firestore is also failing. Wrap the entire handler in a try/catch so that any unexpected error returns 503 rather than a 500 with an unhandled exception.

For GET /api/ready, if any required environment variable (FIREBASE_PROJECT_ID, INTERNAL_SYNC_SECRET, EMAIL_PROVIDER_API_KEY, EMAIL_FROM_ADDRESS) is missing, return HTTP 503 with { ready: false, failedCheck: 'envVars' }. If Firebase Admin SDK isn't initialized (admin.apps.length === 0), return 503 with { ready: false, failedCheck: 'firebaseAdmin' }. Never return which specific env var is missing — just report that the envVars check failed. Wrap in try/catch so unexpected errors also return 503.

For GET /api/deployment/status, handle the three distinct failure modes: Firebase token verification failure → 401 with a generic 'Unauthorized' message (don't reveal why the token failed); Firestore role lookup failure → 500 with a generic error (don't expose Firestore error details); missing environment variables when building the response → 500 with a clear message that deployment metadata is unavailable. In all 500 cases, log the actual error server-side for debugging but return a sanitized message to the client.
