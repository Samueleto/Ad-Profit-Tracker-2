We're using Firestore as the database, and we need to make sure the right indexes exist so our queries run fast, and the right security rules are in place so users can only see their own data.

**Indexes to create** — Add these composite indexes to your `firestore.indexes.json`:

- `adStats` collection: `(userId ASC, networkId ASC, date ASC)` — used by the stats, by-country, and raw-response endpoints to filter by user + network + date range
- `adStats` collection: `(userId ASC, networkId ASC, date DESC)` — used by the latest stats endpoint which orders by date descending
- `networkConfigs` collection: `(networkId ASC, isActive ASC, syncSchedule ASC)` — used by the scheduled sync endpoint's collection-group query to find all active Propush configs
- `auditLogs` collection: `(userId ASC, createdAt DESC)` — for any future audit log queries

**Security rules** — In `firestore.rules`, make sure:
- `adStats`: authenticated users can only read documents where `userId == request.auth.uid`. No client-side writes (all writes go through the Admin SDK via API routes).
- `apiKeys`: authenticated users can read their own documents. No client-side writes.
- `networkConfigs`: authenticated users can read and write their own documents (where `userId == request.auth.uid`).
- `auditLogs`: authenticated users can read their own logs. No client-side writes.

All write operations to `adStats` and `auditLogs` happen server-side via the Admin SDK, which bypasses these rules — that's intentional and correct.
