We're building a dark mode and theming feature for a Next.js + Firebase app. For this step, I need you to add a new 'Appearance' section — called ThemeSettingsSection — into the existing SettingsPage. Don't create a new route. Instead, integrate it as a new tab or panel within the existing SettingsPage tab layout at /settings.

Here's what ThemeSettingsSection should contain:

**Theme mode selector** — A three-option segmented control with icon+label pairs: a sun icon for 'Light', a moon icon for 'Dark', and a monitor icon for 'System' (which follows the OS preference). Use the next-themes package to apply the selected theme. The active option should be visually distinct.

**Accent color picker** — A horizontal row of 8 preset color swatches, each a 32px circle. The preset colors are: indigo, emerald, rose, amber, sky, violet, orange, and slate. The currently active swatch shows a checkmark. Beyond the 8 presets, include a 'Custom' option — when the user clicks it, reveal a hex input field (progressive disclosure — hidden until 'Custom' is clicked). The selected accent color should be applied as a CSS custom property so it cascades through the UI immediately.

**Live preview strip** — A compact card, about 120px tall, that shows a miniature KPI card mockup rendered in the currently selected theme and accent color. This updates in real time as the user changes the controls — before they save anything.

**Save and reset** — A 'Save Appearance' button that sends a PATCH request to /api/settings/preferences with the body `{ themePreference, accentColor }`. Below it, a 'Reset to Defaults' link that restores the theme to 'Light' and the accent color to indigo (both in the UI and via another PATCH call).

**On mount** — Call GET /api/settings/preferences to load the user's saved values and pre-populate all controls. While this is loading, show shimmer placeholders in place of the segmented control and swatches.

All changes should apply optimistically to the page immediately when the user interacts with the controls. The save only persists them to the backend. Show a 'Saved' toast on a successful PATCH response.
