We need two UI components for the RBAC feature. Both live in src/features/rbac/components/ and both integrate into the existing TeamManagementPage at /settings/team.

**PermissionsTab** — This is a tab panel that gets added to the existing TeamManagementPage. It's not a new page or route — it's a new tab called 'Permissions' added to the tab bar on /settings/team.

At the top of the tab, render a role badge legend: four colored badges mapping Owner (indigo), Admin (blue), Member (gray), and Viewer (teal) to their role names.

Below that, the main content is a permissions matrix table. The table has a sticky first column showing permission category names (Dashboard View, Export Data, Manage API Keys, Sync Networks, View Reports, Create Reports, Manage Team, Manage Billing, View Audit Logs, Manage Benchmarks) and four columns for Owner, Admin, Member, and Viewer — each column header uses the role's color. Each cell shows a green checkmark (✓) or red X based on the role's permission value from GET /api/rbac/permissions. Use compact row height to keep the table dense but scannable. Tailwind semantic green/red classes for the icons.

Below the matrix, add a 'Role Descriptions' accordion (collapsed by default) that expands to show a plain-language description of each role's intended use case.

Below that, an 'Effective Permissions' section showing the current user's own resolved permissions from GET /api/rbac/my-permissions. Display this as a two-column card layout — each permission key shown with a granted/denied indicator. This helps members understand what they can and cannot do.

For users with Owner or Admin workspaceRole, show a 'Customize Role Permissions' button that opens the RolePermissionsEditModal. Also show a 'Copy Role' dropdown that lets Owners duplicate an existing role's permission set as a starting point (this pre-populates the modal with the copied role's values).

Loading state: shimmer rows matching the matrix structure, with the 4 column headers rendering immediately. Error states: inline banners for 403 ('Access Denied — insufficient permissions to view role matrix'), 404 ('Permissions configuration not found' with retry), and 500 (error banner with Retry button that re-calls GET /api/rbac/permissions). For 401 errors, call getIdToken(true) to refresh the Firebase token — if that fails, redirect to / with a 'Session expired' toast.

**RolePermissionsEditModal** — A centered modal (max-w-lg) with a backdrop overlay, triggered by the 'Customize Role Permissions' button in PermissionsTab.

At the top, a segmented control role selector with options: Admin, Member, Viewer (Owner is intentionally excluded — Owner permissions are immutable). Switching the selector changes which role's permissions are shown.

The modal body is a scrollable list of all permission keys grouped into categories: Data Access (canViewDashboard, canExportData), Network Management (canSyncNetworks, canManageApiKeys), Team Management (canManageTeam), Export & Reports (canViewReports, canCreateReports), Settings (canViewAuditLogs, canManageBenchmarks). Each row shows the permission name, a brief description, and a toggle switch. Pre-populate values from the GET /api/rbac/permissions response when the modal opens.

If the logged-in user is an Admin (not Owner), disable the Admin tab in the segmented control — admins can only edit Member and Viewer.

Add inline validation: prevent saving if all permissions for a role would be turned off. Show a warning banner if a change would remove a permission that active workspace members currently rely on (you can approximate this by checking if any users have that permission currently granted in their effectivePermissions).

At the bottom: 'Reset to Defaults' button (bottom-left, styled as destructive) calls PATCH /api/rbac/permissions with resetToDefaults: true. 'Cancel' closes without saving. 'Save Changes' calls PATCH /api/rbac/permissions with the updated permission map.

Saving state: spinner on the Save button, all toggles disabled. On success: show a 'Permissions updated' toast, close the modal, and trigger a refresh of the parent PermissionsTab matrix. Error states: 403 shown inline inside the modal ('Only Owners and Admins can edit role permissions'), 500 shown as an inline error banner with Retry, 401 handled the same way as PermissionsTab.
