We need TypeScript interfaces that match the users collection schema exactly, so every part of the notification system — API routes, hooks, and components — is working with the same shape of data.

Create a notifications types file in the feature's types folder. Define these types:

First, a NotificationType enum (or union type) covering all nine values: 'sync_success', 'sync_failure', 'export_complete', 'export_failure', 'schedule_delivered', 'schedule_failed', 'reconciliation_anomaly', 'circuit_breaker_opened', 'rate_limit_exceeded'. This is used for validation in POST /api/notifications/create and PATCH /api/notifications/preferences.

Second, a NotificationSeverity type: 'info' | 'success' | 'warning' | 'error'.

Third, a Notification interface with all the fields from the users.notifications array: id (string), type (NotificationType), title (string), body (string), networkId (string | null), resourceType (string), resourceId (string | null), isRead (boolean), isDismissed (boolean), severity (NotificationSeverity), actionUrl (string | null), metadata (Record<string, unknown>), createdAt (Firestore Timestamp or ISO string — pick whichever the API serializes to).

Fourth, a NotificationPreference interface: { type: NotificationType, enabled: boolean, isDefault: boolean }.

Fifth, API response types for each endpoint — for example, GetNotificationsResponse with { notifications: Notification[], unreadCount: number, hasMore: boolean, nextCursor: string | null }, DismissNotificationResponse with { dismissed: boolean, notificationId: string, unreadCount: number }, MarkAllReadResponse, ClearNotificationsResponse, CreateNotificationResponse, and PreferencesResponse.

Finally, a partial UserNotificationFields interface representing the notification-related fields on the users document: notifications (Notification[]), unreadNotificationCount (number), notificationsUpdatedAt (Timestamp), notificationPreferences (Record<NotificationType, { enabled: boolean }>).

Keep all these in one file or split logically — just make sure they're all exported and importable from a single entry point.
