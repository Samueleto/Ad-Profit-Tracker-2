We're building the state layer for the EmailAlertPreferencesSection. The component has enough moving parts — 9 independent toggles, a master toggle, an editable email field, per-row save indicators, and a test-send flow — that keeping all of this in local component state will get messy fast. Let's extract it into a clean custom hook.

Create a hook called useEmailAlertPreferences. Here's what it needs to manage:

**Fetching preferences** — On mount, call GET /api/notifications/preferences. Store the full array of 9 preference objects (each with type, enabled, emailEnabled, isDefault), plus alertDeliveryEmail and lastTestEmailSentAt. Track a loading boolean and a fetchError state.

**Per-toggle updates** — Expose a toggleEmailEnabled(type, newValue) function. When called, it should optimistically update the local state immediately (so the UI feels instant), then call PATCH /api/notifications/preferences with { [type]: { emailEnabled: newValue } }. If the PATCH fails, roll back the optimistic update and set an error for that specific row. Track a per-type saving state (a map of type → 'idle' | 'saving' | 'saved' | 'error') so each row can independently show its 'Saved ✓' flash or an error indicator.

**Master toggle** — Derive the master toggle state from the 9 emailEnabled values: true if all are enabled, false if all are disabled, and indeterminate if mixed. Expose a setAllEmailEnabled(value) function that calls PATCH /api/notifications/preferences for all 9 types at once (or batch them into a single call if the API supports it).

**Delivery email** — Store the editable email address in local state, separate from the fetched value. Expose a saveDeliveryEmail() function that calls PATCH /api/notifications/preferences with { alertDeliveryEmail: value }. Track a saving/saved/error state for the email save button.

**Test alert** — Expose a sendTestAlert() function that calls POST /api/email/send-test. Track isSendingTest (boolean) and testError state. On success, update lastTestEmailSentAt in local state from the response's sentAt field.

Keep the hook focused — no business logic in the component itself. The component just reads state and calls the hook's functions.
