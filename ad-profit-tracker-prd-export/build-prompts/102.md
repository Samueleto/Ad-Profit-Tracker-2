The filter endpoints query the `adStats` Firestore collection in several different ways — by userId + date range, by userId + date range + networkId, and by userId + date range + country. We need to make sure the right composite indexes exist so these queries don't fail or time out.

Check the existing `firestore.indexes.json` (or the Firebase console) for the `adStats` collection. We need these composite indexes to exist — add any that are missing:

1. `adStats`: `userId` (ASC) + `date` (ASC) — the base query used by all filter endpoints
2. `adStats`: `userId` (ASC) + `date` (ASC) + `networkId` (ASC) — used when GET /api/filters/options or GET /api/filters/stats filters by a specific network
3. `adStats`: `userId` (ASC) + `date` (ASC) + `country` (ASC) — used when filtering by country
4. `adStats`: `userId` (ASC) + `date` (ASC) + `validationStatus` (ASC) — used by GET /api/filters/stats when dataQuality filter is applied

Also check the `users` collection — the filter preset endpoints read and write `users.savedFilterPresets` (an array field embedded in the user document). No special index is needed for this since it's a direct document read by uid, but confirm the `users` collection has documents keyed by uid.

For the `auditLogs` collection (written by POST /api/filters/presets and DELETE /api/filters/presets/:presetId), confirm it exists or will be auto-created on first write. No special indexes needed there.

Update `firestore.indexes.json` with any missing indexes and deploy them with `firebase deploy --only firestore:indexes`. Note that index builds can take a few minutes on large collections.
