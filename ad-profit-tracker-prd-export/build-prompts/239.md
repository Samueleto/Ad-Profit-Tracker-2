We're building a performance benchmarking feature inside an existing financial metrics dashboard. There are two UI pieces to build here: a tab panel and a modal.

**PerformanceBenchmarkingTab** — This is NOT a new route. Add a 'Benchmarks' tab to the existing FinancialMetricsDashboardPage tab bar, positioned after the Compare tab. The tab panel should only load its data lazily — fire the data fetching when the user first activates this tab, not on page load. When activated, kick off two requests in parallel using Promise.all: GET /api/networks/comparison and GET /api/benchmarks/performance. The date range comes from the parent dashboard's shared state — this tab has no independent date picker of its own.

At the top-right of the tab, show a 'Performance Score' card with a circular gauge displaying a 0–100 score. Color-code it: green-500 for scores ≥ 80, amber-500 for 50–79, and red-500 for below 50. This score comes from GET /api/benchmarks/score.

Below that, show a metric toggle (reuse the MetricToggle component from ComparativeNetworkAnalysisTab) letting users switch between ROI%, CTR%, CPM, Revenue, Cost, Impressions, and Clicks.

The main content is a table of four KPI comparison rows — one per ad network. Each row shows: the actual value (from GET /api/networks/comparison), the historical average (from GET /api/benchmarks/performance with type='historical'), the industry benchmark target (from GET /api/benchmarks/performance with type='industry'), and a MetricShareBar (reused from ComparativeNetworkAnalysisTab) showing where the actual value sits relative to the benchmark range. Color-code rows: green-500 if above benchmark, red-500 if below, amber-500 if within 10% of benchmark. If a network's actual value is null, render '—'. If a network row has a lastSyncStatus of 'failed' or circuitBreakerOpen is true, show an amber warning badge on that row — non-blocking, just informational.

Below the KPI table, add a trend comparison strip: a sparkline (using recharts) showing the user's selected metric over the last 30 days, with a second line overlaid for the historical average. This data also comes from GET /api/benchmarks/performance.

In the tab's header area, include a 'Benchmark Settings' button that opens the BenchmarkSettingsModal.

Handle these states: loading shows a shimmer circle where the gauge goes, four shimmer rows matching the table structure, and a 120px shimmer rectangle for the sparkline area. Empty state (no data for the date range) shows 'No data for this date range. Sync your networks to see benchmark comparisons.' with a 'Sync Now' CTA linking to ManualRefreshPanel. Error states are described in error_handling.

**BenchmarkSettingsModal** — A centered modal (max-w-lg) with a backdrop overlay, triggered by the 'Benchmark Settings' button. When it opens, fetch the user's current settings from GET /api/benchmarks/settings and pre-populate the form.

Show seven metric sections: ROI%, CTR%, CPM, Revenue, Cost, Impressions, Clicks. Each row has a label, the system default shown in muted text, a number input for the custom target, and a toggle to switch between 'Use Industry Default' and 'Custom Target'. When 'Use Industry Default' is active, de-emphasize the input visually. Each row also has a 'Reset to Defaults' button that restores just that metric's system default.

Inline validation: all numeric inputs must be non-negative. ROI% and CTR% cap at 10000. CPM caps at 1000. Show a 'Last updated' timestamp from the saved settings.

At the bottom: 'Reset All' (destructive, bottom-left), and 'Cancel' + 'Save All' (bottom-right). 'Save All' submits all changed fields via PATCH /api/benchmarks/settings. While saving, disable all inputs and show a spinner on the 'Save All' button. Changes are NOT auto-saved — the user must explicitly click 'Save All'. 'Cancel' closes without saving.
