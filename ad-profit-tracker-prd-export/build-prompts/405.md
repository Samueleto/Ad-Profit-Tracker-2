We're building the Help & Documentation Center on Next.js + Firebase. For this step, make sure every error state across HelpCenterPage and HelpArticlePage fails gracefully and gives users a clear path forward.

**HelpCenterPage error states**

- **Initial load failure (500):** If GET /api/help/articles fails on first load, show an inline error banner at the top of the article list area: 'Something went wrong loading articles.' with a Retry button that re-calls GET /api/help/articles. Don't show a full-page error — the category grid might still be usable if it was already loaded.
- **Category filter failure (500):** If the re-fetch after clicking a category fails, show the same inline error banner but keep the previously loaded articles visible. Revert the selected category chip back to its previous state so the UI isn't stuck showing a selected category with no results.
- **Search failure (500):** If GET /api/help/search fails, show a brief notification ('Search temporarily unavailable') but keep the current article list visible. Don't clear the search input — the user shouldn't lose their query.
- **Empty search results (not an error, but handle it):** If the search returns zero results, show 'No articles found for "[query]". Try a different search or browse categories.' with a 'Clear Search' button.
- **Empty category (404):** If a category has no articles, show 'No articles found for this category.' with a 'Browse All' button that clears the category filter.
- **Role-specific guides failure:** If GET /api/rbac/my-permissions or the subsequent role-filtered GET /api/help/articles fails, silently hide the 'Role-Specific Guides' section rather than showing an error — this is supplementary content and its absence shouldn't alarm the user.
- **401 handling:** On any 401 response, attempt getIdToken(true) and retry once. If the retry fails, redirect to / with a 'Session expired' toast.
- **403 handling:** Show an 'Access Denied' card with a link to /dashboard.

**HelpArticlePage error states**

- **Article load failure (500):** Show an inline error banner with a Retry button that re-calls GET /api/help/articles/:articleId. Don't navigate away — let the user retry in place.
- **Article not found (404):** Show a centered 'Article not found' message with a link back to /help and a 'Browse all articles' CTA. No shimmer, no retry — the article genuinely doesn't exist.
- **Access denied (403):** Show 'Access Denied — this article requires elevated permissions' with a link back to /help. Don't show any article content, title, or metadata.
- **Related articles failure:** If GET /api/help/articles?category=...&limit=4 fails, silently hide the 'Related Articles' section. This is supplementary — don't show an error for it.
- **Feedback submission failure (500):** Show a brief notification: 'Couldn't submit feedback. Please try again.' Re-enable the thumbs buttons so the user can retry. Don't show the thank-you message.
- **Feedback already submitted (429):** Show 'You've already rated this article.' and disable both buttons. This is not an error from the user's perspective — treat it gracefully.
- **View increment failure:** Completely silent — this is fire-and-forget. Never surface an error for PATCH /api/help/articles/:articleId/view.
- **401 handling:** Same as HelpCenterPage — attempt token refresh once, redirect to / with toast if it fails.

**General principles**

Never show raw error messages or HTTP status codes to the user. Map every error to a friendly, actionable message. For errors that block the primary content (article load failure, article not found), give a clear path back (link to /help). For errors on supplementary content (related articles, role guides, view increment), fail silently. Always log the actual error to the console in development so developers can debug.
