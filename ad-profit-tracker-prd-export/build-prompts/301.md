Now we need to connect the SettingsPage into the rest of the app so users can actually find and navigate to it.

Add /settings to the app's route configuration. This route should be protected — if the user isn't authenticated, redirect them to /login (or wherever the app's login page lives). After they log in, they should be sent back to /settings rather than the default home page.

In the app's main navigation (sidebar, top nav, or wherever the primary nav lives), add a 'Settings' link that goes to /settings. Place it somewhere logical — typically at the bottom of a sidebar or in a user profile dropdown menu.

Wire up the toast notification system so the SettingsPage can trigger toasts. If the app already has a toast library set up (like react-hot-toast or sonner), make sure the SettingsPage is wrapped within its provider. If there's no toast system yet, add one — the SettingsPage needs it for the 'API key saved successfully' and 'Key disconnected' feedback messages.

Make sure the Firebase Auth context is available on the /settings route. The useApiKeyStatus hook and the form submission handlers all need access to the current user's ID token. If the app has an AuthProvider wrapping the layout, confirm /settings is inside it. If not, add the provider at the appropriate layout level.

Finally, confirm that the /settings page is excluded from any static generation or aggressive caching — it's user-specific and should always be rendered fresh.
