# Step 256 of 437: Set up Zustand store for dashboard date range

We're building shared date range state so that every tab and section of FinancialMetricsDashboardPage reacts to the same selection. For this step, set up a Zustand store that the DateRangeToolbar writes to and all dashboard sections read from.

The store should hold:
- `preset` — the currently active preset label (one of: 'last7', 'last14', 'last30', 'last90', 'thisMonth', 'custom')
- `startDate` — a string in YYYY-MM-DD format
- `endDate` — a string in YYYY-MM-DD format
- `lastSynced` — a timestamp (or null) representing the most recent sync time across all networks — this is what the 'Last synced' badge in the DateRangeToolbar reads
- `dataAvailability` — one of 'complete', 'partial', 'none', 'loading', 'error' — reflecting the result of the most recent GET /api/stats/dates call

The store should expose these actions:
- `setPreset(preset)` — sets the preset and automatically calculates and sets startDate/endDate using date-fns (e.g., `subDays(new Date(), 7)` for last7, `startOfMonth` for thisMonth). Dates should be serialized to YYYY-MM-DD with date-fns `format` before storing.
- `setCustomRange(startDate, endDate)` — sets the preset to 'custom' and stores the provided dates. This should only be called after client-side validation passes (range ≤ 90 days).
- `setLastSynced(timestamp)` — updates the lastSynced field, called by whichever dashboard section receives a fresh sync timestamp from the backend.
- `setDataAvailability(status)` — updates the dataAvailability field, called by the DateRangeToolbar after GET /api/stats/dates resolves.

Initialize the store with a sensible default — 'last30' preset with the corresponding calculated dates. This way, the dashboard loads with a meaningful date range immediately without waiting for user interaction.

Make sure the store is accessible from both the DateRangeToolbar and all other dashboard tabs/sections. Any component that needs to re-fetch when the date range changes should subscribe to `startDate` and `endDate` from this store and trigger their data fetches when those values change.
