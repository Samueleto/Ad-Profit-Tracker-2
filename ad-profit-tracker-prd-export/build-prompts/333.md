The DateRangeToolbar inherits Firebase Auth protection from FinancialMetricsDashboardPage, but we still need to make sure the toolbar's own API call to GET /api/stats/dates is properly secured and that the date range inputs can't be abused.

Here's what needs to be locked down:

**Bearer token on GET /api/stats/dates** — When the DateRangeToolbar calls GET /api/stats/dates, it must attach the current user's Firebase ID token as a Bearer token in the Authorization header. Use `getIdToken()` from the Firebase Auth SDK to get the token before each request. Don't cache the token in a variable for reuse — always call `getIdToken()` so Firebase can handle token refresh automatically.

**90-day cap enforced before any API call** — The client-side validation in the custom date picker must run before `setCustomRange` is called and before any API request is made. The store should never receive a date range exceeding 90 days. This is a UX guard, not a security boundary — the server enforces its own limits — but it prevents unnecessary API calls with invalid ranges.

**Date serialization** — Always serialize dates to YYYY-MM-DD using date-fns `format` before writing to the store or sending as query params. Never pass raw Date objects or locale-formatted strings to the API. This prevents malformed date params from reaching GET /api/stats/dates.

**userId always from server** — The DateRangeToolbar never sends a userId in its request to GET /api/stats/dates. The server extracts userId from the verified Firebase ID token. The toolbar only sends startDate and endDate as query params.

**Session expiry handling** — If GET /api/stats/dates returns a 401, attempt a token refresh using `getIdToken(true)` (the `true` forces a refresh). If the refresh succeeds, retry the request once with the new token. If the refresh fails, redirect the user to / and show a 'Session expired' toast. This logic should live in a shared fetch utility so it's consistent across all dashboard API calls.
