We're building a data reconciliation and validation feature for an ad network dashboard. There are no dedicated standalone pages for this feature — instead, we need to build two integrated views that surface reconciliation data to the user.

First, build a **Reconciliation Overview Panel** — this is the main hub for the feature. It should show an overall health badge at the top (healthy / degraded / critical, pulled from GET /api/reconciliation/status) with a color-coded indicator. Below that, show a per-network status grid — one card per network (exoclick, rollerads, zeydoo, propush) showing: last validation status, last validated timestamp, and anomaly count. Each card should have a 'Run Reconciliation' button that calls POST /api/reconciliation/run for just that network, plus a 'Run All' button at the top that runs all networks at once. The date range for a run should default to yesterday but let the user pick a custom range (max 90 days). Show a loading spinner on the button while the run is in progress and display the summary results (documents checked, anomalies found, duration) inline after the run completes.

Second, build an **Anomaly List View** — a filterable, paginated table of anomalous adStats records. Pull data from GET /api/reconciliation/anomalies. Show columns for: date, network, country, validation status, anomaly flags (render each flag as a badge showing its type and severity — warning in yellow, critical in red), cost, revenue, and a 'Resolve' action. Support filtering by networkId, date range, severity (warning/critical), and anomaly type. Implement cursor-based pagination using the nextCursor field from the response — show a 'Load More' button rather than numbered pages. Allow multi-select (checkboxes) so users can bulk-resolve up to 50 records at once. The resolve action should open a small inline form asking for resolution type (acknowledged / corrected / false_positive) and an optional note (max 500 chars), then call PATCH /api/reconciliation/resolve.

Third, build a **Validation Rules Editor** — a form-based view showing the current validation thresholds per network, loaded from GET /api/reconciliation/rules. Display each network's rules in a collapsible section. For each field (maxDailyRevenueUSD, minDailyRevenueUSD, maxDailyCostUSD, maxCtrPercent, minImpressions, allowNullCountry), show the current effective value and a subtle 'using default' label if that field is falling back to the system default. Make each field editable inline. On save, call PATCH /api/reconciliation/rules with only the changed fields for that network. Show the system defaults as a reference panel on the side.

All three views should handle loading skeletons while data is fetching, empty states (e.g., 'No anomalies found for this period'), and error states with retry buttons.
