We need to make sure every error state in the PDF export flow is handled clearly — both on the server and in the PdfExportTab UI.

Server-side error responses (in POST /api/export/pdf):
- 400: Return a JSON body with a clear message explaining which field failed validation (e.g., 'dateFrom must be a valid YYYY-MM-DD date', 'sections must be a non-empty array', 'filename contains invalid characters'). Don't just return a generic '400 Bad Request'.
- 401: Return a minimal JSON body ('Invalid or expired authentication token'). Don't leak any details about the token structure.
- 404: Return 'No data found for the selected date range' — this tells the client to disable the Export Now button.
- 429: Return 'Export limit reached. You can export up to 10 times per hour.' Include a Retry-After header if possible.
- 500: Log the full error server-side (never send stack traces to the client). Return 'PDF generation failed. Please try again.' to the client.

Client-side error handling in PdfExportTab (driven by the usePdfExport hook):
- 401 response: Close the modal, call getIdToken(true) to attempt a token refresh. If the refresh succeeds, the user can try again. If it fails, redirect to / and show a 'Session expired. Please sign in again.' toast.
- 403 response: Show an inline 'Access Denied' message inside the PDF tab with a close button. Do not close the modal automatically.
- 404 response: Disable the Export Now button and show an inline message 'No data found for this date range' below the checkboxes. This is a soft error — the user can change their date range and try again.
- 429 response: Show an inline banner 'Export limit reached. Try again later.' with the modal staying open.
- 500 response: Show an inline error banner inside the PDF tab with the message 'PDF generation failed. Please try again.' and a 'Retry' button that re-calls POST /api/export/pdf with the same parameters.
- Network failure (fetch throws): Treat the same as 500 — show the inline error banner with Retry.

Make sure that after any error, the Export Now button returns to its enabled state (assuming the error isn't a 404 or 429 that should keep it disabled), and all checkboxes and selectors become interactive again. The user should never be stuck in the 'exporting' disabled state after an error.
