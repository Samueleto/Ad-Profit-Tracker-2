# Step 386 of 437: Handle errors gracefully in the Activity Log UI

The ActivityLogTab needs to handle several distinct error scenarios gracefully. Here's how each one should behave:

**401 — Session expired:** When any API call (GET /api/audit/logs, GET /api/audit/logs/export, DELETE /api/audit/logs/clear) returns 401, first attempt a token refresh using firebase/auth's currentUser.getIdToken(true) (the true forces a refresh). If the refresh succeeds, retry the original request once. If the refresh fails (user's session is truly expired), redirect to / and show a 'Session expired — please sign in again' toast notification. Don't show an error state in the table — just redirect.

**403 — Access denied:** Show an inline 'Access Denied' panel inside the tab content area (not a full-page error). Include a link to /dashboard. This shouldn't happen in normal usage but handle it defensively.

**404 / empty filtered results:** When GET /api/audit/logs returns an empty array because the filters matched nothing, show a 'No logs found' state with a 'Reset filters' button that clears all active filters and re-fetches. This is different from the 'no logs ever' empty state — make the messaging distinct.

**500 — Server error on load:** Show an inline error banner above the table (not a full-page error) with the message 'Something went wrong loading your activity log' and a 'Retry' button. If logs were already loaded before the error (e.g., error on 'load more'), keep the existing rows visible and show the error banner only for the failed page load.

**500 — Export failure:** Show a toast notification: 'Export failed — please try again.' Don't navigate away or clear the table.

**500 — Clear logs failure:** Show a toast notification: 'Failed to clear logs — some entries may not have been deleted.' This is important because batch deletes can partially succeed. Close the confirmation dialog and leave the table in its current state so the user can retry.

**429 — Rate limited:** Show a toast notification with a human-readable message: 'Too many requests — please wait a moment before trying again.' For the export rate limit specifically, say 'Export limit reached — you can export up to 5 times per hour.'

**Network offline:** If the fetch fails due to no network connection (navigator.onLine is false or the request throws a network error), show an inline banner: 'You appear to be offline. Check your connection and retry.' with a Retry button.

**Filter validation:** If the user enters an invalid date range (end date before start date), show an inline validation message below the date picker before even making an API call. Don't let invalid params reach the API.

All error states in the useAuditLogs, useExportLogs, and useClearLogs hooks should expose a typed error object (with a code and message) so the ActivityLogTab can render the right UI for each case.
