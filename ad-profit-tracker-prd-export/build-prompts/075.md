We need two things configured before building the email endpoints: the transactional email provider SDK and the environment variables that control which provider gets used.

Install `nodemailer` and its TypeScript types (`@types/nodemailer`). The app uses `firebase-admin` already, so that should be in place — just confirm it's initialized correctly for server-side use.

For the email provider, we're using an environment-variable-driven approach so the provider can be swapped without code changes. Add these to your `.env.local` (and document them in `.env.example`):
- `EMAIL_PROVIDER` — which provider to use (e.g., `smtp`, `resend`, `sendgrid`, `postmark`)
- `EMAIL_PROVIDER_API_KEY` — the API key for the chosen provider
- `EMAIL_FROM_ADDRESS` — the sender address (e.g., `alerts@yourapp.com`)
- `INTERNAL_SYNC_SECRET` — a long random string used to authenticate internal server-to-server calls (Cloud Scheduler → your API). This should already exist if other internal endpoints use it — reuse the same value.

Create a shared email client factory at `src/lib/email/client.ts`. This file reads `EMAIL_PROVIDER` and returns a configured Nodemailer transporter. For SMTP-based providers (Resend, SendGrid, Postmark all support SMTP), configure the transporter with the appropriate host, port, and auth using the API key as the password. This way all three email endpoints (`POST /api/email/send-alert`, `POST /api/email/send-summary`, `POST /api/email/send-test`) import from one place and we only configure the provider once.

Also create a rate limiter utility at `src/lib/email/rateLimit.ts`. We'll use Firestore as the rate limit store (since we're already on Firebase) — store a counter document per rate limit key with a window expiry. The keys follow the pattern `{uid}_email_alert_{type}` for alerts, `{uid}_email_test` for test emails, and `{userId}_email_summary_{summaryType}_{dateFrom}` for summaries. Export a `checkRateLimit(key, maxCount, windowMs)` function that returns `{ allowed: boolean, retryAfter?: number }`.
