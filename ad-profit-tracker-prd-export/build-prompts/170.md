We're building the single API endpoint that powers the entire Financial Metrics Dashboard. This is GET /api/dashboard/metrics and it does a lot — verify auth, validate dates, check cache, query Firestore, crunch numbers, and return a consolidated payload. Let's walk through it carefully.

First, authentication. Extract the Bearer token from the Authorization header and call `adminAuth.verifyIdToken(token)` from your Firebase Admin setup. If the token is missing, malformed, or expired, return 401 immediately. The uid comes only from the verified token — never from query params or the request body.

Second, input validation. Read dateFrom and dateTo from the query string. Both are required and must be valid YYYY-MM-DD strings. dateFrom must be <= dateTo. The range between them must not exceed 90 days — return 400 with a clear message if any of these checks fail. You can use date-fns for the date math here.

Third, cache check. Build a cache key like `{uid}_dashboard_metrics_{dateFrom}_{dateTo}` and check the node-cache instance from src/lib/cache.ts. If there's a hit, return the cached result immediately with a `X-Cache: HIT` response header. This avoids hitting Firestore on repeat requests.

Fourth, on a cache miss, run two Firestore queries in parallel using Promise.all(): query the adStats collection where userId == uid AND date >= dateFrom AND date <= dateTo, and query the networkConfigs collection where userId == uid. Use the Firebase Admin Firestore instance.

Fifth, compute the metrics in memory from the adStats results. Calculate totalRevenue (sum of revenue from rollerads, zeydoo, and propush network rows), totalCost (sum of cost from exoclick rows only), netProfit (totalRevenue - totalCost), and roi ((netProfit / totalCost) * 100, or null if totalCost is zero). Build the dailySeries by grouping documents by date and computing per-day profit. Build topCountries by grouping by country, summing netProfit, sorting descending, and taking the top 10 — include a metricShare percentage for each. Build perNetwork subtotals.

Sixth, compute period-over-period changes. If the date range is 7 days or more, shift the dateFrom/dateTo back by the same number of days and filter the already-fetched adStats data (no extra Firestore query needed) to compute prior-period KPIs. Calculate percentChange for each KPI as ((current - prior) / Math.abs(prior)) * 100, handling division-by-zero gracefully.

Seventh, merge in the networkConfigs data — extract lastSyncedAt and lastSyncStatus per network and include them in the perNetwork array.

Eighth, store the result in node-cache with a 5-minute TTL and return it with a 200 status. Include cachedAt as the current ISO timestamp. If no adStats documents were found for the date range, return 404. If the Firestore queries throw, return 500.

The response shape should match the DashboardMetricsResponse type exactly.
