The PerNetworkAnalyticsTabsSection and ApiExplorerTab make several API calls that all need proper authentication and input validation on the client side.

**Bearer token on every request** — Every call to `GET /api/networks/stats`, the four raw-response endpoints (`GET /api/networks/exoclick/raw-response`, `GET /api/networks/rollerads/raw-response`, `GET /api/networks/zeydoo/raw-response`, `GET /api/networks/propush/raw-response`), and `POST /api/networks/config/test-connection` must include the Firebase ID token as a Bearer token in the Authorization header. Use `getIdToken()` from `firebase/auth` to retrieve the current token before each request. Never cache the token in component state for reuse — always call `getIdToken()` fresh (Firebase handles caching internally and will return a valid token or refresh it automatically).

**Token refresh on 401** — If any API call returns a 401, call `getIdToken(true)` to force a token refresh, then retry the request once with the new token. If the retry also returns 401, treat it as a session expiry — redirect the user to / and show a 'Session expired' toast. This logic should live in a shared API utility function used by both `useNetworkAnalytics` and `useApiExplorer`, not duplicated in each hook.

**Date range cap** — Before sending any request to `GET /api/networks/stats`, validate client-side that the date range doesn't exceed 90 days. If the parent dashboard somehow passes a range over 90 days, clamp it to 90 days and show a warning toast. Don't rely solely on the server-side validation — fail fast on the client.

**NetworkId validation** — The networkId passed to `GET /api/networks/stats` and `POST /api/sync/manual` must be one of the four valid values: 'exoclick', 'rollerads', 'zeydoo', 'propush'. Validate this before making the call. This is a defense-in-depth measure since the values come from the tab UI (not user input), but it prevents any accidental bad calls.

**Raw response safety** — The JSON viewer in ApiExplorerTab displays raw API responses fetched server-side. Make sure the JSON is rendered as text content only — never use dangerouslySetInnerHTML or eval() to display it. Use a safe JSON formatter that escapes all content. This prevents any XSS risk if a raw API response contains unexpected HTML or script content.

**No sensitive data in client state** — The raw API responses shown in ApiExplorerTab come from the server-side endpoints, which should never include decrypted API keys. On the client side, don't log API responses to the console in production builds. Add a check that strips any field named 'apiKey', 'accessToken', 'secret', or 'password' from displayed JSON as a safety net.

**POST /api/sync/manual protection** — Disable the 'Sync This Network' button immediately on click and don't re-enable it until the response comes back. This prevents accidental double-sync calls. The server handles rate limiting, but the client should not spam the endpoint.
