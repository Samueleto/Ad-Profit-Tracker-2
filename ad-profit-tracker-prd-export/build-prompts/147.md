We need TypeScript interfaces and types that match our Firestore data shapes. These will be used across the API routes, hooks, and components — so getting them right here saves pain later.

Create the following types in features/team/types/:

1. WorkspaceInvitation — This maps to the workspaceInvitations Firestore collection. Fields: id (string), workspaceId (string), workspaceName (string), invitedByUid (string), invitedByName (string), invitedEmail (string), role ('admin' | 'member'), token (string), tokenHash (string), personalMessage (string | null), status ('pending' | 'accepted' | 'declined' | 'revoked' | 'expired'), expiresAt (Timestamp), acceptedAt (Timestamp | null), acceptedByUid (string | null), createdAt (Timestamp), updatedAt (Timestamp). Make a separate WorkspaceInvitationSafe type that omits token and tokenHash — this is what gets returned from API responses.

2. WorkspaceMember — Represents a user who is part of a workspace, shaped from the users collection fields returned by GET /api/team/members. Fields: uid (string), displayName (string), email (string), workspaceRole ('owner' | 'admin' | 'member'), workspaceJoinedAt (Timestamp), photoURL (string | null).

3. WorkspaceMetadata — Returned by GET /api/team/workspace. Fields: workspaceId (string), workspaceName (string), ownerUid (string), memberCount (number), createdAt (Timestamp), currentUserRole ('owner' | 'admin' | 'member').

4. A WorkspaceRole type alias: type WorkspaceRole = 'owner' | 'admin' | 'member'. Export this separately so it can be imported anywhere without pulling in the full interfaces.

5. API response wrapper types — CreateInvitationRequest (email: string, role: 'admin' | 'member', personalMessage?: string), UpdateRoleRequest (role: 'admin' | 'member'), and a generic ApiError type (status: number, message: string, code?: string).

Also create a small helper at lib/team/tokens.ts that exports a hashToken(token: string): string function using Node's built-in crypto module to produce a SHA-256 hex hash. This is used when creating invitations (store the hash, send the plaintext in the link) and when validating them (hash the incoming token and look up by hash). Keep this server-side only — never import it in client components.
