We need three things configured before building the dashboard feature:

1. Firebase Admin SDK (server-side) — This is used in the API route to verify Firebase ID tokens and query Firestore. If Firebase Admin isn't already initialized in this project, create a shared initialization file at src/lib/firebase-admin.ts. It should initialize the admin app once (guard against re-initialization in Next.js hot reload with a check like `getApps().length === 0`). Pull the service account credentials from environment variables — never hardcode them. You'll need FIREBASE_PROJECT_ID, FIREBASE_CLIENT_EMAIL, and FIREBASE_PRIVATE_KEY in your .env.local. Export the initialized `adminAuth` and `adminDb` (Firestore) instances from this file so API routes can import them.

2. node-cache (server-side in-memory cache) — Install node-cache and create a shared cache instance at src/lib/cache.ts. Export a single instance with a default TTL of 5 minutes (300 seconds). This instance will be imported by the /api/dashboard/metrics route to cache dashboard payloads per user per date range. Make sure the instance is created as a module-level singleton so it persists across requests in the same Node.js process.

3. recharts (client-side chart library) — Install recharts. It'll be used in the dashboard's trend line chart component. No special initialization needed — just make sure it's in package.json and importable. Since recharts uses browser APIs, any component that imports it will need to be a client component (add 'use client' at the top) or be dynamically imported with `ssr: false` in Next.js.

Also make sure firebase (client SDK) and date-fns are installed — these are used on the frontend for auth token retrieval and date formatting respectively.
