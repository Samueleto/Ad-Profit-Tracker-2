We need two things set up in Firestore before building the endpoints.

First, update the security rules for the `networkConfigs` collection to support the three new caching fields we're adding (`serverCacheTTLSeconds`, `clientCacheStaleSeconds`, `clientCacheFocusRevalidate`). The rules should be:
- Any authenticated user can create a document (check `request.auth != null`)
- Any authenticated user can read documents
- Only the document owner can update it (check `request.auth.uid == resource.data.user_id`)
- Only the document owner can delete it
- The `createdAt` field must be immutable on updates â€” add a rule that rejects any update that tries to change `createdAt`

Second, make sure the `auditLogs` collection is ready to receive writes from the cache endpoints. The `POST /api/cache/invalidate` and `POST /api/cache/warm` endpoints both write audit log entries. Each entry should have: `userId`, `action` (e.g., `'cache_invalidated'` or `'cache_warmed'`), `timestamp`, and a `details` object with relevant metadata (like `scope` and `invalidatedCount` for cache invalidation, or `tasksProcessed` for warming). No special security rules are needed for `auditLogs` since it's written server-side via the Admin SDK â€” but make sure client-side reads are restricted to the document owner.

You don't need to create any new Firestore indexes for this feature since the cache endpoints don't query Firestore directly (they use the in-memory node-cache). The `networkConfigs` collection changes are additive â€” existing documents just won't have the new fields until they're updated, which is fine since we'll use defaults.
