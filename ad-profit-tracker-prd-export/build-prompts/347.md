We're building an ad network analytics dashboard. For this step, I need you to make sure all six ExoClick endpoints are properly secured and that no sensitive data leaks to the client.

**Authentication on all user-facing endpoints** — `POST /api/networks/exoclick/sync`, `GET /api/networks/exoclick/stats`, `GET /api/networks/exoclick/stats/by-country`, `GET /api/networks/exoclick/raw-response`, and `GET /api/networks/exoclick/stats/latest` must all verify the Firebase ID token using `admin.auth().verifyIdToken()`. The `uid` used for any Firestore query must always come from the verified token — never from the request body or query params. If the token is missing, malformed, or expired, return 401 immediately before doing anything else.

**Internal endpoint protection** — `POST /api/networks/exoclick/scheduled-sync` must NOT accept Firebase ID tokens. It should only accept requests where the `x-internal-secret` header exactly matches the `INTERNAL_SYNC_SECRET` environment variable. Return 401 if the header is missing or wrong. This endpoint should never be reachable from the client — consider adding a middleware check that blocks it if the request origin looks like a browser.

**API key handling** — The ExoClick API key is decrypted server-side using `cryptr` in `POST /api/networks/exoclick/sync` and `POST /api/networks/exoclick/scheduled-sync`. Make sure the decrypted key is never logged (not even at debug level), never included in any response body, and never written to Firestore. The `rawResponse` stored in the `adStats` collection must be sanitized before writing — strip any field whose name contains 'key', 'token', 'secret', 'auth', or 'credential' (case-insensitive).

**Firestore Security Rules for adStats** — The `adStats` collection rules should enforce: only authenticated users can create documents, and only if the `userId` field in the document matches their own UID. Users can only read and update their own documents (where `userId == request.auth.uid`). Deletes should be soft-deleted (set a `deleted` flag) — block hard deletes at the rules level. The `createdAt` and `userId` fields must be immutable after creation.

**Rate limiting on sync** — `POST /api/networks/exoclick/sync` enforces a max of 10 manual syncs per hour per user. Implement this using a Firestore counter or a rate-limit utility. Return 429 with a clear message including when the limit resets.

**Input validation** — All date parameters across all endpoints must be validated as proper `YYYY-MM-DD` strings, with `dateFrom <= dateTo` and a max 90-day range. The `limit` param on the by-country endpoint must be an integer between 1 and 100. Reject anything outside these bounds with 400 before touching Firestore or the ExoClick API.
