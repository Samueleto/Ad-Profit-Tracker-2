We need TypeScript types for the request/response shapes of all three endpoints. Create these in `src/lib/deployment/types.ts`.

For GET /api/health, define a `HealthResponse` type that covers both the success and unhealthy cases. The success shape (HTTP 200) should have: `status` as `'healthy' | 'degraded'`, `firestore` as `'ok' | 'slow'`, `firestoreLatencyMs` as a number, `memoryMB` as a number, `uptime` as a number, and `version` as a string. The unhealthy shape (HTTP 503) should have: `status` as `'unhealthy'` and `error` as a string.

For GET /api/ready, define a `ReadinessResponse` type. The ready shape (HTTP 200) should have: `ready` as `true` and `checks` as an object with `envVars: 'ok'` and `firebaseAdmin: 'ok'`. The not-ready shape (HTTP 503) should have: `ready` as `false` and `failedCheck` as a string.

For GET /api/deployment/status, define a `DeploymentStatusResponse` type for the success case (HTTP 200): `service` as string, `revision` as string, `region` as string, `environment` as string, `uptimeSeconds` as number, `nodeVersion` as string, and `deployedAt` as `string | null`. Also define an `ErrorResponse` type for the 401 and 403 cases with a `message` string field.

Finally, define a `RequiredEnvVars` type or constant that lists the four env vars the readiness probe checks: `FIREBASE_PROJECT_ID`, `INTERNAL_SYNC_SECRET`, `EMAIL_PROVIDER_API_KEY`, `EMAIL_FROM_ADDRESS`. This makes it easy to update the list in one place if new required vars are added later.

Keep all these types in one file — this feature is small enough that splitting them up would just add noise.
