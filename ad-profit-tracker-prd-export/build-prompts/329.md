Every API call the ManualRefreshPanel makes needs to be authenticated. Let's make sure the security is airtight on the frontend side.

**Token attachment** — In the `useManualRefresh` hook, before every call to POST /api/sync/manual, GET /api/scheduled/sync-status, and GET /api/scheduled/sync-history, call `getIdToken()` from firebase/auth on the current user and attach it as `Authorization: Bearer <token>` in the request headers. Never hardcode tokens or read them from localStorage — always get a fresh token from the Firebase SDK.

**Token refresh on 401** — If any API call returns 401, don't immediately redirect. First, try to silently refresh the token by calling `getIdToken(true)` (the `true` forces a refresh). If the refresh succeeds, retry the original request once with the new token. If the refresh also fails (the user's session is truly expired), then redirect to / and show the 'Session expired' toast. This prevents unnecessary logouts due to token expiry during normal use.

**403 handling** — If any call returns 403, show an 'Access Denied' inline banner inside the ManualRefreshPanel with a link to /dashboard. Don't redirect — the user is already on the dashboard, so just surface the error in context.

**No sensitive data in the UI** — Make sure the panel never displays API keys, decrypted credentials, or raw error messages from the network APIs. The results from POST /api/sync/manual include errorMessage fields — sanitize these before displaying them (truncate at 200 characters, strip anything that looks like a key or token pattern).

**Rate limit transparency** — The 429 response includes a Retry-After header. Read this value and use it to drive the countdown timer. Don't let users work around the rate limit by refreshing the page — the rate limit is enforced server-side, so a page refresh will just get another 429. The UI countdown is a courtesy, not a security control.

**Confirmation on destructive action** — The 'Refresh All Data Now' button triggers syncs across all active networks simultaneously, which has real cost implications (API calls, Firestore writes). Keep the confirmation tooltip on first use to prevent accidental mass syncs. Store whether the user has confirmed before in component state (not localStorage — reset on page load is fine).
