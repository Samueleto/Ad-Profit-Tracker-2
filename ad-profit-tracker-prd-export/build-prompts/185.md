We need five API route handlers for the help center. Here's what each one does:

**GET /api/help/articles** — The main article listing endpoint used by HelpCenterPage. Verify the Firebase ID token from the Authorization header using admin.auth().verifyIdToken(). Read the user's document from the users Firestore collection to get their workspaceRole and effectivePermissions. Build a Firestore query on helpArticles filtering isPublished == true, optionally adding a category filter if the category query param is provided (validate it against the HELP_CATEGORIES enum — return 400 if invalid). Fetch all matching docs, then filter in memory: exclude any article where requiredPermission is set and the user's effectivePermissions doesn't include it, and exclude articles where requiredRole is set and the user's workspaceRole doesn't meet it. Sort by updatedAt descending. Apply cursor-based pagination (default limit 20, max 50). Compute isNew for each article (updatedAt within last 7 days). Group results by category to build the categoryGroups array. Return the paginated articles list, categoryGroups, hasMore, and nextCursor. Never return body in this response — only list fields.

**GET /api/help/articles/:articleId** — Single article fetch used by HelpArticlePage. Verify token, validate articleId is non-empty. Read the helpArticles document — return 404 if not found or isPublished == false. Read the user's permissions. If article.requiredPermission is set, check effectivePermissions — return 403 if missing. If article.requiredRole is set, check workspaceRole — return 403 if missing. Return the full article including body.

**PATCH /api/help/articles/:articleId/view** — View count increment, fire-and-forget. Verify token. Check a rate limit key '{uid}_view_{articleId}' in the helpCache — if already set within the last hour, return 200 with { incremented: false } silently. Otherwise, use FieldValue.increment(1) to bump viewCount on the helpArticles document, set the rate limit key in cache with 3600s TTL, and return { incremented: true }.

**GET /api/help/search** — In-memory search used by HelpCenterPage's search bar. Verify token. Validate q is non-empty after trimming (max 100 chars — return 400 if empty). Check helpCache for key '{uid}_help_search_{q}_{category}_{limit}' — return cached result with X-Cache: HIT header if found. Fetch all isPublished == true articles from helpArticles. Filter by user permissions (same logic as the list endpoint). Run the scoring logic from src/lib/helpSearch.ts: normalize q to lowercase, score title matches at 3 points, summary matches at 2 points, tags matches at 1 point. Filter out zero-score results. Sort by score descending, then viewCount descending as tiebreaker. Slice to limit (default 10, max 20). Cache the result for 120 seconds. Return results with matchedIn array and score.

**POST /api/help/feedback** — Feedback submission used by HelpArticlePage's thumbs widget. Verify token. Validate articleId (required), rating ('helpful' or 'not_helpful'), and optional comment (max 500 chars). Check rate limit key '{uid}_feedback_{articleId}' in helpCache — return 429 if already submitted. Read the article — return 404 if not found or unpublished. Use FieldValue.increment(1) to bump either helpfulCount or notHelpfulCount on the helpArticles document. Write an entry to the auditLogs Firestore collection with action='preferences_updated', resourceType='help_article', resourceId=articleId, metadata={ rating, hasComment: !!comment }, status='success'. Set the rate limit key in cache. Return { submitted: true, articleId, rating }.

Put the search scoring logic (normalize, match, score) in src/lib/helpSearch.ts so it's testable separately from the route handler.
