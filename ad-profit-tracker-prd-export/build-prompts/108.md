# Step 108 of 437: Configure Firestore rules and indexes for notifications

We're storing notifications as an embedded array inside each user's document in the users Firestore collection. There's no separate notifications collection — everything lives on users.notifications. This step sets up the security rules and any indexes needed for the notification queries to work correctly and safely.

For Firestore security rules, the users collection needs these rules applied: only authenticated users can create a users document (auth.uid != null), users can only read their own document (resource.data.userId == request.auth.uid or match on the document ID being the uid), users can only update their own document, and deletes are soft — meaning the app never hard-deletes user documents, it just marks things as dismissed or cleared. The createdAt field on individual notification objects inside the array should be treated as immutable — don't allow clients to change it after creation. Note that all the actual notification writes happen server-side via the Admin SDK (which bypasses security rules), so these rules primarily protect against direct client-side Firestore access.

For indexes: since we're doing all the filtering and sorting of the notifications array in application code (server-side in the API routes, not as Firestore queries), we don't need composite indexes on the notifications array fields. However, if the project uses any direct Firestore queries on the users collection elsewhere (like querying by email or role), make sure those existing indexes aren't broken.

Also set up the auditLogs Firestore collection that DELETE /api/notifications/clear and PATCH /api/notifications/preferences write to. This collection should only be writable by the server (Admin SDK) and readable by admins only — regular users should not be able to read or write it directly. Add that to the security rules.

Deploy the updated rules using the Firebase CLI and confirm they apply without errors.
