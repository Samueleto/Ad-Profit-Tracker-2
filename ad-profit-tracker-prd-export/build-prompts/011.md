We're building an automated daily data sync system that pulls ad network statistics from ExoClick, RollerAds, Zeydoo, and Propush on a schedule and stores them in Firestore. For this step, set up the folder structure so everything has a home before we start writing logic.

On the backend (Firebase Functions or Next.js API routes), create a folder for the scheduled sync endpoints. We'll have five endpoints living here: POST /api/scheduled/daily-sync (the master orchestrator), GET /api/scheduled/sync-status, GET /api/scheduled/sync-history, POST /api/scheduled/retry-failed, and GET /api/scheduled/health.

Also create a shared utilities folder for the pieces that multiple endpoints will reuse — things like the API key decryption helper, the per-network API client wrappers (one each for ExoClick, RollerAds, Zeydoo, and Propush), the Firestore batch-upsert helper, the exponential backoff retry wrapper, and the audit log writer. Keeping these separate from the route handlers will make the code much easier to maintain and test.

Create a types folder for shared TypeScript interfaces — we'll define the shapes for network configs, ad stats rows, audit log entries, and sync summary objects here.

This feature has no frontend pages of its own, so there's no client-side folder structure needed beyond what already exists in the project.
