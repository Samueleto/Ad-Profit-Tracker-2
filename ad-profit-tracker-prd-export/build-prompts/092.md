# Step 92 of 437: Set up Firestore collections and indexes for ad stats

================================================================================

**Category:** Database Setup
**Flow:** feature_historical-data-storage

================================================================================

================================================================================

We need to configure Firestore so the stat queries run efficiently. There's no schema migration like a SQL database — instead, we need to set up composite indexes and security rules.

Create or update firestore.indexes.json with composite indexes for the adStats collection. The endpoints query this collection in several combinations, so we need indexes for:
- userId + date (for the snapshot endpoint's single-date lookup)
- userId + date range queries (userId ASC, date ASC — for trend, summary, coverage, and dates endpoints)
- userId + date + networkId (for filtered queries where networkId is provided)
- userId + date + country (for the geo-breakdown endpoint which filters out null country rows)

For the auditLogs collection, add an index on userId + resourceType + action + createdAt (for the coverage endpoint's sync history query).

For the networkConfigs collection, add an index on userId + isActive (for the backfill endpoint's active network lookup).

For the apiKeys collection, add an index on userId + networkId (for the backfill endpoint's key verification check).

Update firestore.rules to ensure the adStats, auditLogs, networkConfigs, and apiKeys collections are only readable and writable server-side (via the Admin SDK, which bypasses rules). Client-side access should be denied for all of these — the stat data flows through the API endpoints only, never directly from the browser to Firestore.

Deploy the indexes with firebase deploy --only firestore:indexes. Note that composite indexes can take a few minutes to build in Firebase.
