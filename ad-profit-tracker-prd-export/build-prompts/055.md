We need the Firebase Admin SDK properly initialized so all six reconciliation endpoints can read and write to Firestore securely on the server side.

First, check if firebase-admin is already initialized somewhere in the project (there's likely an existing admin SDK setup for other features). If it is, just make sure the reconciliation endpoints can import and use the shared instance — don't initialize it twice, as that causes errors.

If it's not already set up, install firebase-admin and create a shared initialization file. Initialize it using a service account — either from a JSON key file (for local dev, never commit this) or from environment variables (FIREBASE_PROJECT_ID, FIREBASE_CLIENT_EMAIL, FIREBASE_PRIVATE_KEY). For production on Firebase hosting or Cloud Run, you can use applicationDefault() credentials instead. Export the initialized Firestore instance and auth instance so all route handlers can import them.

The reconciliation feature specifically needs:
- admin.auth().verifyIdToken(token) — to authenticate every single request. All six endpoints use this.
- admin.firestore() — for reading adStats, networkConfigs, and writing to auditLogs.
- Firestore batch writes (db.batch()) — the run endpoint writes validation results in chunks of 500 documents, and the resolve endpoint does the same.
- admin.firestore.FieldValue.serverTimestamp() — used heavily for validationCheckedAt, reconciledAt, lastValidationAt, and updatedAt fields.

Also set up a simple in-memory or Firestore-backed rate limiter utility. The run endpoint allows max 10 reconciliation runs per hour per user, the resolve endpoint allows max 20 resolve operations per hour per user, and the rules PATCH endpoint allows max 20 rule updates per hour per user. A simple approach is to store a counter + window start time in memory (fine for single-instance deployments) or in a Firestore subcollection for multi-instance setups.
