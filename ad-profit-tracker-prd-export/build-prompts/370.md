We need the SyncStatusPanel to fail gracefully across all the error scenarios defined for this feature. Here's how each one should behave:

**401 — Session Expired** — When any API call returns 401, first try to refresh the Firebase ID token by calling `getIdToken(true)` (the `true` forces a refresh from the server). If the refresh succeeds, immediately retry the failed request once with the new token. If the refresh fails (user is truly signed out or token is revoked), stop all polling, show a 'Session expired' toast notification, and redirect to / (the root/login page). Don't show an error banner inside the panel — just redirect.

**403 — Access Denied** — Show an inline 'Access Denied' banner inside the SyncStatusPanel with a link to /dashboard (in case they navigated here via a deep link). Stop polling — a 403 won't resolve on its own. Don't redirect.

**500 — Server Error** — Show an inline error banner inside the panel: 'Unable to load sync status' with a 'Retry' button. Pause polling when a 500 occurs. When the user clicks 'Retry', re-fetch immediately and resume polling if the retry succeeds. If the retry also returns 500, keep the error banner and stay paused — don't loop.

**Network Error (fetch failed)** — Show a subtle inline notice: 'Connection issue — retrying...' Keep polling at the normal interval (don't pause). If the next poll succeeds, clear the notice automatically. This handles brief connectivity blips without alarming the user.

**Partial Failure (anomalies or reconciliation call fails)** — If GET /api/reconciliation/anomalies fails but GET /api/sync/live-state succeeds, render the panel normally without the CriticalAnomaliesStrip. Don't show a full error state — just silently omit the anomalies section. Log the failure in development mode only.

**Empty State** — If GET /api/sync/live-state returns successfully but all four networks have `isActive=false`, show the empty state: 'No networks configured — add API keys in Settings' with a link to /settings. Don't show this as an error — it's a valid state for new users.

**Loading State** — On initial mount, before the first response arrives, show shimmer skeleton placeholders for all four network cards and gray placeholder badges. The health banner should show a neutral 'Checking...' state. Don't show any error UI during the initial load — wait for the response first.

**Stale Data Warning** — If the last successful poll was more than 2 minutes ago (detectable via `lastPolledAt` in the hook state) and polling is still running, show a subtle 'Data may be stale' badge near the 'Refresh Status' button. This catches cases where polls are firing but returning cached or stale responses.

**Error Recovery** — When the user clicks 'Retry' after a 500, or when polling resumes after a network error resolves, clear the error state and error banner as soon as a successful response arrives. Don't require a page reload to recover.
