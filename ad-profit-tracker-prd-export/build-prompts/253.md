We need a clean state management layer for the SyncStatusPanel so the component itself doesn't get cluttered with polling logic, token fetching, and interval management. Build this as a custom React hook called `useSyncStatus`.

Here's what the hook needs to manage:

**Data State** — Track: the array of four network status objects (from GET /api/sync/live-state), overall health string, the activity feed entries (from GET /api/sync/activity-feed), critical anomalies (from GET /api/reconciliation/anomalies), a loading boolean for the initial load, an error object (with a `type` field so the UI can distinguish 401 vs 403 vs 500), and a `lastPolledAt` timestamp.

**Token Fetching** — Before every API call, get a fresh Firebase ID token using `getIdToken()` from firebase/auth. Pass the current user from Firebase auth. Don't cache the token between polls — always call `getIdToken()` fresh (Firebase caches it internally and refreshes when needed). If `getIdToken()` throws, treat it as a 401 scenario.

**Polling Logic** — On mount, immediately fetch all data. Then set up a polling interval. The interval duration should be reactive: check if any network in the current state has `syncPhase` of 'fetching' or 'writing' — if yes, use 5000ms; otherwise use 30000ms. Every time new data arrives, re-evaluate the interval duration and reset the timer if it changed. This means you may need to clear and re-create the interval when the sync state transitions between active and idle.

**Fetch Orchestration** — The main fetch function should call these in parallel (Promise.all or similar): GET /api/sync/live-state, GET /api/sync/activity-feed (with limit=5), and GET /api/reconciliation/anomalies. Only call GET /api/reconciliation/status if the live-state response doesn't already include `overallHealth` — avoid redundant calls. Handle partial failures gracefully: if the anomalies call fails, don't fail the whole panel — just set anomalies to an empty array and log the error.

**Error State Tracking** — Track errors with enough detail for the UI to react differently: `{ type: '401' | '403' | '500' | 'network', message: string }`. On a 401, set a flag `sessionExpired: true` that the component can use to trigger a redirect. On a 500, pause polling and set `pollingPaused: true`. On a network error, show the error but keep polling (the user might just have a brief connectivity blip).

**Manual Refresh** — Expose a `refresh()` function that immediately re-fetches all data and resets the polling timer. The component calls this when the user clicks the 'Refresh Status' button.

**Cleanup** — The hook must clear all intervals and cancel any in-flight fetch calls on unmount. Use an `isMounted` ref or AbortController to avoid setting state after unmount.

**Exposed API** — The hook should return: `{ networks, overallHealth, activityFeed, criticalAnomalies, isLoading, error, sessionExpired, pollingPaused, lastPolledAt, refresh }`.

Keep all the polling, token, and fetch logic inside this hook. The SyncStatusPanel component should just call `useSyncStatus()` and render based on what it gets back.
