Next, fully implement the two utility functions in src/lib/schedules/date-utils.ts:

computeNextRunAt(frequency, deliveryHour, timezone, dayOfWeek?, dayOfMonth?) — This should use date-fns-tz to figure out the next time this schedule should fire. For 'daily': find the next occurrence of deliveryHour:00 in the given timezone (if it's already past that hour today, use tomorrow). For 'weekly': find the next occurrence of the given dayOfWeek at deliveryHour:00 in the timezone. For 'monthly': find the next occurrence of dayOfMonth at deliveryHour:00 — if dayOfMonth has already passed this month, use next month. Return a JavaScript Date object. Handle edge cases: what if dayOfMonth is 29, 30, or 31 and the month doesn't have that many days (cap at last day of month).

resolveDateRangePreset(preset, timezone) — Returns { dateFrom: string, dateTo: string } in YYYY-MM-DD format, computed relative to 'now' in the given timezone. 'yesterday' → yesterday's date. 'last_7' → 7 days ago to yesterday. 'last_30' → 30 days ago to yesterday. 'last_90' → 90 days ago to yesterday. 'this_month' → first day of current month to today.

Also implement a validateScheduleFields(input) function in src/lib/schedules/validation.ts that checks all the field rules from the POST endpoint spec: frequency enum, dayOfWeek 0–6 if weekly, dayOfMonth 1–28 if monthly, deliveryHour 0–23, timezone valid IANA string (use Intl.supportedValuesOf('timeZone')), deliveryEmail valid format, dateRangePreset enum, format enum. Return an array of field-level error messages so the API routes can return specific 400 errors.
