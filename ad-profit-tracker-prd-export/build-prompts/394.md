We need to make sure the Financial Metrics Dashboard fails gracefully for every error scenario. The page has several distinct error states and each needs specific handling.

**401 — Session expired** — When GET /api/dashboard/metrics or POST /api/cache/invalidate returns 401, the useDashboardMetrics hook should first attempt to force-refresh the Firebase ID token using getIdToken(true) and retry the request once. If the retry also returns 401 (or if getIdToken(true) itself throws), treat the session as expired: clear the dashboard state, show a toast notification saying 'Session expired. Please sign in again.', and redirect the user to /. Don't show the 401 error inline on the dashboard — redirect immediately.

**403 — Access denied** — If the API returns 403, show an 'Access Denied' card in the main content area of FinancialMetricsDashboardPage. Include a brief explanation and a link back to /dashboard (in case they navigated here via a stale link). Don't show the KPI cards or chart.

**404 — No data for range** — If the API returns 404 (no data found for the selected date range), show an inline message: 'No data found for the selected date range.' Include a 'Reset Date Range' button that resets the selector back to Last 7 days and re-fetches. This is different from the empty state — 404 means the server confirmed there's nothing, while empty means the response came back but arrays were empty.

**500 — Server error** — If the API returns 500 or the fetch throws a network error, show a dismissible error banner at the very top of the page (above the KPI cards). The banner should say something like 'Something went wrong loading your dashboard.' and include a 'Retry' button that re-fetches all data from GET /api/dashboard/metrics. Keep any previously loaded data visible below the banner if it exists — don't wipe the page.

**Date range validation error** — If the user selects a custom range over 90 days, show an inline validation message directly below the date picker: 'Date range cannot exceed 90 days.' Disable the fetch until they correct it.

**Partial data / null values** — The roi field can be null when totalCost is 0 (division by zero). The KPI card for ROI should show '—' instead of null or NaN. The profitChange, revenueChange, costChange, and roiChange fields can also be null if there's no prior period data — show '—' for the percentage change and hide the directional arrow in that case.

**Chart with no data points** — If dailySeries is an empty array, show a 'No trend data available for this period' message inside the chart area instead of rendering an empty recharts canvas.

**GEO table with no rows** — If topCountries is empty, show 'No geographic data available' in the table body.

**POST /api/cache/invalidate failure** — If the invalidate call fails, don't block the re-fetch. Log the error, show a brief non-blocking toast ('Cache refresh failed, showing latest available data'), and proceed with the GET /api/dashboard/metrics fetch anyway.
