This feature reads from and writes to four Firestore collections: `networkConfigs`, `apiKeys`, `adStats`, and `auditLogs`. These collections likely already exist in the project — this step is about making sure the right fields and indexes are in place for the queries this feature needs to run efficiently.

**networkConfigs** — The manual sync endpoint queries this collection with `where('userId', '==', uid).where('isActive', '==', true)` to find which networks to sync when no specific `networkId` is provided. It also writes back `lastSyncedAt`, `lastSyncStatus`, and `lastSyncError` after each sync. Make sure a composite index exists on `(userId, isActive)` so this query doesn't fail in production. Also confirm the documents have `retryAttempts` (defaults to 3 if missing), `networkId`, and `isActive` fields.

**apiKeys** — The endpoint queries `where('userId', '==', uid).where('networkId', '==', network)` to find the encrypted API key for each network. Make sure a composite index on `(userId, networkId)` exists. The `encryptedKey` field (or whatever it's called) must be present — this is what `cryptr` decrypts server-side.

**adStats** — The `syncNetworkForUser` helper batch-upserts rows here in chunks of 500. Confirm the collection exists and that documents have a consistent structure (networkId, userId, date, country, impressions, clicks, revenue, etc.). The batch upsert needs a deterministic document ID strategy (e.g., `{userId}_{networkId}_{date}_{country}`) to avoid duplicates on re-sync.

**auditLogs** — Each sync writes a new document here with `action='manual_sync_triggered'`, `resourceType='sync'`, `resourceId=networkId`, `status`, `metadata` (latencyMs, rowsFetched, dateFrom, dateTo, triggeredBy), and a timestamp. No special indexes needed for writes, but if the sync history drawer reads from here, confirm the query used by `GET /api/scheduled/sync-history` is indexed.

If any composite indexes are missing, add them to `firestore.indexes.json` and deploy with `firebase deploy --only firestore:indexes`.
