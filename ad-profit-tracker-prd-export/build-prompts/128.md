We need TypeScript interfaces for every Firestore collection the stat endpoints touch. This gives us type safety across all eight route handlers and makes the aggregation logic much easier to write correctly.

Create src/lib/stats/types.ts with the following interfaces:

AdStatDocument — represents one row in the adStats collection. Fields: id (string), userId (string), networkId (one of 'exoclick'|'rollerads'|'zeydoo'|'propush'), dataRole (string — 'cost' for exoclick, 'revenue' for the others), date (string, YYYY-MM-DD), country (string|null — null for total rows, ISO alpha-2 for country-level rows), cost (number|null), revenue (number|null), impressions (number), clicks (number), ctr (number), cpm (number), createdAt (Firestore Timestamp), updatedAt (Firestore Timestamp).

NetworkConfig — represents a row in networkConfigs. Fields: id, userId, networkId, isActive (boolean), lastSyncedAt (Timestamp|null), lastSyncStatus (string), lastSyncError (string|null).

ApiKey — represents a row in apiKeys. Fields: id, userId, networkId, encryptedKey (string), createdAt (Timestamp).

AuditLog — represents a row in auditLogs. Fields: id, userId, action (string), resourceType (string), resourceId (string), metadata (Record<string, unknown>), status ('success'|'failure'), createdAt (Timestamp).

Also create response shape types for each endpoint's success response — SnapshotResponse, TrendResponse, SummaryResponse, GeoBreakdownResponse, CoverageResponse, BackfillResponse, DatesResponse. These match the SUCCESS response shapes documented in the endpoint specs.

Finally, create a NETWORK_IDS constant (the array ['exoclick','rollerads','zeydoo','propush']) and a REVENUE_NETWORKS constant (['rollerads','zeydoo','propush']) that the aggregation helpers will use to distinguish cost vs. revenue networks. Export everything from an index file.
