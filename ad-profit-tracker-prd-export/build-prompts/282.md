We need to manage the state for the ScheduleReportModal cleanly so the component itself stays focused on rendering. For this step, build a custom hook — something like useScheduleForm — that handles all the data fetching, form state, and API interactions for the modal.

Here's what the hook needs to manage:

**Form state:** Track all the form field values — frequency, dayOfWeek, dayOfMonth, deliveryHour, timezone, deliveryEmail, dateRangePreset, format, and isActive. Initialize these with sensible defaults (Daily, user's email from Firebase Auth, Last 7 days, Excel, active=true, deliveryHour=8, timezone from the user's browser or stored preference). When an existing schedule loads, replace the defaults with the loaded values.

**Existing schedule state:** Track whether a schedule already exists for the given reportId (loaded from GET /api/schedules?reportId=...). Store the full schedule object including its id — this is what determines whether Save calls POST /api/schedules/create or PATCH /api/schedules/:id, and whether the Delete and Send Test Now buttons are shown.

**Loading state:** Track a loading boolean for the initial GET /api/schedules fetch. This drives the shimmer placeholder display.

**Saving state:** Track a saving boolean for when POST or PATCH is in flight. This drives the spinner on the Save button and the disabled state on all inputs.

**Error state:** Track an error object that can hold field-level validation errors (from 400 responses) or a general error message. This gets cleared when the user starts editing.

**Next delivery preview:** Compute the human-readable next delivery label client-side based on the current form values (frequency, dayOfWeek, dayOfMonth, deliveryHour, timezone). This should update reactively as the user changes the form. You can use date-fns-tz for the computation — find the next occurrence of the scheduled time in the user's timezone and format it as something like 'Tomorrow at 8:00 AM EST' or 'Next Monday at 8:00 AM'.

**Actions the hook exposes:**
- fetchSchedule(reportId): calls GET /api/schedules?reportId=reportId, populates form if found
- saveSchedule(): calls POST /api/schedules/create or PATCH /api/schedules/:id depending on whether an existing schedule is loaded. On success, triggers a 'Schedule saved' toast and calls an onSuccess callback.
- deleteSchedule(): calls DELETE /api/schedules/:id. On success, triggers a toast and calls an onSuccess callback.
- sendTestNow(): calls POST /api/schedules/:id/send-now. Shows a success or error toast.
- updateField(field, value): updates a single form field and clears any error for that field.

**Token handling:** Before every API call, get a fresh Firebase ID token using getIdToken() and include it as a Bearer token in the Authorization header. Don't cache the token in state — always fetch it fresh to avoid sending expired tokens.

Keep all the API call logic inside this hook. The ScheduleReportModal component should just call hook methods and render based on hook state — no fetch calls directly in the component.
