We're building a Propush ad network integration. For this step, I need you to make sure every failure mode across all six endpoints is handled cleanly — both on the server side and surfaced helpfully to the user.

**Server-side error handling:**

For POST /api/networks/propush/sync:
- If the Propush API v5 call to `/v5/pub-statistics` fails (network error, 4xx, 5xx), update the user's networkConfigs document with `lastSyncStatus='failed'` and return 502 with a message like 'Propush API unavailable — sync failed'. Don't let the raw Propush error message leak to the client.
- If the Firestore batch write fails partway through, log the error and return 500. Don't leave the sync status as 'success' if writes failed.
- If the API key can't be decrypted, return 500 without revealing anything about the key or encryption.

For POST /api/networks/propush/scheduled-sync:
- Process each user's sync independently — if one user's sync fails, continue with the others. Collect all failures and log them.
- Write an audit log entry for each user regardless of success or failure.
- Return a summary response like `{ succeeded: 5, failed: 2, errors: [...] }` so the scheduler can detect partial failures.

For GET /api/networks/propush/stats and GET /api/networks/propush/stats/by-country:
- If the date range is invalid or missing, return 400 with a clear message explaining what's wrong.
- If Firestore returns no results, return an empty array (not 404) — no data is a valid state.

For GET /api/networks/propush/stats/latest:
- If no data exists for this user, return 404 with a message like 'No stats available yet — try syncing first'.
- If the networkConfigs document is missing, still return the stats summary without the sync status (don't fail the whole request).

For GET /api/networks/propush/raw-response:
- If the date param is missing or invalid, return 400.
- If no records exist for that date, return an empty array with the schema definition still included.

**Client-side error handling:**

In the Propush API client module, wrap every call in try/catch. Map HTTP status codes to user-friendly messages:
- 401 → redirect to login or show 'Session expired, please sign in again'
- 429 → show 'Sync limit reached — you can sync up to 10 times per hour'
- 400 → show the server's error message (it should be safe and descriptive)
- 502 → show 'Propush is temporarily unavailable — please try again later'
- 500 → show 'Something went wrong on our end — please try again'
- Network failure (no response) → show 'Check your internet connection'

Make sure errors from the sync operation are surfaced clearly — this is an action the user triggers manually, so they need to know if it failed and why.
