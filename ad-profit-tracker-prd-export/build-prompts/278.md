# Step 278 of 437: Set up per-network state with Zustand

We're using Zustand to manage the API Explorer's state. The key design decision here is that state is tracked per network — ExoClick, RollerAds, Zeydoo, and Propush each have their own independent slice of state so that fetching a fresh sample for one network doesn't affect the others.

Create a Zustand store for the API Explorer. For each network, track:
- `cachedResponse`: the raw JSON response object (or null if none exists yet)
- `lastFetched`: a timestamp string (or null)
- `isLoadingCached`: boolean — true while fetching the last stored response from the backend
- `isFetchingFresh`: boolean — true while a live 'Fetch Fresh Sample' call is in progress
- `error`: an error message string (or null)
- `isConfigured`: boolean — whether an API key exists for this network (derived from the apiKeys/networkConfigs data)

Expose these actions:
- `loadCachedResponse(network)`: calls the appropriate raw-response endpoint (/api/networks/exoclick/raw-response, /api/networks/rollerads/raw-response, /api/networks/zeydoo/raw-response, or /api/networks/propush/raw-response) and stores the result. Sets isLoadingCached during the call.
- `fetchFreshSample(network)`: calls /api/networks/config/test-connection with the network identifier, updates cachedResponse and lastFetched on success. Sets isFetchingFresh during the call.
- `clearError(network)`: resets the error field for a specific network
- `setConfigured(network, value)`: updates the isConfigured flag

The store should initialize with all networks in a clean state. When an accordion is expanded for the first time, the component should trigger loadCachedResponse for that network if cachedResponse is null and isConfigured is true. Don't auto-fetch on every expand — only on the first open.

Keep all the actual fetch logic inside the store actions, not in the components. Components should only call store actions and read store state.
