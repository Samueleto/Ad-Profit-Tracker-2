We need two things wired up before building the rate limiting logic:

**1. Firebase Admin SDK** — This is used for verifying Firebase ID tokens on every request. If the project doesn't already have a Firebase Admin initialization file, create one at `lib/firebase/admin.ts`. Initialize the Admin SDK using a service account (from environment variables — never hardcode credentials). Export the `admin` instance so route handlers can call `admin.auth().verifyIdToken(token)` and `admin.firestore()` for Firestore reads/writes. Make sure initialization is guarded so it only runs once even in hot-reload environments (check `getApps().length` before calling `initializeApp`).

Add these to your `.env.local`: `FIREBASE_PROJECT_ID`, `FIREBASE_CLIENT_EMAIL`, `FIREBASE_PRIVATE_KEY` (the private key needs newline characters preserved — wrap it carefully). Also add `INTERNAL_SYNC_SECRET` — this is the shared secret used by the dual-auth pattern on the reset endpoint.

**2. Bottleneck** — Install the `bottleneck` package. This is the core throttling library. In `lib/rate-limit/limiterConfig.ts`, define the static configuration for all four networks. Each network config should include: `maxConcurrent` (how many outbound API calls can run at once), `minTime` (minimum milliseconds between requests), `reservoir` (max requests per time window), `reservoirRefreshInterval` (window size in ms), and `reservoirRefreshAmount` (how many tokens to add per refresh). Set sensible defaults for each of the four networks — ExoClick, RollerAds, Zeydoo, and ProPush may have different rate limits so stub them out with placeholder values that can be tuned later.

In `lib/rate-limit/limiterRegistry.ts`, create a module-level Map that stores Bottleneck instances keyed by a string like `userId:networkId`. Export a `getLimiter(userId, networkId)` function that returns the existing instance or creates a new one using the config from `limiterConfig.ts`. This in-memory registry is what the status and reset endpoints will read from.
