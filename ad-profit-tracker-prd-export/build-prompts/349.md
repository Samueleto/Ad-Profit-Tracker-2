We're building a Zeydoo ad network integration. For this step, I need you to make sure all six endpoints are properly secured.

**User-facing endpoints** — POST /api/networks/zeydoo/sync, GET /api/networks/zeydoo/stats, GET /api/networks/zeydoo/stats/by-country, GET /api/networks/zeydoo/stats/latest, and GET /api/networks/zeydoo/raw-response all require a valid Firebase ID token. Verify it server-side using admin.auth().verifyIdToken(token) on every request — never trust a userId passed in the request body or query params. The userId used for all Firestore queries must come from the verified token, not from the client.

**Data isolation** — Every Firestore query against adStats, networkConfigs, apiKeys, and auditLogs must be scoped to the authenticated user's uid. A user should never be able to read or trigger syncs for another user's data. Double-check that the queries in all five user-facing endpoints filter by userId === token.uid.

**API key handling** — The Zeydoo API key is fetched from the apiKeys collection and decrypted entirely server-side in POST /api/networks/zeydoo/sync and POST /api/networks/zeydoo/scheduled-sync. The decrypted key must never appear in API responses, logs, or error messages. Make sure the server-side decryption helper doesn't accidentally leak the key in exception stack traces.

**Rate limiting** — POST /api/networks/zeydoo/sync enforces a limit of 10 requests per hour per user. Implement this using a counter in Firestore or an in-memory store. Return 429 with a clear message when the limit is hit, including when the rate limit resets.

**Scheduled sync security** — POST /api/networks/zeydoo/scheduled-sync must NOT accept Firebase ID tokens — it's internal only. It should only accept requests with the correct x-internal-secret header matching the INTERNAL_SYNC_SECRET environment variable. Return 401 immediately if the header is missing or wrong. Make sure this endpoint is not accidentally exposed through any client-side code.

**Input validation** — All date parameters (dateFrom, dateTo, date) must be validated as valid YYYY-MM-DD strings and the range must not exceed 90 days. The limit param on by-country must be clamped between 1 and 100. Reject malformed inputs with 400 before doing any database work.
