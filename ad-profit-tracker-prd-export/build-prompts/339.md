We need to make sure the three cache endpoints and the `networkConfigs` Firestore collection are properly secured.

For the API endpoints, all three have different security models and each needs to be airtight:

`POST /api/cache/invalidate` and `GET /api/cache/status` both require a valid Firebase ID token. Verify the token server-side using the Firebase Admin SDK — don't trust anything the client sends about who they are. For the invalidate endpoint, cache keys must be namespaced by the authenticated user's UID (e.g., `uid_exoclick_stats_...`) so a user can never invalidate another user's cached data. Double-check that the key construction logic enforces this — if a user passes `keys: ['other_user_uid_stats']`, it should be ignored or rejected. The rate limit on invalidate is 15 per hour per user — make sure this is enforced server-side, not just documented.

`POST /api/cache/warm` must only be callable with the correct `x-internal-secret` header. This secret should live in an environment variable (never hardcoded). If the header is missing or wrong, return 401 immediately. This endpoint should never accept Firebase user tokens as an alternative auth method — keep the two auth paths completely separate.

For Firestore, the `networkConfigs` collection rules should enforce: any authenticated user can create a document (with their UID set as `user_id`) and read documents, but only the document owner (where `user_id == request.auth.uid`) can update or delete. The `createdAt` field must be immutable — add a rule that rejects writes that try to change it after creation. Make sure these rules are deployed and tested with the Firestore rules simulator before going live.

Also audit the `auditLogs` collection that both `invalidate` and `warm` write to — make sure users can't write to it directly from the client, only via the server-side API routes.
