Let's make sure the email alert endpoints are properly secured. There are a few distinct security concerns here that need careful attention.

**Dual-auth on POST /api/email/send-alert** — This endpoint accepts either a Firebase ID token (user-initiated) or an x-internal-secret header (server-side Cloud Function triggers). Make sure the logic is strictly either/or: if both are present, prefer the Firebase token path. If neither is valid, return 401. Critically, when the x-internal-secret path is used, the userId must come from the request body — but only after the secret is validated. Never trust userId from the request body on the Firebase token path; always derive it from the verified token.

**deliveryEmail must never come from the request body** — For POST /api/email/send-alert, POST /api/email/send-test, and POST /api/email/send-summary, the delivery email address must always be read from Firestore (users.preferences.alertDeliveryEmail) or Firebase Auth (admin.auth().getUser(uid).email). If a client sends an email address in the request body, ignore it entirely. This prevents a user from redirecting alerts to an arbitrary address.

**Internal-only endpoints** — POST /api/email/send-summary and POST /api/email/schedule-summary must only accept the x-internal-secret header. Add an explicit check at the top of each handler: if an Authorization header is present (suggesting a user is trying to call it), still reject with 401 unless the x-internal-secret is also valid. These endpoints should never be callable by end-users.

**Rate limiting** — Verify that all rate limits are enforced server-side using a reliable store (e.g., Firestore counters or a Redis-backed rate limiter, not in-memory): 10 alert emails/hour per user per type for POST /api/email/send-alert, 3 test emails/hour per user for POST /api/email/send-test, 1 summary email/day per summaryType per user for POST /api/email/send-summary. Return 429 with a Retry-After header on all rate limit hits.

**Email address validation** — On PATCH /api/notifications/preferences when alertDeliveryEmail is provided, validate RFC 5322 format server-side and enforce the 254-character max. Don't rely on client-side validation alone.

**Audit log scoping** — GET /api/email/log must only return auditLog documents where userId matches the verified token's uid. Add this as a Firestore query filter, not a post-fetch filter, so users can never access each other's email history.

**INTERNAL_SYNC_SECRET** — Make sure this environment variable is set in your Firebase Functions config and is never exposed in client-side code, API responses, or logs.
