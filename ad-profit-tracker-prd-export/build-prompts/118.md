We need TypeScript types that match the `apiKeys` Firestore schema, plus some shared types for the API request/response shapes. This keeps everything type-safe across the frontend and backend.

Create a types file at `src/types/api-keys.ts`. Define the following:

**`ApiKeyDocument`** — This represents what's actually stored in Firestore. Fields: `encryptedKey: string`, `networkId: string`, `userId: string`, `createdAt: FirebaseFirestore.Timestamp`, `updatedAt: FirebaseFirestore.Timestamp`. Mark `createdAt`, `userId`, and `networkId` as `readonly` to reflect that they're immutable after creation.

**`NetworkStatus`** — This is what the GET /api/keys/status endpoint returns per network. Fields: `networkId: string`, `status: 'connected' | 'not_connected'`, `updatedAt: string | null` (ISO string, since we serialize timestamps before sending to the client).

**`SaveKeyRequest`** — The request body shape for POST /api/keys/save. Fields: `networkId: string`, `key: string`.

**`DeleteKeyRequest`** — The request body shape for DELETE /api/keys/delete. Fields: `networkId: string`.

**`NetworkId`** — A union type derived from the `SUPPORTED_NETWORKS` constant so TypeScript can validate network IDs at compile time. It should look like `'exoclick' | 'rollerads' | 'zeydoo' | 'propush'`. Derive this from the constants file rather than duplicating the values.

Also create a simple validation helper in `src/lib/validation.ts` that exports an `isValidNetworkId(id: string): id is NetworkId` function. This checks whether a given string is in the `SUPPORTED_NETWORKS` array. The API endpoints will use this to validate incoming `networkId` values before doing anything else.
