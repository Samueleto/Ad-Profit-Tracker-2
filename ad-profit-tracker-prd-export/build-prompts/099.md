The `GET /api/networks/stats` endpoint queries the `adStats` Firestore collection with multiple filters and sort conditions. Without the right composite indexes, Firestore will reject these queries with an error. Let's set those up now.

The main query pattern is: filter by `userId == uid` AND `networkId == <networkId>` AND `date >= dateFrom` AND `date <= dateTo`. This needs a composite index on `(userId, networkId, date)` — Firestore can't handle range filters on multiple fields without it.

For the `groupBy='daily'` case, results are sorted by `date ASC`, so the index should include date in ascending order.

For the `groupBy='country'` case, results are grouped and sorted by the primary metric descending in application code (not in Firestore), so no additional index is needed for that sort — just the base filter index.

Add these composite indexes to your `firestore.indexes.json`:
1. Collection: `adStats`, fields: `userId ASC`, `networkId ASC`, `date ASC` — this covers all three groupBy modes

Also check the `networkConfigs` collection, which the endpoint reads to get `lastSyncedAt`, `lastSyncStatus`, and `circuitBreakerOpen` per network. The query there is by `userId` and `networkId` — if there's no index already covering that, add one: Collection: `networkConfigs`, fields: `userId ASC`, `networkId ASC`.

Deploy the indexes with `firebase deploy --only firestore:indexes`. Note that new indexes can take a few minutes to build in Firebase — you'll see a 'Building' status in the Firebase console. The API will return errors until they're ready, so do this before testing the endpoint.
