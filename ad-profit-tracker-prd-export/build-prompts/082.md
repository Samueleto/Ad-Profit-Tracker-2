We need to configure Firestore so the `apiKeys` subcollection is properly structured and locked down. The data lives at `users/{uid}/apiKeys/{networkId}` — each document represents one ad network connection for one user.

**Firestore Security Rules** — Update your `firestore.rules` file to add rules for the `apiKeys` subcollection. The rules should enforce:
- Only authenticated users can create documents, and only in their own subcollection (the `userId` field in the document must match the requesting user's UID)
- Users can only read documents from their own subcollection
- Users can only update documents in their own subcollection
- For deletion, implement a soft-delete pattern: instead of actually deleting the document, users can update it to mark it as deleted. However, looking at the endpoint spec, the DELETE endpoint does a hard delete — so allow hard deletes but only on the user's own documents
- The `createdAt` and `userId` fields must be immutable after creation (use `request.resource.data.createdAt == resource.data.createdAt` to enforce this on updates)

**Firestore Indexes** — The GET /api/keys/status endpoint queries the entire `users/{uid}/apiKeys` subcollection. This is a simple collection read with no complex ordering, so no composite indexes are needed. However, document this in a comment in your `firestore.indexes.json` so future developers know it was considered.

**Supported Networks** — Define the list of supported network IDs in a shared constants file at `src/lib/constants.ts`. Export a `SUPPORTED_NETWORKS` array containing `'exoclick'`, `'rollerads'`, `'zeydoo'`, and `'propush'`. Both the API endpoints and the frontend will import from this file, so the list stays in sync. This is also where you should define any display names or metadata for each network (like a human-readable label for the UI).
