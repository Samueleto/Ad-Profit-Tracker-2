We need TypeScript types and validation schemas for the helpArticles collection so every part of the app agrees on the shape of the data.

Create a HelpArticle interface in src/features/help/types/helpArticle.ts. It should include all the fields from the helpArticles Firestore collection: id (string), title (string), slug (string), category (string — use a union type or enum for the six allowed values: 'faq' | 'api_guide' | 'troubleshooting' | 'video_tutorial' | 'network_setup' | 'account'), body (string), summary (string), tags (string[]), requiredPermission (string | null), requiredRole (string | null), readTimeMinutes (number), videoUrl (string | null), viewCount (number), helpfulCount (number), notHelpfulCount (number), isPublished (boolean), authorName (string), createdAt (Timestamp or ISO string depending on whether it's coming from the Admin SDK or the client), and updatedAt (same).

Also create a HelpArticleListItem type — a slimmed-down version used in list views that omits the body field and adds an isNew computed boolean (true if updatedAt is within the last 7 days). This is what GET /api/help/articles returns in its articles array.

Create a CategoryGroup type: { category: string, count: number } — used in the categoryGroups array returned by GET /api/help/articles.

Create a HelpSearchResult type matching what GET /api/help/search returns: id, title, slug, category, summary, readTimeMinutes, matchedIn (string[]), score (number), updatedAt.

Add a HelpFeedbackPayload type for the POST /api/help/feedback request body: { articleId: string, rating: 'helpful' | 'not_helpful', comment?: string }.

Finally, define a HELP_CATEGORIES constant array with the six category values and their display labels and icon names — this will be used by both HelpCenterPage's category grid and the API validation logic to check that a provided category is in the allowed enum.
