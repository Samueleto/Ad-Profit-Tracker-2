We need a few focused pieces of state management to keep the RBAC pages clean and avoid duplicating data-fetching logic.

**Permission matrix state** — Create a hook (something like `useRolePermissions`) that fetches from GET /api/rbac/permissions and exposes: the full roles permission map, the customizedRoles array, a loading boolean, an error value, and a `refresh` function. The PermissionsTab uses this hook to populate the matrix table and to pre-populate the RolePermissionsEditModal when it opens. The refresh function should be called after a successful PATCH so the matrix stays in sync without a full page reload.

**My permissions state** — Create a separate hook (something like `useMyPermissions`) that fetches from GET /api/rbac/my-permissions and exposes: the current user's permissions map, their workspaceRole, isOwner boolean, loading, and error. This is used by the 'Effective Permissions' section in PermissionsTab. It's also the hook that other parts of the app should use to gate UI elements — things like 'can this user see the Export button?' should read from this hook rather than re-fetching.

**Edit modal state** — The RolePermissionsEditModal needs local state for: which role is currently selected in the segmented control, the in-progress permission toggles (a copy of the fetched values that can be mutated without affecting the displayed matrix), whether a save is in progress, and any inline validation errors. When the selected role changes, reset the toggle state to the fetched values for that role. Don't mutate the fetched data directly — keep a local draft copy.

**Permission check utility** — Create a small helper function (something like `checkPermission(permissions, key)`) that takes the permissions map from `useMyPermissions` and a permission key string, and returns a boolean. This makes it easy for any component in the app to do a quick `checkPermission(permissions, 'canExportData')` guard without re-fetching. This utility will be used in the feature_wiring step to gate navigation items and action buttons.

Keep these hooks and utilities in a shared location (something like `features/rbac/` or `hooks/rbac/`) so they can be imported by both the RBAC pages and by other features that need to gate access.
