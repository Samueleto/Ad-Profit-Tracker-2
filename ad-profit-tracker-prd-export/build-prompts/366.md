We're building a Cloud Run deployment pipeline for a Next.js + Firebase app. For this step, I need you to make sure the security model across all three endpoints is correctly implemented and airtight.

The security split here is intentional and important: GET /api/health and GET /api/ready must be completely public — no auth, no token, no session check. Cloud Run's infrastructure calls these automatically and has no way to provide credentials. If you add any auth check to these, the instance will fail its health checks and get killed. Verify that no middleware, Next.js route config, or global auth wrapper is touching these two routes.

GET /api/deployment/status is the opposite — it must be strictly protected. Here's what the security chain should look like: extract the Bearer token from the Authorization header, verify it using Firebase Admin's admin.auth().verifyIdToken(token), and if that fails for any reason (expired, malformed, revoked), return 401 immediately. Never trust a uid from the request body or query params — always use the uid from the verified token. After verifying the token, look up the user's Firestore document and check that their workspaceRole is either 'owner' or 'admin'. If it's anything else (or the document doesn't exist), return 403. Only after both checks pass should you read and return the deployment metadata.

For the deployment metadata itself, be careful about what you expose. The endpoint returns service name, revision, region, environment, uptime, Node.js version, and a deployedAt timestamp — this is infrastructure context that's useful for admins but shouldn't be public. Make sure the response never includes actual environment variable values (especially secrets like INTERNAL_SYNC_SECRET or EMAIL_PROVIDER_API_KEY). The /api/ready endpoint already follows this pattern — it only reports presence/absence of env vars, never their values. Apply the same discipline to /api/deployment/status.

Also make sure the 2-second Firestore timeout in /api/health is enforced with a real Promise.race or AbortController — don't rely on Firestore's default timeout, which can be much longer and would cause Cloud Run to consider the probe failed.
