This feature has two distinct security boundaries that need to be airtight: internal endpoints protected by a shared secret, and user-facing endpoints protected by Firebase ID tokens. Let's make sure both are properly enforced.

For the internal endpoints — POST /api/scheduled/daily-sync and GET /api/scheduled/health — the only valid authentication is the `x-internal-secret` header matching the `INTERNAL_SYNC_SECRET` environment variable exactly. These endpoints must reject any request that is missing this header or has a mismatched value with a 401 immediately, before doing any other work. Never fall through to any other auth check. These endpoints must never be callable from the browser — they should only be reachable by Cloud Scheduler or your monitoring system. If your Next.js setup allows it, consider adding middleware that blocks these routes from being called with a `Cookie` or `Authorization` header (a browser would send those, Cloud Scheduler wouldn't).

For the user-facing endpoints — GET /api/scheduled/sync-status, GET /api/scheduled/sync-history, and POST /api/scheduled/retry-failed — every request must be authenticated by verifying the Firebase ID token via `admin.auth().verifyIdToken()`. The `userId` used in every Firestore query must always come from the verified token, never from a query parameter or request body. Double-check that the sync-history endpoint scopes its `auditLogs` query to `userId == uid` — a user must never be able to see another user's sync history by passing a different userId.

For the retry endpoint (POST /api/scheduled/retry-failed), enforce the rate limit of 5 retries per hour per user per network before doing any expensive work like fetching the API key or calling the network. Store rate limit state in Firestore or an in-memory counter — whichever fits the existing app's pattern.

API keys must be decrypted server-side only using the `cryptr`/AES helper with `ENCRYPTION_SECRET`. The decrypted key must never appear in logs, error messages, API responses, or audit log metadata. If a decryption fails, return 500 without revealing why it failed.

Make sure `INTERNAL_SYNC_SECRET` and `ENCRYPTION_SECRET` are set in your Firebase Functions environment config (or `.env` for local dev) and are never committed to source control. Add both to your `.gitignore`-protected env files and document them in a `.env.example`.
