The reconciliation feature runs some fairly specific Firestore queries, so we need the right indexes in place or those queries will fail with Firestore index errors at runtime.

For the adStats collection, we need composite indexes to support the query patterns used by the six endpoints:
- (userId, networkId, date) — used by POST /api/reconciliation/run to fetch documents for a specific network and date range
- (userId, validationStatus, date DESC) — used by GET /api/reconciliation/anomalies to list anomalous records sorted by date
- (userId, networkId, validationStatus, date DESC) — used by GET /api/reconciliation/anomalies when filtering by both network and status
- (userId, networkId, validationStatus) — used by GET /api/reconciliation/status to fetch the 5 most recent anomalous records per network

For the networkConfigs collection:
- (userId, networkId) — used by GET /api/reconciliation/status, GET /api/reconciliation/rules, and PATCH /api/reconciliation/rules
- (userId, isActive) — used by POST /api/reconciliation/run when no networkId is provided, to find all active networks

Add these to your firestore.indexes.json file and deploy them. Note that composite indexes can take a few minutes to build after deployment.

For Firestore security rules, the adStats and networkConfigs collections already have rules defined, but make sure they're correctly set up: any authenticated user can create and read documents, but updates and deletes are restricted to the document owner (userId == request.auth.uid). The createdAt field should be immutable — reject writes that try to change it after creation. The auditLogs collection should be write-only from the server (Admin SDK bypasses rules, so this is mainly about preventing client-side tampering).
