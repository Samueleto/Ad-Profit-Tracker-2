## Description

FUNCTIONALITY: A modal dialog triggered by the 'Customize Role Permissions' button on PermissionsTab. Allows Owners (and Admins for Member/Viewer roles only) to toggle individual permissions on or off per role. Displays a scrollable list of all permission keys grouped by category (Data Access, Network Management, Team Management, Export & Reports, Settings). Each permission row shows the permission name, a brief description, and a toggle switch. A role selector at the top (segmented control: Admin / Member / Viewer — Owner permissions are immutable) switches which role is being edited. A 'Reset to Defaults' button restores the selected role's permissions to system defaults. A 'Save Changes' button calls PATCH /api/rbac/permissions with the updated permission map. A 'Cancel' button closes without saving. Current values pre-populated from GET /api/rbac/permissions on modal open. Inline validation prevents removing all permissions from a role. A warning banner appears if a change would remove a permission that active workspace members currently rely on.
DESIGN APPROACH: Centered modal (max-w-lg) with backdrop overlay using existing ThemeProvider tokens. Grouped permission rows with category headers. Toggle switches use accent color CSS custom property. Role selector at top as segmented control. Destructive 'Reset to Defaults' at bottom-left, Cancel and Save at bottom-right. Saving state disables all toggles and shows spinner on Save button.
STATES: loading — modal body shows 8 shimmer permission rows while GET /api/rbac/permissions resolves; saving — Save button shows spinner, all toggles disabled; error_401 — close modal, call getIdToken(true), if fails redirect to / with 'Session expired' toast; error_403 — show 'Only Owners and Admins can edit role permissions' inside modal; error_500 — inline error banner inside modal with Retry; success — 'Permissions updated' toast, modal closes, parent matrix refreshes.
SECURITY: Protected — Owner can edit Admin/Member/Viewer permissions. Admin can only edit Member/Viewer permissions. Server enforces hierarchy — client role state is display-only.
